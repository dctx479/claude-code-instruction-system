# Claude Code CLI 最佳实践指南

> 本文是 **Claude Code 使用与配置指南**，聚焦 `CLAUDE.md`、Subagents、Commands、MCP、Hooks 等工程化实践。
> 
> 若要查看项目内沉淀的**策展型最佳实践条目库**（BP-001 / BP-017 等），请参考 `memory/best-practices.md`。

## 一、CLAUDE.md 配置最佳实践

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **保持简洁** | 控制在 150-200 条指令以内 |
| **使用指针** | 引用文件位置而非复制代码 |
| **迭代优化** | 根据使用反馈持续改进 |
| **分层配置** | 全局 → 项目 → 本地 |

### 1.2 文件位置优先级

```
1. ~/.claude/CLAUDE.md          (全局)
2. 项目根目录/CLAUDE.md          (项目)
3. 项目根目录/CLAUDE.local.md    (本地,不提交)
4. 子目录/CLAUDE.md              (按需加载)
```

### 1.3 推荐结构

```markdown
# 项目名称

## 项目概述
[简要描述]

## 技术栈
[主要技术和版本]

## 核心命令
[构建、测试、部署命令]

## 代码规范
[关键编码规则]

## 项目结构
[目录说明]

## 自主决策授权
[可自主执行 vs 需确认]
```

### 1.4 避免的做法

- ❌ 把 CLAUDE.md 当作 linter 配置
- ❌ 添加大量代码片段
- ❌ 包含过时的信息
- ❌ 添加与当前任务无关的指令

### 1.4 规则放置决策表

| 规则类型 | 推荐位置 | 说明 |
|---------|---------|------|
| 全局行为约束 | `CLAUDE.md` | 所有会话都要遵守的高优先级规则 |
| 稳定参考规则 | `.claude/reference/` | 详细规范、最佳实践、长文档说明 |
| 可执行提示模块 | `.claude/prompts/` | 需要被 prompt 组合系统直接复用的话术或模板 |
| 技能特例规则 | 对应 `SKILL.md` / `resources/` | 仅在某个 skill 或工作流内生效的规则 |

**原则**：新增规则先判断应该挂在哪一层，优先复用既有模块，不在多个层级重复拷贝同一条规则。

## 二、子智能体 (Subagents) 最佳实践

### 2.1 创建专业化智能体

```yaml
---
name: 智能体名称
description: 详细描述,说明何时使用
tools: 限制工具访问
model: 选择合适的模型
---

[系统提示词]
```

### 2.2 模型选择策略

| 任务类型 | 推荐模型 | 原因 |
|----------|----------|------|
| 快速搜索 | haiku | 速度快、成本低 |
| 代码实现 | sonnet | 平衡能力和成本 |
| 架构设计 | opus | 需要深度思考 |
| 代码审查 | sonnet | 需要仔细分析 |

### 2.3 工具权限原则

- 只授予必要的工具访问权限
- 读取任务使用只读工具
- 使用 hooks 进行额外验证

## 三、自定义命令最佳实践

### 3.1 命令组织

```
.claude/commands/
├── general/          # 通用命令
│   ├── commit.md
│   └── fix-issue.md
├── dev/              # 开发命令
│   ├── review.md
│   └── check.md
└── project-specific/ # 项目特定命令
```

### 3.2 命令设计原则

1. **单一职责** - 每个命令做一件事
2. **参数化** - 使用 `$ARGUMENTS` 传递参数
3. **清晰输出** - 定义明确的输出格式
4. **可组合** - 命令之间可以协作

### 3.3 示例命令模板

```markdown
[简短描述任务目标]

## 执行步骤
1. [步骤1]
2. [步骤2]
3. [步骤3]

## 验证
[如何验证成功]

## 输出格式
[期望的输出结构]
```

## 四、MCP 服务器最佳实践

### 4.1 常用 MCP 服务器

| 服务器 | 用途 | 安装命令 |
|--------|------|----------|
| github | GitHub 操作 | `claude mcp add github npx -y @modelcontextprotocol/server-github` |
| filesystem | 文件操作 | `claude mcp add fs npx -y @modelcontextprotocol/server-filesystem` |
| postgres | 数据库 | `claude mcp add pg npx -y @modelcontextprotocol/server-postgres` |

### 4.2 配置作用域

- **local**: 仅当前用户在当前项目可用
- **project**: 通过 `.mcp.json` 与团队共享
- **user**: 在所有项目中可用

### 4.3 安全注意事项

- 使用环境变量存储敏感信息
- 限制 MCP 服务器的权限
- 定期审查 MCP 配置

## 五、Hooks 最佳实践

### 5.1 Hook 类型

| 类型 | 触发时机 | 用途 |
|------|----------|------|
| PreToolUse | 工具执行前 | 验证、阻止危险操作 |
| PostToolUse | 工具执行后 | 后处理、通知 |
| Notification | 通知时 | 外部通知 |
| Stop | 停止时 | 清理 |

### 5.2 退出码含义

- `0`: 继续执行
- `1`: 记录错误并继续
- `2`: 阻止工具执行

### 5.3 常用 Hook 场景

- 文件写入前的备份
- 危险命令的阻止
- 编辑后的 lint 检查
- 提交前的测试运行

## 六、工作流模式

### 6.1 探索-规划-实现-验证

```
1. 探索: 阅读相关代码,理解上下文
2. 规划: 制定实现方案
3. 实现: 编写代码
4. 验证: 运行测试,确认功能
```

### 6.2 测试驱动开发

```
1. 编写失败的测试
2. 实现最小代码使测试通过
3. 重构代码
4. 重复
```

### 6.3 多智能体协作

```
1. 研究员收集信息
2. 架构师设计方案
3. 实现者编写代码
4. 审查员检查质量
```

## 七、提示词工程技巧

### 7.1 有效提示词的特点

- **具体明确**: 说清楚要做什么
- **提供上下文**: 包含必要背景信息
- **定义输出**: 说明期望的结果格式
- **设置约束**: 明确什么不应该做

### 7.2 提示词模式

**任务分解**
```
将任务分解为以下步骤:
1. [步骤1]
2. [步骤2]
...
```

**角色设定**
```
作为一名 [角色],你需要 [任务]。
注意 [约束]。
```

**少样本学习**
```
示例:
输入: A
输出: B

现在处理:
输入: C
输出: ?
```

### 7.3 强化关键指令

- 使用 "IMPORTANT" 强调关键点
- 使用 "YOU MUST" 表示必须遵守
- 使用 "NEVER" 表示禁止事项

## 八、性能优化

### 8.1 上下文管理

- 定期使用 `/clear` 清理上下文
- 使用子智能体隔离任务上下文
- 避免在对话中累积无关信息

### 8.2 成本控制

- 为简单任务使用 haiku 模型
- 使用子智能体处理特定任务
- 批量处理相似任务

### 8.3 并行处理

- 使用 git worktrees 并行开发
- 多个子智能体并行执行独立任务
- 使用后台执行长时间任务

## 九、安全实践

### 9.1 权限管理

```json
{
  "permissions": {
    "allow": ["安全操作列表"],
    "deny": ["危险操作列表"]
  }
}
```

### 9.2 敏感信息保护

- 使用环境变量存储密钥
- 不在 CLAUDE.md 中包含敏感信息
- 使用 .gitignore 排除敏感文件

### 9.3 代码安全审查

- 使用安全审计命令
- 检查依赖漏洞
- 验证输入输出

## 十、故障排除

### 10.1 常见问题

| 问题 | 解决方案 |
|------|----------|
| 上下文溢出 | 使用 `/clear` 或子智能体 |
| 指令不被遵循 | 强化指令,减少指令数量 |
| MCP 连接失败 | 检查配置和网络 |
| 工具执行失败 | 检查权限和 hooks |

### 10.2 调试技巧

- 使用 `--mcp-debug` 调试 MCP
- 查看 Claude Code 日志
- 逐步执行复杂任务

### 10.3 寻求帮助

- 使用 `/help` 查看可用命令
- 查阅官方文档
- 提交问题到 GitHub Issues

# 知识架构指南 (Knowledge Architecture Guide)

> 构建可靠知识系统的四层架构模式  
> 来源: Anthropic 数据分析系统设计 (2026-06-05)

---

## 核心理念

**知识系统的核心挑战不是信息获取，而是精准映射**。

给 AI 访问 1000 个文档不会自动提升准确率。80% 的错误答案中，正确信息其实就在语料库里，但 AI 没能把问题映射到正确的实体。

**设计原则**: 从治理良好的真相源（Truth Sources）开始，而非从原始数据堆积开始。

---

## 各层职责

### Layer 1: 数据基础设施 (Data Infrastructure)

**职责**: 存储和传输数据

**组件**:
- 数据仓库 (Snowflake, BigQuery, Redshift)
- 原始表 (Raw Tables)
- ETL/ELT 管道
- 权限和安全控制

**对 AI 可见性**: 仅在 Layer 2/3 无法覆盖时才直接访问

**常见问题**:
- ❌ 直接让 AI 扫描原始表 → 字段名歧义、缺少业务语义
- ❌ 自动推断 schema → 技术结构 ≠ 业务含义

---

### Layer 2: 真相源 (Truth Sources)

**职责**: 提供治理过的、业务语义明确的数据访问入口

**核心组件**:

#### 1. 语义层 / Metric Store

**定义**: 由领域专家定义的、消除歧义的指标和维度。

**示例**:
```yaml
metric: monthly_active_users
definition: |
  COUNT(DISTINCT user_id) 
  WHERE last_activity_date >= DATE_TRUNC('month', CURRENT_DATE)
  AND user_type = 'organic'  -- 排除测试账号
dimensions: [region, product, cohort]
owner: analytics_team
updated: 2026-05-15
```

**价值**: 
- 封装复杂 JOIN 和过滤逻辑
- 一处定义，全局一致
- 业务语义显式化（"活跃"的定义不再模糊）

#### 2. Data Lineage (数据血缘)

**定义**: 追踪数据从源到最终报表的完整流动路径。

**用途**:
- "这个指标是怎么算出来的？" → 回溯到源表和转换逻辑
- "修改这个表会影响哪些报表？" → 正向追踪下游依赖

#### 3. 历史查询模式 (Query Templates)

**定义**: 资深分析师验证过的查询模板和 SQL 片段。

**示例**:
```sql
-- 模板: 用户留存率计算
-- 作者: senior_analyst@company.com
-- 验证: 2026-04 月报
WITH cohort AS (
  SELECT user_id, DATE_TRUNC('month', signup_date) AS cohort_month
  FROM users
),
activity AS (
  SELECT user_id, DATE_TRUNC('month', event_date) AS activity_month
  FROM events
  WHERE event_type = 'core_action'
)
SELECT 
  cohort_month,
  activity_month,
  COUNT(DISTINCT a.user_id) * 1.0 / COUNT(DISTINCT c.user_id) AS retention_rate
FROM cohort c
LEFT JOIN activity a ON c.user_id = a.user_id
GROUP BY 1, 2
```

#### 4. 业务上下文 (Glossary + Context Docs)

**定义**: 术语表、业务流程文档、产品规格。

**示例**:
```markdown
# Glossary

## 用户 (User)
- **注册用户** (Registered User): 完成注册流程的账号
- **活跃用户** (Active User): 过去 30 天内有核心操作
- **付费用户** (Paying User): 当前订阅状态为 active
- **月活** (MAU): 自然月内至少登录 1 次

⚠️ 在分析时必须先确认用户指哪一种，不同指标使用不同定义。
```

**关键设计要素**:
- **优先级排序**: 语义层 → 历史查询 → 原始表（强制路由）
- **显式歧义消解**: 同一术语的多种含义必须全部列出
- **来源标注**: 每个定义注明负责人和最后更新时间
- **版本追踪**: schema 变更时同步更新文档

### Layer 3: 技能层 (Skills)

**职责**: 将用户问题精准映射到 Layer 2 的真相源

**核心模式**: Pairwise Skills

```
Knowledge Skill (router) + Runbook Skill (SOP)
```

#### Knowledge Skill (路由器)

**功能**: 索引 30-50 个参考文档，快速定位相关知识源。

**结构**:
```markdown
## 可用参考文档

### 指标定义
- [用户指标定义](metrics/user-metrics.md) — 注册/活跃/付费用户定义
- [收入指标定义](metrics/revenue-metrics.md) — 确认/预计/已结算收入

### 查询模板
- [留存率计算](queries/retention.sql) — 标准留存率计算模板
- [漏斗分析](queries/funnel.sql) — 多步骤转化漏斗

### 数据源
- [用户表说明](tables/users.md) — users 表 schema 和注意事项
- [事件表说明](tables/events.md) — events 表字段和过滤规则
```

**使用方式**:
```
用户: "帮我查上个月的用户留存率"
↓
Knowledge Skill: 路由到 queries/retention.sql + metrics/user-metrics.md
↓
确认: "留存率需要明确用户定义，请问是指所有注册用户还是仅付费用户？"
```

#### Runbook Skill (标准操作流程)

**功能**: 封装资深分析师的分析流程。

**四步强制流程**:

```markdown
## 数据查询 SOP

### 第一步：澄清需求（Clarify）
在执行任何查询前，必须确认：
1. **时间范围**: 是截止日期还是滚动窗口？
2. **实体定义**: "用户"指什么？"收入"是哪种口径？
3. **统计口径**: 去重逻辑？排除条件？

**歧义词检查表**（查询前必须确认）：
- "用户" → 注册/活跃/付费？
- "收入" → 确认/预计/已结算？
- "活跃" → DAU/WAU/MAU？

### 第二步：查找数据源（Find Sources）
**信息源优先级（强制遵守）**：
1. 语义层/Metric Store（必须首选）
2. 历史查询模板
3. 原始表（仅在上述均无法覆盖时）

**禁止跳过语义层的常见借口**：
- ❌ "需要自定义日期筛选" → 语义层已支持
- ❌ "需要 join 多张表" → 语义层已封装

### 第三步：执行查询（Run Query）
查询前检查：
- 数据新鲜度：MAX(date) = ?
- 必须过滤条件：排除测试账号？
- 命名规范人群：使用已定义的 segment

执行后验证：
- 结果数量级合理？
- 关键字段有 NULL？
- 时间戳在预期范围？

### 第四步：对抗性审查（Adversarial Review）
在给出最终答案前，自我挑战：
1. 我选对表了吗？
2. 过滤条件完整吗？
3. 时间窗口对吗？
4. 指标定义准确吗？

如果任何一个问题的答案是"不确定"，返回第一步重新澄清。
```

#### 歧义消解表 (Disambiguation Table)

**功能**: 列出所有高频歧义术语及其正确含义。

**示例**:
| 术语 | 候选实体 | 如何区分 |
|------|---------|---------|
| "用户" | 注册用户 / 活跃用户 / 付费用户 | 需问清楚统计口径 |
| "上周" | 自然周 / 过去 7 天 | 需确认时间窗口定义 |
| "收入" | 确认收入 / 预计收入 / 已结算收入 | 需明确会计口径 |

**使用时机**: 在第一步"澄清需求"时强制检查。

### Layer 4: 验证层 (Validation)

**职责**: 确保输出质量可信、可追溯、可改进

**核心组件**:

#### 1. 离线评估集 (Offline Evaluation)

**设计原则**:
- **快照锁定**: Ground truth 锁定到特定日期（如 2026-06-01），避免数据变化影响评估
- **版本追踪**: Agent/Skill 每次修改都重跑评估，记录准确率变化
- **边际递减**: 50 个典型查询已足够，100 个提升 <2%

**评估协议**:
```markdown
## 评估集结构
- 50 个历史仪表盘的典型查询
- Ground truth: 2026-06-01 快照数据
- 准确率门槛: ≥90%（低于此值不可上线）

## 版本追踪
| 版本 | Git SHA | 准确率 | Token 消耗 | 备注 |
|------|---------|--------|-----------|------|
| v1.0 | abc123 | 89% | 12K | 未通过，需改进 |
| v1.1 | def456 | 93% | 13K | 已上线 |
| v1.2 | ghi789 | 91% | 11K | 回退至 v1.1（准确率降低） |
```

> 详细协议: `memory/best-practices.md` BP-043

#### 2. 对抗性审查 (Adversarial Review)

**功能**: 主动挑战所有假设和边界条件。

**成本收益**:
| 指标 | 变化 |
|------|------|
| 准确率提升 | +6% (89% → 95%) |
| Token 消耗 | +32% |
| 响应延迟 | +72% |

**适用场景**: 核心业务逻辑、对外 API、安全敏感功能

**实施方式**: 见 `docs/QA-SYSTEM.md` 对抗性审查章节

#### 3. 来源标注 + 置信度追踪

**强制输出格式**:
```markdown
## 查询结果
[数据表格或关键数字]

## 方法论
- **数据源**: [语义层指标名 / 参考文档 / 原始表名]
- **时间范围**: [2026-05-01 至 2026-05-31]
- **过滤条件**: [去重逻辑、排除条件]
- **数据新鲜度**: [MAX(date) = 2026-06-05，截止至昨日]

## 置信度
- **高** — 数据来自语义层，指标定义明确，查询模板已验证
```

**禁止**: 不带来源标注的回答（即使答案正确）

#### 4. 主动自修复 (Proactive Self-Healing)

**机制**: 定时扫描用户纠错信号，自动草拟参考文档修正。

**实施协议**:
```markdown
## 信号检测
每日扫描:
- 用户追问"不是这个，我要的是 X"
- 手动修改查询结果
- 标记为"答案不对"

## 自动修正流程
1. Agent 扫描检测到的错误模式
2. 草拟参考文档修正（不直接修改）
3. 发送给负责人审核
4. 审核通过后合入主分支
5. 重跑离线评估验证改进效果
```

> 详细协议: `memory/best-practices.md` BP-044

## 架构决策记录 (ADRs)

### ADR-001: 为什么从语义层开始，而非原始表？

**背景**: 可以让 AI 直接查询原始数据仓库表。

**决策**: 强制优先查询语义层（Metric Store）。

**理由**:
1. **歧义消解**: 原始表字段名技术化（user_id_v2），缺少业务语义
2. **逻辑复用**: JOIN 和过滤逻辑在语义层统一封装，避免每次重写
3. **数据治理**: 语义层由领域专家维护，确保定义一致性
4. **错误减少**: 80% 的查询错误源于实体歧义，语义层解决根因

**权衡**: 需要前期投入建设语义层（2-4 周），但长期准确率提升 21% → 95%

**状态**: 已采纳

---

### ADR-002: 为什么不让 AI 自动生成 Skills？

**背景**: 可以让 Claude 扫描历史查询自动生成指标定义。

**决策**: 禁止自动生成知识域 Skill，必须由领域专家设计。

**理由**:
1. **歧义永久化**: 历史数据本身存在歧义（"活跃用户"有 10 种定义），AI 学习会固化混乱
2. **准确率天花板**: Anthropic 实测自动生成仅达 ~40% 准确率，人工设计可达 95%
3. **维护成本**: 自动生成的 Skill 需要大量清理，不如从一开始就正确设计
4. **责任归属**: 人工设计的 Skill 有明确负责人，出错时可追溯

**权衡**: 需要领域专家时间投入，但这是必要成本

**例外**: 通用编程 Skill（如 PyTorch/pandas 用法）可以自动生成

**状态**: 已采纳

> 详见: `memory/error-patterns.md` EP-007

---

### ADR-003: 为什么需要对抗性审查？

**背景**: QA Reviewer 已覆盖功能和质量检查。

**决策**: 对核心功能增加对抗性审查步骤。

**理由**:
1. **边界条件覆盖**: 普通 Review 验证"代码是否符合规范"，对抗性审查验证"边界情况会不会崩溃"
2. **量化效果**: 实测准确率 +6%，虽然成本 +32% token / +72% 延迟，但对核心功能值得
3. **失败成本**: 认证/支付/数据迁移等功能的失败成本远高于审查成本

**权衡**: 不适用于 UI 样式、文档更新、简单 bug 修复

**适用场景**: 
- ✅ 核心业务逻辑（支付/认证/数据迁移）
- ✅ 对外 API 变更
- ✅ 安全敏感功能
- ❌ UI 样式调整
- ❌ 文档更新

**状态**: 已采纳

> 详见: `docs/QA-SYSTEM.md` 对抗性审查章节

## 实施检查清单

### 启动新知识系统时

- [ ] **Layer 1**: 数据基础设施已就绪（仓库、权限、ETL）
- [ ] **Layer 2**: 至少完成语义层核心指标定义（10-20 个关键指标）
- [ ] **Layer 2**: 业务术语表已创建，高频歧义词已列出
- [ ] **Layer 3**: Knowledge Skill 索引至少 10 个参考文档
- [ ] **Layer 3**: Runbook Skill 定义四步强制流程
- [ ] **Layer 4**: 准备 20-50 个典型查询作为离线评估集
- [ ] **Layer 4**: 快照锁定 ground truth 到特定日期
- [ ] **验证**: 首次评估准确率 ≥70%（否则回到 Layer 2 补充歧义消解）

### 维护运行中的系统

- [ ] **每月审计**: 检查语义层定义是否过期
- [ ] **每月审计**: 检查歧义消解表是否覆盖新出现的歧义词
- [ ] **每周扫描**: 运行主动自修复协议，检测用户纠错信号
- [ ] **每次 schema 变更**: 同步更新语义层和参考文档
- [ ] **每次 Skill 修改**: 重跑离线评估，记录准确率变化
- [ ] **季度审计**: 评估集是否需要扩充（边际收益 <2% 时停止）

### 扩展到新领域时

- [ ] **评估数据 vs 代码**: 新领域是唯一答案（数据）还是开放解空间（代码）？
- [ ] **如果是数据类**: 遵循四层架构，强制语义层优先
- [ ] **如果是代码类**: Skill 可以更灵活，提供模式但不强制唯一路径
- [ ] **如果是混合类**: 分层处理（如 API 文档查询 = 数据类，代码示例生成 = 代码类）

## 反模式识别

| 反模式 | 表现 | 后果 | 正确做法 |
|--------|------|------|---------|
| **跳过语义层** | "我直接查原始表更快" | 歧义未消解，准确率 <60% | 强制语义层优先，绕过需书面审批 |
| **自动生成 Skills** | "让 AI 扫描历史数据生成指标定义" | 歧义永久化，准确率卡在 40% | 人工设计语义层，AI 负责路由 |
| **堆信息量** | "给 AI 访问 1000 个文档会更好" | 准确率提升 <1%，token 消耗暴增 | 精简到 30-50 个关键文档 + 强制歧义消解 |
| **无来源标注** | 回答不附带数据来源和置信度 | 无法追溯、无法验证、无法改进 | 强制输出方法论和置信度 |
| **跳过离线评估** | "我觉得应该没问题" | 上线后发现准确率 <70% | 准确率 ≥90% 才可上线 |
| **静默处理歧义** | AI 自己猜测"用户"指什么 | 看似流畅实则错误 | 强制澄清流程，不允许猜测 |

## 适用场景

### ✅ 强烈推荐

- 数据分析和 BI 系统
- API 文档查询系统
- 业务知识库（产品规格、流程文档）
- 合规和审计查询
- 客户支持知识库

### ⚠️ 部分适用

- 代码库导航（CodeGraph 已提供轻量方案）
- 技术文档查询（可以简化为 2 层：文档 + Skill）
- 故障排查 Runbook（需要 Layer 3，但 Layer 2 可简化）

### ❌ 不适用

- 开放式创意任务（代码实现、内容创作）
- 实时数据流处理（架构假设是批处理/查询）
- 无需歧义消解的场景（如纯计算任务）

## 成本收益分析

### 前期投入

| 投入项 | 估算时间 | 负责人 |
|--------|---------|--------|
| 语义层设计（10-20 个核心指标） | 2-4 周 | 资深分析师 |

# 上下文工程指南 (Context Engineering Guide)

> 上下文工程 = 让 AI 在正确的时间获得正确的信息，而不是更多的信息。
> 核心指标: **信噪比 (SNR)** — 每个 token 对任务完成的贡献度。

---

## 核心原则: 信噪比优先

```
信噪比 = 与当前任务相关的信息 / 上下文中的总信息量
```

**实践含义**:
- 加载 10K tokens 的精准 Spec 优于 50K tokens 的泛化文档
- 上下文不是越多越好；无关信息稀释注意力，降低输出质量
- 每次向上下文注入内容前自问: "这对当前任务的信噪比贡献是正还是负？"

---

## 三大失效模式

### 1. 上下文投毒 (Context Corruption)

**定义**: 错误、过时或矛盾的信息污染上下文，导致 AI 产出系统性偏差。

**来源**:
- 过时的 CLAUDE.md 规则（已被新规则替代但未删除）
- 旧会话的错误结论被压缩摘要保留
- 多个文件中对同一概念的矛盾描述
- 终端输出中的错误信息被当作事实

**检测信号**:
- AI 引用了已废弃的规则或配置
- 产出与当前最佳实践矛盾
- 解决方案基于过时的假设

**防治**:
- 废弃规则添加 `> ⚠️ 已废弃` 标记（参见 lessons-learned.md #003）
- 定期审计 CLAUDE.md，删除无效条目
- 同一概念只在一处定义，其他地方引用

### 2. 注意力漂移 (Attention Drift)

**定义**: 随着上下文增长或批量任务推进，AI 逐渐"忘记"早期的约束和规范。

**来源**:
- 长对话中上下文压缩丢失关键约束
- 批量任务（>30项）中任务规范被新内容挤出注意力窗口
- 多次工具调用的输出淹没原始指令

**检测信号**:
- 格式从第 N 项开始漂移
- 跳过之前一直执行的分析步骤
- 忽略早期设定的约束条件

**防治**:
- 批量任务使用持久化规划文件（task_plan.md），每轮开头重读约束
- 关键约束写在 CLAUDE.md（每次调用都加载），而非对话中
- 长任务使用 Ralph 的检查点机制，在检查点重新注入约束
- Skills 路由规则属于关键约束，长上下文下需特别防护（见 BP-019）

### 3. 上下文衰减 (Context Rot)

**定义**: 随着上下文增长，模型注意力被分散到更多 tokens，性能逐渐退化。上下文虽未溢出，但质量已下降。

**量化数据** (Anthropic 官方, 1M 窗口):
- 衰减起始: ~300-400k tokens
- 高度依赖任务类型（调试/架构 > 格式化/文档）

**来源**:
- 长会话中累积了大量工具输出、中间过程
- 早期无关内容持续占据注意力权重
- 在衰减最严重的时刻触发自动压缩 → 坏压缩

**检测信号**:
- Claude 开始忽略之前明确设定的约束
- 输出质量明显下降（逻辑错误、遗漏条件）
- 压缩后丢失了"看似无关但实际重要"的信息

**防治**:
- ~250k tokens 时主动 `/compact` 并附上未来方向
- 优先用 /rewind 丢弃失败尝试，而非在上下文中纠正
- Subagent 隔离会产生大量中间输出的探索性工作
- 新任务开新会话，而非在旧上下文中叠加

## 上下文卫生 5 习惯

### 习惯 1: 精准加载，按需注入

```
❌ 启动时加载所有 Agent 定义（~50K tokens）
✅ 启动时加载 INDEX.md（~600 tokens），按需加载 1 个 Agent（~3K tokens）
```

太一元系统已实现此模式: Skills INDEX → 按需加载 SKILL.md

### 习惯 2: 压缩终端输出

```
❌ 直接将 npm test 的 500 行输出注入上下文
✅ 提取关键信息: "3 tests failed: auth.test.ts:45, user.test.ts:78, api.test.ts:112"
```

工具调用后，只保留对下一步有用的信息，丢弃冗余的成功输出。

### 习惯 3: 单一事实源

每个概念只在一个文件中权威定义:
- Hooks 配置格式 → `CLAUDE.md` 第八节
- Agent 路由表 → `CLAUDE.md` 第零节
- 经验教训 → `memory/lessons-learned.md`

其他文件引用，不复制。违反此规则导致矛盾和投毒（参见 CLAUDE.md 超限事件 lessons-learned #011）。

### 习惯 4: 定期审计固定上下文税

"固定上下文税"指每次对话启动时必须加载的内容。审计清单:

| 固定加载项 | 当前大小 | 审计问题 |
|-----------|---------|---------|
| CLAUDE.md | ~39K chars | 是否有可移出的内容？ |
| intent-state.json | ~50 chars | 合理 |
| memory/MEMORY.md | ~200 行 | 是否有过期条目？ |

**审计频率**: 每月一次，或 CLAUDE.md 接近 39K 时。

**目标**: CLAUDE.md ≤39,500 chars（40K 限制减 500 余量）。

### 习惯 5: 避免上下文内容重复

```
❌ CLAUDE.md 和 docs/X.md 各写一份完整的规则
✅ CLAUDE.md 写摘要（2-3 行），指向 docs/X.md 获取详情
```

重复内容的代价: 增大固定税 + 修改时漏改一处 → 矛盾 → 投毒。

## 上下文衰减与量化阈值

### Context Rot 量化

模型性能随上下文增长而退化（Context Rot），原因: 注意力被分散到更多 tokens，旧的无关内容开始干扰当前任务。

**Anthropic 官方实测数据** (1M 上下文窗口):
- 衰减开始: **~300-400k tokens**
- 高度依赖任务类型，非硬性规则
- 低智力敏感任务（如格式化、文档生成）对衰减容忍度更高
- 高智力敏感任务（如复杂调试、架构决策）更早出现质量下降

**实践规则**:
- 1M 窗口 ≠ 可以无脑填充到 1M
- 主动在 ~250k tokens 时评估是否需要压缩或新开会话
- 上下文长度是成本（注意力稀释），不是免费资源

## 会话管理决策框架

每个回合结束、用户准备发送下一条消息时，是一个**决策分支点**。以下是 5 种选择及其适用场景:

### 决策矩阵

| 操作 | 命令 | 适用场景 | 上下文代价 |
|------|------|---------|-----------|
| **Continue** | 直接发消息 | 同一任务延续，上下文仍高度相关 | 继续累积 |
| **Rewind** | `/rewind` (Esc Esc) | 方向错误，回退重试 | 丢弃回退点之后的所有上下文 |
| **Clear** | `/clear` | 新任务，或上下文已严重衰减 | 全部清空，需手写交接摘要 |
| **Compact** | `/compact [指令]` | 同任务但上下文过重 | 有损压缩，信任模型判断保留什么 |
| **Subagent** | Agent tool | 子任务会产生大量中间输出 | 隔离在独立上下文，仅回传结论 |

### 规则 1: 新任务 → 新会话

**通用规则**: 开始新任务时，开始新会话。

**灰色地带**: 相关但不同的任务（如实现功能后写文档）。此时上下文中的文件读取仍有价值，避免重新读取的成本。权衡:
- 低智力敏感任务 + 上下文仍相关 → 继续
- 高智力敏感任务 + 上下文已 >200k → 新会话，手写交接

### 规则 2: 优先 Rewind 而非纠正

**反直觉但重要**: 当 Claude 尝试的方法失败时，不要在当前上下文中纠正，而是回退重试。

```
❌ "那个方法不行，试试 X"  — 失败的尝试和中间输出留在上下文中，浪费注意力
✅ /rewind 回到文件读取后，重新提示 "不要用方法 A，foo 模块不暴露那个接口 — 直接用 B"
```

**原理**: 失败的尝试是上下文噪声。回退丢弃噪声，从干净状态重新开始。

**进阶**: "summarize from here" — 让 Claude 总结已学到的教训，生成交接消息给"未来的自己"，然后从该点继续。

### 规则 3: Compact vs Clear 的选择

| 维度 | /compact | /clear + 新会话 |
|------|----------|---------------|
| 谁决定保留什么 | 模型（自动） | 人类（手动摘要） |
| 工作量 | 零 | 需要手写交接 |
| 信息保真 | 有损，可能丢失看似无关但重要的细节 | 精确，只保留你判断重要的 |
| 风险 | 坏压缩（见下文） | 遗漏自己未意识到的关键信息 |

**最佳实践**:
- 使用 `/compact focus on X, drop Y` 引导压缩方向
- 在上下文尚未严重衰减时**主动压缩**（~250k），而非等自动压缩触发
- 自动压缩在长调试会话末尾触发时最容易出问题

### 坏压缩 (Bad Compact) 的根因

**核心问题**: 模型在最不智能的时刻（上下文衰减后）被要求判断什么信息对未来有用。

**典型场景**:
1. 长调试会话 → 自动压缩聚焦于调试内容
2. 用户下一条消息: "顺便修一下 bar.ts 里那个 warning" → 但该 warning 已被压缩丢弃
3. 因为调试方向占主导，看似无关的 warning 被判定为低优先级

**防治**:
- 在 ~250k tokens 时主动 `/compact` 并附上未来方向说明
- 压缩指令中明确提及待办事项: `/compact keep the bar.ts warning and TODO list`
- 用 Subagent 隔离探索性工作，避免大量中间输出撑大上下文

### 规则 4: Subagent 作为上下文隔离手段

**判断标准**: "我还需要这些工具输出吗，还是只需要结论？"

**适用场景**:
- 验证工作: "起一个 subagent 根据这个 spec 验证结果"
- 跨代码库参考: "让 subagent 读那个代码库的 auth 实现，总结后我来实现"
- 文档生成: "让 subagent 根据 git diff 写文档"
- 大规模搜索: 探索性搜索产生大量中间结果，只需最终答案

**原理**: Subagent 获得独立的新鲜上下文窗口，可以自由产出中间输出，只有最终报告回传到父上下文。天然实现"中间过程隔离，结论汇入"。

## HAM 分层规则模式（参考技巧）

HAM (Hierarchical Attention Management) 是一种将指令按层级组织的模式，确保 AI 在不同场景下关注正确的规则层:

```
Level 0 — 绝对约束（Always）
  安全红线、授权边界、不可违反的规则
  位置: CLAUDE.md 顶部

Level 1 — 流程约束（Process）
  编排策略、QA 流程、Ralph 循环规则
  位置: CLAUDE.md 对应章节

Level 2 — 领域知识（Domain）
  Agent 定义、Skill 内容、技术规范
  位置: 按需加载的 agents/*.md 和 skills/*/SKILL.md

Level 3 — 上下文信息（Context）
  当前任务描述、对话历史、工具输出
  位置: 对话上下文
```

**实践意义**: 当规则冲突时，低 Level 优先。例如 Level 0 的安全约束始终优先于 Level 2 的领域知识建议。

## 动态 Skill 路由思想

太一元系统的 Skills 渐进式披露机制体现了动态路由思想:

```
用户请求 → 匹配 INDEX.md 中的 Skill → 仅加载匹配的 SKILL.md → 执行
```

**优化方向**: 路由不仅看"匹配什么"，还看"不加载什么"。
- 前端任务不加载 pytorch Skill
- 数据分析任务不加载 seedance Skill
- 负向路由减少的 token 消耗 = 信噪比提升

## 检查清单: 向上下文添加内容前

1. [ ] 这个信息对当前任务的 SNR 是正贡献吗？
2. [ ] 这个信息是否已经在上下文中存在（避免重复）？
3. [ ] 这个信息是否是最新版本（避免投毒）？
4. [ ] 能否用 3 行摘要替代 30 行原文？
5. [ ] 是否可以延迟加载到实际需要时？

## 检查清单: 会话管理决策

1. [ ] 当前任务是否已结束？（是 → 新会话）
2. [ ] 上下文是否 >250k tokens？（是 → 主动 compact 或新会话）
3. [ ] Claude 的上一次尝试是否失败？（是 → 优先 /rewind 而非纠正）
4. [ ] 下一步是否会产生大量中间输出？（是 → 考虑 Subagent 隔离）
5. [ ] 是否需要 compact？如果是，是否附上了未来方向说明？（避免坏压缩）

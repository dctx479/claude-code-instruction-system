## 常见使用场景

### 场景A: 我已经写好代码，只想做QA

```bash
# 直接调用qa-reviewer
/agent qa-reviewer "验证我的代码"

# 系统会:
1. 检查代码质量
2. 运行测试
3. 生成QA报告
4. 如果有P2问题，自动修复
5. 给出最终评分和建议
```

### 场景B: 我想先看规范再开发

```bash
# 调用spec-writer
/agent spec-writer "用户注册功能"

# 系统会:
1. 生成详细的技术规范
2. 包含数据模型、API、前端设计
3. 定义明确的验收标准

# 你审查规范，确认后开始开发
按照规范实现代码

# 开发完成后QA验证
/agent qa-reviewer "验证注册功能"
```

### 场景C: 我的代码有很多小问题

```bash
# 运行QA审查
/agent qa-reviewer "检查代码"

# 系统发现20个P2问题 (代码格式、类型标注等)

# 自动修复
系统会自动调用qa-fixer
- 修复18个问题
- 2个问题需人工处理
- 重新审查，评分提升

# 你只需要关注少数需要人工处理的问题
```

## 理解评分系统

### 评分构成 (总分100)

```
功能完整性 (40分)
├── 所有功能已实现 (20分)
├── 边界条件处理 (10分)
└── 异常场景处理 (10分)

代码质量 (30分)
├── 符合代码规范 (10分)
├── 类型定义完整 (5分)
├── 错误处理健全 (5分)
├── 代码可读性 (5分)
└── 无技术债务 (5分)

测试覆盖 (20分)
├── 单元测试覆盖率 ≥80% (10分)
├── 集成测试 (5分)
└── 边界测试 (5分)

性能指标 (5分)
├── API响应时间 <200ms (3分)
└── 前端加载 <1s (2分)

安全检查 (5分)
├── 输入验证 (2分)
└── 无安全漏洞 (3分)
```

### 评分等级

```
90-100分: 优秀 ⭐⭐⭐⭐⭐
  - 立即发布
  - 代码质量极高
  - 可作为最佳实践

80-89分: 良好 ⭐⭐⭐⭐
  - 可以发布
  - 质量达标
  - 有小幅优化空间

70-79分: 改进 ⭐⭐⭐
  - 建议优化后发布
  - 存在一些问题
  - 功能基本完整

<70分: 不及格 ⭐⭐
  - 需要返工
  - 问题较多
  - 不建议发布
```

## 理解问题严重程度

### 🔴 P0 - 严重 (阻塞发布)

**定义**: 核心功能不可用，或存在数据安全风险

**示例**:
- 核心功能缺失 (登录功能没实现)
- 严重安全漏洞 (SQL注入)
- 数据丢失风险 (删除操作没确认)
- 系统崩溃 (未捕获的异常)

**处理**: 必须人工修复，阻塞发布

### 🟡 P1 - 重要 (建议修复)

**定义**: 功能受限，用户体验下降

**示例**:
- 边界情况未处理 (空输入没验证)
- 性能不达标 (响应时间>500ms)
- 测试覆盖不足 (<60%)
- 错误处理缺失 (没有try-catch)

**处理**: 优先修复，可能需要人工或自动

### 🟢 P2 - 轻微 (可选优化)

**定义**: 代码质量问题，不影响功能

**示例**:
- 代码格式问题 (缩进不统一)
- 类型标注缺失 (函数参数没类型)
- 代码重复 (相同逻辑重复3次)
- 注释不足 (复杂逻辑没注释)

**处理**: 自动修复，不影响发布

## 技巧和建议

### 技巧1: 善用规范模板

```bash
# 不要从零开始
❌ 自己写完整的设计文档

# 使用规范模板
✅ cp specs/SPEC-TEMPLATE.md specs/SPEC-your-feature.md
✅ 或让spec-writer自动生成
```

### 技巧2: 验收标准要具体

```bash
# 模糊的标准
❌ "用户可以登录"

# 具体的标准
✅ "使用有效邮箱和密码可以登录成功"
✅ "错误的密码返回401状态码"
✅ "登录成功返回JWT token"
```

### 技巧3: 先修复P0，再优化P2

```bash
问题优先级:
1. P0问题 - 立即修复，阻塞发布
2. P1问题 - 优先修复，可能阻塞
3. P2问题 - 自动修复，不阻塞

不要陷入完美主义:
❌ 花2小时优化P2问题
✅ 先确保P0问题解决
```

### 技巧4: 利用自动修复

```bash
# 让系统帮你做琐事
代码格式问题 - 自动修复
类型标注缺失 - 自动添加
未使用的导入 - 自动移除

# 你专注于核心问题
业务逻辑正确性
架构合理性
用户体验优化
```

### 技巧5: 持续改进

```bash
# 每次QA后回顾
- 哪些问题重复出现？
- 如何在规范阶段预防？
- 自动修复策略是否有效？

# 更新流程
- 完善规范模板
- 优化检测规则
- 改进自动修复策略
```

## 常见问题

### Q: 评分不到80分怎么办？

A: 看问题类型:
- P0问题: 必须修复
- P1问题: 优先修复
- P2问题: 系统会自动修复

修复后重新运行qa-reviewer

### Q: 自动修复失败了怎么办？

A: 不用担心:
1. 系统会自动回滚
2. 标记为需人工处理
3. 在FIX-REPORT.md中说明原因

你可以按照建议手动修复

### Q: 我不想写规范，能直接写代码吗？

A: 可以，但不推荐:
- 没规范: 开发中需求可能不清晰
- 有规范: QA有明确的验收标准

建议: 至少让spec-writer生成一个基础规范

### Q: QA审查需要多久？

A: 通常很快:
- 简单功能: 2-5分钟
- 中等复杂: 5-10分钟
- 复杂功能: 10-15分钟

主要时间在运行测试

### Q: 能不能跳过QA直接发布？

A: 技术上可以，但强烈不建议:
- QA能发现90%+的常见问题
- 自动修复能解决70%+的小问题
- 节省后期修bug的时间

投入10分钟QA，节省数小时debug

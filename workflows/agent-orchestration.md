# Agent 编排与自进化工作流

## 概述

本工作流定义了如何完美驾驭多Agent协作，以及系统如何自我进化和改进。

## 一、Agent 编排模式

### 1.1 编排策略决策树

```
                    ┌─────────────────┐
                    │   接收任务      │
                    └────────┬────────┘
                             │
                    ┌────────▼────────┐
                    │  任务可分解?    │
                    └────────┬────────┘
                       /           \
                     是             否
                     /               \
            ┌───────▼───────┐    ┌────▼────┐
            │ 子任务独立?   │    │单Agent  │
            └───────┬───────┘    └─────────┘
               /         \
             是           否
             /             \
    ┌───────▼───────┐  ┌────▼────────────┐
    │   PARALLEL    │  │   有依赖链?     │
    │   并行策略    │  └────┬────────────┘
    └───────────────┘       /          \
                          是            否
                          /              \
              ┌──────────▼──────┐  ┌─────▼─────────┐
              │   SEQUENTIAL    │  │  需要专家?    │
              │   串行策略      │  └─────┬─────────┘
              └─────────────────┘        /        \
                                       是          否
                                       /            \
                          ┌───────────▼───┐  ┌──────▼────────┐
                          │  HIERARCHICAL │  │ 需要多视角?   │
                          │  层级策略     │  └──────┬────────┘
                          └───────────────┘         /       \
                                                  是         否
                                                  /           \
                                    ┌────────────▼──┐  ┌──────▼──────┐
                                    │ COLLABORATIVE │  │ COMPETITIVE │
                                    │ 协作策略      │  │ 竞争策略    │
                                    └───────────────┘  └─────────────┘
```

### 1.2 各策略详解

#### PARALLEL (并行策略)
```
最大化并行度，适用于独立子任务

执行模式:
Task → 分解 → [T1] [T2] [T3] → 并行执行 → 合并结果
                 ↓    ↓    ↓
              [A1] [A2] [A3]

关键配置:
- 最大并行度: 5
- 超时: 120s/任务
- 重试: 2次

使用命令: /project:parallel
```

#### SEQUENTIAL (串行策略)
```
保证依赖顺序，适用于链式任务

执行模式:
Task → [T1] → result1 → [T2] → result2 → [T3] → Final
        ↓                 ↓                 ↓
       [A1]              [A2]              [A3]

关键点:
- 每步验证输出
- 支持中途调整
- 保留中间状态
```

#### HIERARCHICAL (层级策略)
```
专家指导，适用于需要决策的任务

执行模式:
Task → [Specialist] → 分解决策
              ↓
       ┌──────┼──────┐
       ↓      ↓      ↓
   [Worker] [Worker] [Worker]
       ↓      ↓      ↓
       └──────┴──────┘
              ↓
      [Specialist] → 审核整合

角色分工:
- Specialist: 决策、审核、整合
- Worker: 执行、报告
```

#### COLLABORATIVE (协作策略)
```
多专家协作，适用于跨领域问题

执行模式:
Task → [Expert1] ←──→ [Expert2] ←──→ [Expert3]
           │              │              │
           └──────────────┼──────────────┘
                          ↓
                     Consensus

协作方式:
- 各专家独立分析
- 交换观点和发现
- 讨论达成共识
- 整合最终方案
```

#### COMPETITIVE (竞争策略)
```
多方案竞争，适用于探索性任务

执行模式:
Task → [Agent1] → Solution1 ─┐
     → [Agent2] → Solution2 ─┼→ 评估 → Best
     → [Agent3] → Solution3 ─┘

评估维度:
- 正确性
- 效率
- 可维护性
- 成本
```

#### SWARM (群体策略)
```
大规模任务，适用于批量处理

执行模式:
Task → 分解为N个小任务
          ↓
    ┌─────┼─────┬─────┬─────┐
    ↓     ↓     ↓     ↓     ↓
   [W1]  [W2]  [W3]  [W4]  [W5]  ... [Wn]
    ↓     ↓     ↓     ↓     ↓
    └─────┴─────┴─────┴─────┘
              ↓
         聚合结果

使用命令: /project:swarm
```

## 二、自进化机制

### 2.1 进化触发条件

| 触发条件 | 优先级 | 自动/手动 |
|----------|--------|-----------|
| 任务失败 | 高 | 自动 |
| 重复错误 (≥2次) | 高 | 自动 |
| 用户纠正 | 中 | 自动 |
| 发现更优方案 | 中 | 手动触发 |
| 复杂任务完成 | 低 | 自动回顾 |

### 2.2 进化流程

```
┌─────────────────┐
│   触发事件      │
└────────┬────────┘
         │
┌────────▼────────┐
│   分析根因      │
│ - 什么出错了?   │
│ - 为什么?       │
│ - 影响多大?     │
└────────┬────────┘
         │
┌────────▼────────┐
│   生成方案      │
│ - 如何防止?     │
│ - 更新什么?     │
└────────┬────────┘
         │
┌────────▼────────┐
│   更新配置      │
│ - CLAUDE.md     │
│ - agents/*.md   │
│ - memory/*.md   │
└────────┬────────┘
         │
┌────────▼────────┐
│   验证效果      │
│ - 测试更新      │
│ - 确认有效      │
└────────┬────────┘
         │
┌────────▼────────┐
│   记录经验      │
│ - lessons-learned│
│ - best-practices│
└─────────────────┘
```

### 2.3 可更新的配置

```yaml
CLAUDE.md:
  - 核心规则
  - 决策权限
  - 命令列表

agents/*.md:
  - 系统提示词
  - 工具权限
  - 能力描述

commands/*.md:
  - 命令流程
  - 输出格式

memory/:
  - lessons-learned.md  # 经验教训
  - error-patterns.md   # 错误模式
  - best-practices.md   # 最佳实践
  - agent-performance.md # 性能追踪
```

### 2.4 经验沉淀格式

```markdown
[具体描述]

### 根因分析

- 文件: [路径]
- 变更: [内容]

### 验证方法
[如何验证]

#tag1 #tag2
```

## 三、完整工作流示例

### 3.1 复杂功能开发

```markdown
用户请求: "开发用户认证系统"

1. Orchestrator 分析任务
   - 可分解: 是
   - 依赖关系: 部分依赖
   - 需要专家: 是
   → 选择: HIERARCHICAL

2. 任务分解
   - T1: 需求分析 (architect)
   - T2: 数据库设计 (architect)
   - T3: API实现 (并行)
     - T3.1: 登录API (worker)
     - T3.2: 注册API (worker)
     - T3.3: JWT处理 (worker)
   - T4: 测试 (code-reviewer)
   - T5: 安全审计 (security-analyst)

3. 执行
   - T1 → T2 (串行)
   - T3.1, T3.2, T3.3 (并行)
   - T4, T5 (并行)

4. 结果整合
   - architect 审核所有输出
   - 解决冲突
   - 生成最终交付

5. 经验沉淀
   - 记录成功模式
   - 更新 best-practices.md
```

### 3.2 大规模重构

```markdown
用户请求: "将所有 CommonJS 迁移到 ES Modules"

1. Orchestrator 分析任务
   - 文件数量: 200+
   - 任务独立: 是
   → 选择: SWARM

2. 任务分解
   - 扫描所有 .js 文件
   - 每个文件一个任务

3. 配置
   - 并行度: 10
   - Worker模型: haiku
   - 超时: 60s/文件

4. 执行
   - 批量处理
   - 实时进度追踪
   - 失败重试

5. 结果聚合
   - 统计成功/失败
   - 验证整体构建
   - 生成迁移报告
```

## 五、性能优化建议

### 5.1 提高并行效率
- 充分分解任务
- 识别独立子任务
- 合理设置并行度

### 5.2 降低失败率
- 明确任务描述
- 设置合理超时
- 配置重试策略

### 5.3 优化成本
- 简单任务用 haiku
- 复杂任务用 sonnet
- 关键决策用 opus

### 5.4 加速进化
- 及时记录经验
- 定期回顾优化
- 主动触发进化

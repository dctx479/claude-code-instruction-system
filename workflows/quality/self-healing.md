# 自我修复质量循环 (Self-Healing QA Loop)

## 概述

自我修复质量循环是一个自动化的质量保障流程，通过QA Agent对（qa-reviewer + qa-fixer）实现代码质量的持续改进和自动修复。

## 核心理念

> **发现问题 → 自动修复 → 重新验证 → 持续改进**

系统能够自主发现代码质量问题，自动修复低风险问题，并触发重新验证，形成闭环。

## 工作流程图

```
┌─────────────────────────────────────────────────────────┐
│                    开发完成代码                          │
└─────────────────────┬───────────────────────────────────┘
                      │
                      ↓
┌─────────────────────────────────────────────────────────┐
│  QA Reviewer: 质量审查                                   │
│  - 读取规范 (SPEC-xxx.md)                               │
│  - 验证功能完整性                                        │
│  - 检查代码质量                                          │
│  - 测试覆盖率分析                                        │
│  - 性能和安全检查                                        │
└─────────────────────┬───────────────────────────────────┘
                      │
                      ↓
┌─────────────────────────────────────────────────────────┐
│  生成 QA-REPORT.md                                       │
│  - 总体评分 (0-100)                                      │
│  - 问题清单 (P0/P1/P2)                                   │
│  - 修复建议                                              │
└─────────────────────┬───────────────────────────────────┘
                      │
                      ↓
                  [评分判断]
                      │
         ┌────────────┼────────────┐
         │            │            │
    ≥80分(PASS)   <80分(FAIL)   有P2问题
         │            │            │
         ↓            ↓            ↓
   ┌─────────┐  ┌──────────┐  ┌─────────────────┐
   │ 发布就绪 │  │ 阻塞发布  │  │ QA Fixer: 自动修复│
   └─────────┘  │ 人工处理  │  │ - 过滤可修复问题   │
                └──────────┘  │ - 按优先级修复     │
                              │ - 验证修复效果     │
                              └─────────┬───────────┘
                                        │
                                        ↓
                              ┌─────────────────┐
                              │ 生成 FIX-REPORT.md│

                              │ - 成功/失败清单   │
                              └─────────┬─────────┘

                                        ↓
                              ┌─────────────────┐
                              │ 触发重新审查     │
                              │ (返回QA Reviewer)│
                              └─────────┬─────────┘
                                        │
                                        ↓
                              [持续循环直到PASS]
```

## 三种循环模式

### 模式1: 快速通过 (Happy Path)
```
开发 → QA Reviewer → PASS (≥80分) → 发布
耗时: ~10分钟
```

**触发条件**:
- 代码质量高
- 测试覆盖充分
- 无严重问题

### 模式2: 自我修复循环
```
开发 → QA Reviewer → FAIL (<80分, 但有P2问题)
     → QA Fixer → 自动修复
     → QA Reviewer → PASS → 发布
耗时: ~30-45分钟
```

**触发条件**:
- 主要是低风险问题 (代码格式、类型标注等)
- 无阻塞性问题

### 模式3: 人工介入
```
开发 → QA Reviewer → FAIL (<80分, 有P0/P1问题)
     → 人工修复 → QA Reviewer → PASS → 发布
耗时: 依具体情况而定
```

**触发条件**:
- 存在严重问题 (P0)
- 重要问题需要业务理解 (P1)
- 架构调整需求

## QA Agent对协作

### QA Reviewer (审查者)

**输入**:
- 功能规范: `specs/SPEC-xxx.md`
- 代码实现: 整个代码库

**输出**:
- QA报告: `QA-REPORT.md`

**职责**:
- 功能完整性验证 (40分)
- 代码质量评估 (30分)
- 测试覆盖验证 (20分)
- 性能指标检查 (5分)
- 安全检查 (5分)

**决策规则**:
```python
if score >= 80:
    return "PASS"
elif has_p0_issues:
    return "FAIL - 需人工处理"
elif has_p2_issues:
    return "TRIGGER_FIXER"
else:
    return "FAIL - 需人工处理"
```

### QA Fixer (修复者)

**输入**:
- QA报告: `QA-REPORT.md`

**输出**:
- 修复报告: `FIX-REPORT.md`
- 代码变更: Git diff

**职责**:
- 识别可自动修复的问题
- 按优先级执行修复
- 验证修复效果
- 触发重新审查

**可修复问题类型**:
- ✅ 代码格式 (Prettier/ESLint)
- ✅ 类型标注缺失
- ✅ 简单的错误处理
- ✅ 未使用的导入
- ✅ 简单的性能优化

**不可修复问题**:
- ❌ 业务逻辑错误
- ❌ 架构调整
- ❌ 复杂安全漏洞

## 评分体系

### 总分: 100分
```
功能完整性: 40分
  - 所有功能已实现 (20)
  - 边界条件处理 (10)
  - 异常场景处理 (10)

代码质量: 30分
  - 符合代码规范 (10)
  - 类型定义完整 (5)
  - 错误处理健全 (5)
  - 代码可读性 (5)
  - 无技术债务 (5)

测试覆盖: 20分
  - 单元测试覆盖率 (10)
  - 集成测试 (5)
  - 边界测试 (5)

性能指标: 5分
  - API响应时间 (3)
  - 前端加载速度 (2)

安全检查: 5分
  - 输入验证 (2)
  - 无安全漏洞 (3)
```

### 通过线: ≥80分
```
优秀: 90-100分 - 立即发布
良好: 80-89分 - 可以发布
改进: 70-79分 - 建议优化后发布
不及格: <70分 - 需要返工
```

## 问题严重程度分类

### 🔴 P0 - 严重 (阻塞发布)
**影响**: 核心功能不可用，数据安全风险

**示例**:
- 核心功能缺失
- 严重安全漏洞 (SQL注入)
- 数据丢失风险
- 系统崩溃

**处理**: 必须人工修复，阻塞发布

### 🟡 P1 - 重要 (建议修复)
**影响**: 功能受限，用户体验下降

**示例**:
- 边界情况未处理
- 性能不达标
- 测试覆盖不足
- 错误处理缺失

**处理**: 根据复杂度决定人工或自动修复

### 🟢 P2 - 轻微 (可选优化)
**影响**: 代码质量问题，不影响功能

**示例**:
- 代码风格问题
- 小幅性能优化
- 代码重复
- 注释不足

**处理**: 优先自动修复

## 使用示例

### 场景1: 完美代码
```bash
# 开发完成
git commit -m "feat: 实现用户认证"

# 触发QA审查
/agent qa-reviewer "验证用户认证功能"

# 输出
✅ QA审查通过
评分: 92/100
所有验收标准已满足
可以发布

# 结果
无需修复，直接发布
```

### 场景2: 代码格式问题
```bash
# 开发完成
git commit -m "feat: 实现用户认证"

# 触发QA审查
/agent qa-reviewer "验证用户认证功能"

# 输出
⚠️ QA审查结果: 75/100
发现8个轻微问题 (P2):
- 代码格式不规范 (5个文件)
- 缺少类型标注 (3处)

建议: 触发自动修复

# 自动触发QA Fixer
/agent qa-fixer "修复QA报告中的问题"

# 输出
✅ 自动修复完成
成功修复: 7/8
失败: 1 (需人工处理)

# 重新审查
/agent qa-reviewer "重新验证"

# 输出
✅ QA审查通过
评分: 85/100 (提升10分)
可以发布
```

### 场景3: 严重问题
```bash
# 触发QA审查
/agent qa-reviewer "验证支付功能"

# 输出
❌ QA审查失败: 45/100
发现2个严重问题 (P0):
- 支付回调处理缺失
- 金额验证逻辑错误

阻塞发布，需要人工修复

# 结果
qa-fixer不会自动修复
需要开发人员处理
```

## 最佳实践

### 1. 早期介入
```
✅ 开发完成后立即QA审查
❌ 等到测试阶段才发现问题
```

### 2. 分阶段修复
```
1. QA Fixer自动修复P2问题
2. 人工修复P1问题
3. 最后处理P0问题
```

### 3. 持续改进
```
发现问题 → 修复 → 总结 → 预防
```

### 4. 知识沉淀
```
常见问题 → 自动修复模式库
特殊问题 → 更新规范
```

### 5. 度量跟踪
```
- 首次通过率
- 平均修复时间
- 问题类型分布
- 自动修复成功率
```

### 目标KPI

| 指标 | 目标值 | 测量方法 |

## 故障处理

### 问题1: QA Fixer修复失败
**症状**: 修复后测试失败

**解决方案**:
```bash
1. 自动回滚修复
2. 标记问题为需人工处理
3. 记录失败原因到FIX-REPORT.md
```

### 问题2: 循环修复不收敛
**症状**: 多次修复仍未通过

**解决方案**:
```bash
1. 设置最大循环次数 (3次)
2. 超过后标记为需人工介入
3. 分析根本原因
```

### 问题3: 误报问题
**症状**: QA Reviewer报告不存在的问题

**解决方案**:
```bash
1. 人工审查确认
2. 更新qa-reviewer的检测规则
3. 记录到经验库
```

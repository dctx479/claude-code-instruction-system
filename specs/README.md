# Specs目录 - 功能规范管理

## 目录说明

本目录用于存放所有功能的技术规范文档，实施**Spec-First开发流程**。

## 核心理念

> **先写规范，后写代码**

在开始任何开发工作之前，必须先完成详细的技术规范。规范是开发、测试和验收的唯一标准。

## 目录结构

```
specs/
├── SPEC-TEMPLATE.md          # 规范模板
├── README.md                 # 本文件
├── examples/                 # 示例规范
│   ├── SPEC-user-authentication.md
│   └── SPEC-payment-integration.md
├── active/                   # 进行中的规范
│   └── SPEC-xxx.md
└── archived/                 # 已实现的规范
    └── SPEC-xxx.md
```

## 规范命名规则

```
SPEC-{feature-name}.md
```

**命名示例**:
- `SPEC-user-authentication.md` - 用户认证
- `SPEC-payment-integration.md` - 支付集成
- `SPEC-realtime-chat.md` - 实时聊天
- `SPEC-data-export.md` - 数据导出
- `SPEC-admin-dashboard.md` - 管理后台

**命名原则**:
- 使用小写字母
- 单词用连字符(-)分隔
- 简洁但清晰描述功能
- 避免使用缩写

## 规范编写流程

### 1. 创建规范 (Spec Writer Agent)
```bash
# 使用spec-writer agent创建新规范
使用模板: specs/SPEC-TEMPLATE.md
```

规范包含章节:
1. 需求概述 - 背景、目标、范围
2. 技术方案 - 架构、数据模型、API、前端
3. 实现计划 - 任务拆分、时间估算
4. 验收标准 - 功能、性能、安全、测试
5. 风险与对策 - 技术风险、业务风险

### 2. 审查规范
```bash
# 技术审查清单
- [ ] 需求是否明确？
- [ ] 技术方案是否可行？
- [ ] API设计是否合理？
- [ ] 验收标准是否可测试？
- [ ] 风险是否识别充分？
```

### 3. 批准规范
```markdown
规范状态: ✅ 已批准
```

### 4. 开发实现
```bash
# 按照规范进行开发
严格遵循规范定义的:
- 数据模型
- API接口
- 验收标准
```

### 5. QA验证
```bash
# 使用qa-reviewer验证
对照规范的验收标准进行验证
```

### 6. 归档规范
```bash
# 开发完成后移动到archived/
mv specs/active/SPEC-xxx.md specs/archived/
```

## 规范状态

使用以下状态标记规范的生命周期:

| 状态 | 符号 | 说明 |
|------|------|------|
| 草稿 | ⏳ | 规范编写中 |
| 审核中 | 🔍 | 等待审查批准 |
| 已批准 | ✅ | 可以开始开发 |
| 开发中 | 🚧 | 正在实现 |
| 已实现 | 📦 | 开发完成，待归档 |
| 已归档 | 📚 | 归档到archived/ |

- ✅ 需求明确，减少返工
- ✅ 设计先行，避免技术债务
- ✅ 分工清晰，并行开发
- ✅ 代码审查有标准

### 对QA团队
- ✅ 明确的验收标准
- ✅ 可测试的功能清单
- ✅ 性能和安全基准
- ✅ 自动化测试用例

### 对产品团队
- ✅ 功能范围明确
- ✅ 时间估算准确
- ✅ 风险提前识别
- ✅ 交付质量保障

## 规范模板使用指南

### 必填章节 ⭐
- 1. 需求概述
- 2.2 数据模型设计
- 2.3 API设计
- 4. 验收标准

### 可选章节
- 2.4 前端组件设计 (纯后端项目可省略)
- 3. 实现计划 (简单功能可简化)
- 5. 风险与对策 (低风险功能可简化)

### 章节详细说明

#### 1. 需求概述
**目的**: 确保所有人对需求有统一理解

**关键要点**:
- 背景: 为什么需要这个功能？
- 目标: 功能要达到什么效果？
- 范围: 包含什么？不包含什么？
- 非目标: 明确不做什么，避免范围蔓延

#### 2.2 数据模型设计
**目的**: 定义数据结构和关系

**关键要点**:
- 数据库表结构 (SQL)
- TypeScript类型定义
- 数据验证规则
- 索引和约束

#### 2.3 API设计
**目的**: 定义前后端接口契约

**关键要点**:
- API端点列表
- 请求/响应格式 (JSON Schema)
- 错误码定义
- 认证和权限

#### 4. 验收标准
**目的**: 提供可测试的功能清单

**关键要点**:
- 功能完整性 (Checkbox清单)
- 性能指标 (具体数值)
- 安全要求 (检查项)
- 测试覆盖 (覆盖率目标)

## 最佳实践

### 1. 保持简洁但完整
```
❌ 冗长: 写10页的详细文档
✅ 恰当: 核心信息完整，无冗余
```

### 2. 图文并茂
```
❌ 纯文字: 复杂的架构用文字描述
✅ 使用图表: ASCII图、流程图、ERD
```

### 3. 示例驱动
```
❌ 抽象描述: "API返回用户信息"
✅ 具体示例: 提供实际的JSON格式
```

### 4. 版本控制
```
✅ 规范文档纳入Git管理
✅ 记录变更历史
✅ 代码和规范同步更新
```

### 5. 迭代更新
```
开发中发现问题 → 更新规范 → 同步团队
```

## 常见问题

### Q: 规范需要多详细？
A: 详细程度取决于功能复杂度:
- 简单CRUD: 可以简化
- 核心业务逻辑: 必须详细
- 涉及外部集成: 必须详细

原则: **足够让其他人能够独立实现**

### Q: 规范写完要多久？
A: 时间估算:
- 简单功能: 1-2小时
- 中等复杂度: 3-4小时
- 复杂功能: 1天

**投入产出比**: 1小时规范可节省3-5小时返工时间

### Q: 规范可以边写边改吗？
A: 可以，但遵循以下原则:
- ✅ 开发前修改: 随时可以
- ⚠️ 开发中修改: 需要评估影响
- ❌ 测试后修改: 需要重新测试

### Q: 规范由谁编写？
A: 推荐方式:
1. **Spec Writer Agent** 生成初稿
2. **技术负责人** 审查和完善
3. **团队** 讨论和批准

### Q: 规范和代码注释有什么区别？
A:
- **规范**: 设计文档，开发前编写
- **注释**: 实现说明，代码中编写

规范关注"为什么"和"怎么做"，注释关注"做了什么"

## 工具和命令

### 创建新规范
```bash
# 复制模板
cp specs/SPEC-TEMPLATE.md specs/SPEC-your-feature.md

# 或使用spec-writer agent
/agent spec-writer "用户认证功能"
```

### 验证规范完整性
```bash
# 检查必填章节
grep -E "^## [0-9]" specs/SPEC-xxx.md

# 检查验收标准
grep -E "- \[ \]" specs/SPEC-xxx.md
```

### 统计规范数量
```bash
# 所有规范
ls specs/SPEC-*.md | wc -l

# 进行中的规范
ls specs/active/*.md | wc -l

# 已归档的规范
ls specs/archived/*.md | wc -l
```

## 一、A.C.E. 自主开发循环

```
┌─────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────┤

│  2. 构建依赖图谱                                         │

│  5. 发布方案摘要                                         │

                           ↓
┌─────────────────────────────────────────────────────────┐
│  C - Code (编码实施)                                     │
├─────────────────────────────────────────────────────────┤
│  1. 智能分批 (基础设施→核心逻辑→UI→测试)                │

│  3. 自动同步(接口/类型/依赖)                             │
│  4. 主动报告进度                                         │
```

## 二、测试驱动开发 (TDD)

### 2.1 Red-Green-Refactor循环

```
┌─────────────────────────────────────────────────────────┐
│  RED - 编写失败的测试                                    │
├─────────────────────────────────────────────────────────┤
│  1. 明确功能需求                                         │
│  2. 编写测试用例                                         │
│  3. 运行测试 → 失败 (红色)                               │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  GREEN - 实现最小代码使测试通过                          │
├─────────────────────────────────────────────────────────┤
│  1. 编写最简单的实现                                     │
│  2. 运行测试 → 通过 (绿色)                               │
│  3. 不关注代码质量,只关注功能                            │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  REFACTOR - 重构代码                                     │
├─────────────────────────────────────────────────────────┤
│  1. 优化代码结构                                         │
│  2. 消除重复                                             │
│  3. 改善命名                                             │
│  4. 运行测试 → 仍然通过                                  │
└─────────────────────────────────────────────────────────┘
                           ↓
                      重复循环
```

### 2.2 实际应用案例

**场景: 实现用户邮箱验证功能**

```typescript
// ========== RED阶段 ==========
// 先写测试

describe('EmailValidator', () => {
  it('should return true for valid email', () => {
    const validator = new EmailValidator();
    expect(validator.isValid('user@example.com')).toBe(true);
  });

  it('should return false for invalid email', () => {
    const validator = new EmailValidator();
    expect(validator.isValid('invalid')).toBe(false);
  });

  it('should return false for empty string', () => {
    const validator = new EmailValidator();
    expect(validator.isValid('')).toBe(false);
  });
});

// 运行测试 → ❌ 失败 (EmailValidator类不存在)

// ========== GREEN阶段 ==========
// 实现最小代码

class EmailValidator {
  isValid(email: string): boolean {
    // 最简单的实现
    return email.includes('@') && email.includes('.');
  }
}

// 运行测试 → ✅ 通过

// ========== REFACTOR阶段 ==========
// 优化实现

class EmailValidator {
  private readonly emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;

  isValid(email: string): boolean {
    if (!email) return false;
    return this.emailRegex.test(email);
  }
}

// 运行测试 → ✅ 仍然通过
// 代码质量提升,功能不变
```

## 三、多Agent协作工作流

### 3.1 瀑布式协作 (SEQUENTIAL)

```
需求分析 (architect)
    ↓
  输出: 设计文档
    ↓
代码实现 (code-implementer)
    ↓
  输出: 代码文件
    ↓
代码审查 (code-reviewer)
    ↓
  输出: 审查报告
    ↓
Bug修复 (debugger)
    ↓
  输出: 修复后代码
    ↓
安全审计 (security-analyst)
    ↓
  输出: 安全报告
    ↓
交付完成
```

### 3.2 并行协作 (PARALLEL)

```
任务分发 (orchestrator)
    ↓
    ├─→ Agent1: 审查模块A
    ├─→ Agent2: 审查模块B
    ├─→ Agent3: 审查模块C
    └─→ Agent4: 审查模块D
         ↓
    所有Agent完成
         ↓
    结果整合 (orchestrator)
         ↓
    生成综合报告
```

### 3.3 实际应用: 全栈功能开发

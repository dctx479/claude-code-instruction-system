# E-Commerce System Template

## 系统概览

**典型场景**: 电商平台（淘宝、京东、拼多多）

**核心特征**:
- 高并发读（商品浏览）
- 瞬时高并发写（秒杀、抢购）
- 强一致性要求（库存、订单、支付）
- 复杂业务流程（下单 → 支付 → 发货 → 售后）

---

## 六灵魂问题典型值

| 维度 | 典型值 | 说明 |
|------|--------|------|
| **规模** | DAU 100 万 - 1000 万 | 中大型电商 |
| **读写比** | 100:1 - 1000:1 | 浏览远多于购买 |
| **一致性** | 混合 | 库存/支付强一致，推荐/评论最终一致 |
| **增长** | 爆发式 | 双 11、618 流量峰值 10-50 倍 |
| **故障代价** | 极高 | 1 小时宕机 = 数十万到数百万损失 |
| **约束** | 高可用 > 性能 > 成本 | 不能让用户付不了钱 |

## 粗算示例

**假设场景**: DAU 500 万，每用户日均浏览 20 个商品，下单 0.2 次

### 存储估算
```
商品数据: 1000 万商品 × 10KB/商品 = 100GB
订单数据: 100 万订单/天 × 365 天 × 2 年 × 5KB/订单 = 365GB
用户数据: 5000 万用户 × 2KB/用户 = 100GB
总计: ~600GB（不含图片、日志）
```

### QPS 估算
```
浏览 QPS: (500 万 × 20) / 86400 = ~1200 QPS
下单 QPS: (500 万 × 0.2) / 86400 = ~12 QPS
峰值（双 11）: 浏览 ~60,000 QPS, 下单 ~600 QPS
```

### 瓶颈识别
- **存储**: 需要分库分表（订单表按用户 ID 分片）
- **计算**: 峰值需要 50+ 台应用服务器
- **网络**: 商品图片 CDN 分发

## 关键决策分叉

### 决策 1: 库存扣减策略

**选项 A: 数据库行锁（悲观锁）**
```sql
BEGIN TRANSACTION;
SELECT stock FROM inventory WHERE product_id = 123 FOR UPDATE;
UPDATE inventory SET stock = stock - 1 WHERE product_id = 123;
COMMIT;
```
- ✅ 优点: 强一致性，逻辑简单
- ❌ 缺点: 高并发下锁竞争严重，性能差
- 📌 适用: 低并发场景（QPS < 100）

**选项 B: Redis + Lua 脚本（乐观锁）**
```lua
local stock = redis.call('GET', 'stock:' .. product_id)
if tonumber(stock) > 0 then
    redis.call('DECR', 'stock:' .. product_id)
    return 1
else
    return 0
end
```
- ✅ 优点: 高性能，支持万级 QPS
- ❌ 缺点: 需要异步同步到 MySQL，有丢失风险
- 📌 适用: 秒杀场景

**选项 C: 预扣库存 + 异步补偿**
```
1. 用户下单 → Redis 预扣库存
2. 订单进入消息队列
3. 后台消费队列 → 扣减 MySQL 库存
4. 如果 MySQL 扣减失败 → 回滚 Redis 库存
```
- ✅ 优点: 平衡性能和一致性
- ❌ 缺点: 复杂度高，需要处理补偿逻辑
- 📌 适用: **推荐方案**（中大型电商）

---

### 决策 2: 订单表设计

**选项 A: 单表**
- ✅ 优点: 简单，查询方便
- ❌ 缺点: 订单量超过 1000 万后性能急剧下降
- 📌 适用: 小型电商（日订单 < 1 万）

**选项 B: 分库分表（按用户 ID）**
```
数据库: db_order_0, db_order_1, ..., db_order_15 (16 个库)
每库表: order_0, order_1, ..., order_15 (16 张表)
分片规则: user_id % 256 → 确定库表
```
- ✅ 优点: 水平扩展，单表数据量可控
- ❌ 缺点: 跨分片查询复杂（如商家查看所有订单）
- 📌 适用: **推荐方案**（中大型电商）

**选项 C: 分库分表（按时间）**
- ✅ 优点: 归档方便，冷热数据分离
- ❌ 缺点: 热点库压力大（最新订单集中）
- 📌 适用: 数据仓库、报表系统

---

### 决策 3: 支付流程

**选项 A: 同步支付**
```
用户点击支付 → 调用支付网关 → 等待结果 → 返回用户
```
- ✅ 优点: 逻辑简单，用户体验直接
- ❌ 缺点: 支付网关超时会阻塞用户
- 📌 适用: 小型电商

**选项 B: 异步支付 + 回调**
```
1. 用户点击支付 → 创建支付订单 → 立即返回"支付中"
2. 调用支付网关（异步）
3. 支付网关回调 → 更新订单状态
4. 用户轮询或 WebSocket 推送结果
```
- ✅ 优点: 不阻塞用户，支持高并发
- ❌ 缺点: 需要处理回调幂等性、重试
- 📌 适用: **推荐方案**（中大型电商）

## 数据模型

### 核心表设计

**商品表（product）**
```sql
CREATE TABLE product (
    id BIGINT PRIMARY KEY,
    name VARCHAR(200),
    price DECIMAL(10,2),
    stock INT,  -- 仅用于展示，真实库存在 Redis
    category_id INT,
    created_at TIMESTAMP,
    INDEX idx_category (category_id)
);
```

**订单表（order - 分库分表）**
```sql
CREATE TABLE order_0 (
    id BIGINT PRIMARY KEY,
    user_id BIGINT,  -- 分片键
    product_id BIGINT,
    quantity INT,
    total_price DECIMAL(10,2),
    status ENUM('pending', 'paid', 'shipped', 'completed', 'cancelled'),
    created_at TIMESTAMP,
    INDEX idx_user_created (user_id, created_at)
);
```

**库存表（inventory - 仅 MySQL 备份）**
```sql
CREATE TABLE inventory (
    product_id BIGINT PRIMARY KEY,
    stock INT,
    version INT,  -- 乐观锁版本号
    updated_at TIMESTAMP
);
```

## ADR 示例

### ADR-001: 库存扣减使用 Redis + 异步补偿

**决策**: 秒杀场景使用 Redis Lua 脚本预扣库存，异步同步到 MySQL

**理由**:
- 峰值 QPS 预计 10,000+，MySQL 行锁无法支撑
- 库存是强一致性要求，不能超卖
- Redis 单线程模型天然保证原子性

**后果**:
- ✅ 优点: 支持万级 QPS，响应时间 < 10ms
- ❌ 缺点: 需要处理 Redis-MySQL 数据一致性，增加系统复杂度

**替代方案**: 
- 数据库行锁（性能不足）
- 分布式锁（如 Redisson）（性能略差，但一致性更强）

---

### ADR-002: 订单表按用户 ID 分 256 片

**决策**: 16 个数据库，每库 16 张表，共 256 个分片

**理由**:
- 日订单量 100 万，3 年累计 10 亿+，单表无法承载
- 用户查询自己订单的频率远高于商家查询所有订单
- 256 片可支撑到日订单 1000 万

**后果**:
- ✅ 优点: 单表数据量控制在 400 万以内，查询性能稳定
- ❌ 缺点: 商家后台查询需要聚合 256 个分片（使用 Elasticsearch 解决）

**替代方案**:
- 按时间分片（热点问题严重）
- 不分片（性能崩溃）

## 风险列表

### 技术风险

1. **Redis 单点故障**
   - 影响: 库存服务不可用
   - 缓解: Redis Sentinel 主从切换 + 持久化

2. **消息队列积压**
   - 影响: 订单处理延迟
   - 缓解: 监控队列长度 + 动态扩容消费者

3. **数据库主库宕机**
   - 影响: 写入服务不可用
   - 缓解: MySQL 主从自动切换（MHA/Orchestrator）

### 业务风险

1. **超卖问题**
   - 影响: 订单无法履约，用户投诉
   - 缓解: Redis 库存 + MySQL 库存双重校验

2. **支付回调丢失**
   - 影响: 用户已付款但订单未更新
   - 缓解: 主动查询 + 补偿任务定时扫描

3. **秒杀黄牛**
   - 影响: 真实用户抢不到
   - 缓解: 验证码 + IP 限流 + 行为分析

## 1. 架构思维基础框架

### 1.1 核心原则

**架构不是"画出来的"，而是"逼出来的"**

架构设计的本质是在约束条件下做权衡（Trade-off）。没有完美的架构，只有最适合当前场景的架构。

### 1.2 三个核心问题

在开始设计之前，必须回答：

1. **为什么要做？**（业务价值）
   - 解决什么问题？
   - 不做的代价是什么？

2. **能不能做？**（技术可行性）
   - 团队是否具备能力？
   - 是否有成熟方案参考？

3. **值不值得做？**（成本收益分析）
   - 投入多少资源？
   - 预期收益是什么？

### 1.3 架构设计四步法

#### Step 1: 明确约束

| 约束类型 | 示例 |
|----------|------|
| **业务约束** | 必须在 3 个月内上线 |
| **技术约束** | 只能用 Java + MySQL |
| **资源约束** | 团队只有 3 个人 |
| **合规约束** | 必须符合 GDPR |

#### Step 2: 粗算瓶颈

用简单数学找到系统瓶颈：
- 存储：1000 万用户 × 10KB/用户 = 100GB
- QPS：100 万 DAU × 10 操作/天 / 86400 秒 = 116 QPS
- 带宽：116 QPS × 10KB = 1.16 MB/s

#### Step 3: 列出方案

列出 2-3 个可行方案，每个方案包括：
- 技术选型
- 优点 / 缺点
- 成本（开发 + 运维）
- 风险

#### Step 4: 做决策

基于约束和粗算结果，选择最合适的方案，并记录 ADR（Architecture Decision Record）。

### 1.4 ADR 模板

```markdown
## ADR-001: [决策标题]

**背景**: 当前面临什么问题？

**决策**: 选择方案 A

**理由**:
- 满足约束条件 X
- 性能满足需求（QPS < 1000）
- 团队熟悉该技术

**后果**:
- 优点: ...
- 缺点: ...
- 技术债务: ...

**替代方案**:
- 方案 B（为什么不选）
- 方案 C（为什么不选）
```

## 2. 分布式系统基础

### 2.1 CAP 定理

在分布式系统中，**不可能同时满足**：
- **C (Consistency)**: 一致性 — 所有节点看到相同数据
- **A (Availability)**: 可用性 — 系统始终能响应
- **P (Partition Tolerance)**: 分区容忍性 — 网络分区时仍可工作

**实际选择**:
- CA（放弃 P）: 单机系统（如传统 MySQL）
- CP（放弃 A）: 分布式数据库（如 HBase、MongoDB 强一致模式）
- AP（放弃 C）: 最终一致系统（如 Cassandra、DynamoDB）

### 2.2 BASE 理论

放松 ACID 的强一致性要求：
- **BA (Basically Available)**: 基本可用
- **S (Soft state)**: 软状态（允许中间状态）
- **E (Eventually consistent)**: 最终一致性

### 2.3 常见分布式模式

#### 模式 1: 主从复制（Master-Slave）

```
写: 主库
读: 从库（1-N 个）
```

- ✅ 读扩展性好
- ❌ 主库单点故障
- ❌ 主从延迟（秒级）

#### 模式 2: 多主复制（Multi-Master）

```
写: 任意主库
读: 任意主库
冲突: 最后写入胜出（Last Write Wins）
```

- ✅ 写扩展性好
- ❌ 冲突解决复杂

#### 模式 3: 分片（Sharding）

```
用户 1-1000: 分片 1
用户 1001-2000: 分片 2
...
```

- ✅ 水平扩展
- ❌ 跨分片查询复杂

## 3. 数据一致性模式

### 3.1 一致性级别

| 级别 | 描述 | 延迟 | 适用场景 |
|------|------|------|----------|
| **强一致** | 读取最新写入 | 高 | 金融交易、库存扣减 |
| **最终一致** | 延迟后一致 | 低 | 社交动态、评论 |
| **因果一致** | 保证因果顺序 | 中 | 聊天消息 |
| **会话一致** | 同一会话内一致 | 低 | 购物车 |

### 3.2 分布式事务方案

#### 方案 1: 2PC（两阶段提交）

```
阶段 1: Prepare（所有参与者准备）
阶段 2: Commit（所有返回 OK 才提交）
```

- ✅ 强一致性
- ❌ 阻塞，性能差

#### 方案 2: TCC（Try-Confirm-Cancel）

```
Try: 预留资源
Confirm: 确认提交
Cancel: 回滚
```

- ✅ 无阻塞，高性能
- ❌ 业务侵入性强

#### 方案 3: Saga（长事务）

```
事务 1 → 事务 2 → 事务 3
失败 → 补偿 3 → 补偿 2 → 补偿 1
```

- ✅ 适合长事务
- ❌ 补偿逻辑复杂

#### 方案 4: 本地消息表（最终一致）

```
1. 本地事务：写业务 + 写消息表
2. 定时任务：扫描消息表 → 调用下游
3. 下游成功 → 删除消息
```

- ✅ 解耦，高可用
- ❌ 延迟（秒到分钟级）

### 3.3 选择指南

| 场景 | 推荐方案 |
|------|----------|
| 金融转账（强一致） | TCC |
| 订单创建（最终一致） | 本地消息表 |
| 长流程（如出差审批） | Saga |
| 低并发（< 10 TPS） | 2PC |

## 4. 故障弹性工程

### 4.1 故障处理金字塔

```
Level 4: 自动恢复（Chaos Engineering）
Level 3: 降级（Graceful Degradation）
Level 2: 熔断（Circuit Breaker）
Level 1: 重试（Retry with Backoff）
Level 0: 监控告警（Monitoring & Alerting）
```

### 4.2 核心模式

#### 模式 1: 重试（Retry）

```python
import time

def retry(func, max_attempts=3, backoff=2):
    for i in range(max_attempts):
        try:
            return func()
        except Exception as e:
            if i == max_attempts - 1:
                raise
            time.sleep(backoff ** i)
```

**适用场景**: 网络抖动、临时故障

#### 模式 2: 熔断（Circuit Breaker）

```
状态机:
Closed（正常）→ 失败率超过阈值 → Open（熔断）
Open → 等待一段时间 → Half-Open（试探）
Half-Open → 成功 → Closed
Half-Open → 失败 → Open
```

**适用场景**: 防止雪崩，快速失败

#### 模式 3: 降级（Degradation）

```python
def get_recommendations(user_id):
    try:
        return ml_service.recommend(user_id)  # 复杂算法
    except:
        return default_recommendations()  # 降级：返回热门商品
```

**适用场景**: 非核心功能，保障主流程

#### 模式 4: 限流（Rate Limiting）

**算法选择**:
- **固定窗口**: 简单，但有突刺问题
- **滑动窗口**: 平滑，但内存占用高
- **令牌桶**: 允许突发流量
- **漏桶**: 平滑流量

**适用场景**: API 限流、防刷

#### 模式 5: 超时控制（Timeout）

```python
import requests

response = requests.get(url, timeout=5)  # 5 秒超时
```

**经验值**:
- 数据库查询: 1-3 秒
- 微服务调用: 3-5 秒
- 第三方 API: 10 秒

### 4.3 故障演练（Chaos Engineering）

定期注入故障，验证系统弹性：
- 随机杀死容器
- 模拟网络延迟
- 模拟磁盘满
- 模拟 CPU 飙高

**工具**: Chaos Monkey (Netflix), Chaos Mesh

## 5. AI 原生架构设计

### 5.1 与传统架构的区别

| 维度 | 传统架构 | AI 原生架构 |
|------|----------|-------------|
| **确定性** | 规则明确 | 概率输出 |
| **延迟** | 毫秒级 | 秒级（LLM 调用） |
| **成本** | 固定 | 按 Token 计费 |
| **可解释性** | 高 | 低（黑盒） |
| **数据依赖** | 低 | 高（需要训练数据） |

### 5.2 核心设计模式

#### 模式 1: Prompt Engineering

```python
def generate_response(user_input):
    prompt = f"""
    你是一个客服机器人。
    
    用户问题: {user_input}
    
    回答要求:
    - 简洁友好
    - 不超过 100 字
    - 如果不确定，说"我不知道"
    """
    return llm.complete(prompt)
```

**关键点**:
- 明确角色和任务
- 提供示例（Few-shot）
- 设置约束和边界

#### 模式 2: RAG（检索增强生成）

```
用户问题 → 向量检索 → Top-K 文档 → 拼接 Prompt → LLM 生成答案
```

**优点**:
- 减少幻觉
- 支持实时知识更新
- 降低成本（不需要微调）

#### 模式 3: Agent 编排

```
用户任务 → Planner（规划）→ 子任务 1, 2, 3
                           ↓
           Worker 1 (Tool Call) → 结果 1
           Worker 2 (Tool Call) → 结果 2
                           ↓
                    汇总 → 最终答案
```

**适用场景**: 复杂任务分解（如旅行规划、代码生成）

#### 模式 4: 混合推理（Hybrid）

```
简单查询（如"今天天气"）→ 规则引擎（快速响应）
复杂查询（如"推荐旅游路线"）→ LLM（深度推理）
```

**优点**: 平衡成本和效果

### 5.3 AI 系统特有风险

| 风险 | 缓解措施 |
|------|----------|
| **幻觉（Hallucination）** | RAG + 事实验证 |
| **提示注入（Prompt Injection）** | 输入过滤 + 权限控制 |
| **成本失控** | Token 限额 + 缓存 |
| **延迟高** | 异步处理 + 流式输出 |
| **偏见（Bias）** | 数据审计 + 人工审核 |

### 5.4 AI 原生架构检查清单

- [ ] LLM 调用是否有超时？
- [ ] 是否有成本监控和限额？
- [ ] 是否有缓存机制（相同问题不重复调用）？
- [ ] 是否有降级方案（LLM 不可用时）？
- [ ] 用户输入是否过滤（防注入）？
- [ ] 输出是否验证（防幻觉）？
- [ ] 是否有人工审核机制（敏感场景）？

## 1. 抢票系统：从崩溃到稳定

### 背景

某演唱会抢票系统，10 万张票，预计 100 万人同时抢购。

### V0.1 - MVP（崩溃版本）

**架构**:
```
用户 → 单台服务器 → MySQL
```

**问题**:
- 开票 10 秒后服务器 CPU 100%，宕机
- 数据库连接数耗尽（max_connections=100）
- 超卖：数据库行锁竞争导致库存扣减失败

**根因**:
- 同步扣库存（单机 TPS 只有 ~50）
- 无限流保护
- 数据库单点

---

### V1.0 - 应急修复

**改进**:
1. **限流**: Nginx 限流 1000 QPS
2. **排队**: 引入 Redis 队列
3. **预热**: 提前加载库存到 Redis

**架构**:
```
用户 → Nginx（限流）→ 应用服务器（3 台）→ Redis 队列 → Worker 扣库存
```

**效果**:
- ✅ 系统不再崩溃
- ✅ QPS 稳定在 1000
- ❌ 用户体验差（大量"系统繁忙，请稍后再试"）

---

### V2.0 - 优化排队体验

**改进**:
1. **虚拟排队**: 用户进入队列后显示"前面还有 X 人"
2. **WebSocket**: 实时推送排队进度
3. **库存预扣**: Redis Lua 脚本原子扣减

**核心代码**:
```lua
-- Redis Lua 脚本（原子扣库存）
local stock = redis.call('GET', 'stock:ticket_123')
if tonumber(stock) > 0 then
    redis.call('DECR', 'stock:ticket_123')
    return 1  -- 成功
else
    return 0  -- 售罄
end
```

**架构**:
```
用户 → Nginx → WebSocket Gateway
              ↓
          Redis 队列（排队）
              ↓
          Worker Pool（扣库存 + 创建订单）
              ↓
          MySQL（订单持久化）
```

**效果**:
- ✅ 用户等待有预期（能看到排队位置）
- ✅ 库存扣减 QPS 达到 5000
- ❌ Redis-MySQL 数据不一致（Redis 扣了但 MySQL 失败）

---

### V3.0 - 数据一致性保障

**改进**:
1. **TCC 分布式事务**:
   - Try: Redis 预扣库存 + MySQL 创建待支付订单
   - Confirm: 用户支付 → 更新订单状态
   - Cancel: 超时未支付 → Redis 回滚库存

2. **对账机制**:
   - 定时任务：对比 Redis 和 MySQL 库存
   - 差异告警 → 人工介入

**架构**:
```
用户 → Nginx → WebSocket Gateway
              ↓
          Redis 队列
              ↓
          TCC 协调器
          ├─ Redis（Try 预扣）
          └─ MySQL（Try 创建订单）
              ↓
          对账服务（定时）
```

**效果**:
- ✅ 数据一致性保障
- ✅ 零超卖
- ✅ 10 万张票 30 分钟售罄

---

### V4.0 - 防刷与公平性

**改进**:
1. **实名制**: 一个身份证只能买一张
2. **验证码**: 进入排队前必须通过
3. **行为分析**: 检测脚本刷票（如点击速度过快）
4. **IP 限流**: 单 IP 每秒最多 5 次请求

**效果**:
- ✅ 黄牛减少 80%
- ✅ 真实用户中签率提升

---

### 关键教训

| 问题 | 解决方案 | 经验教训 |
|------|----------|----------|
| 超卖 | Redis Lua 原子扣减 | 库存扣减必须原子操作 |
| 崩溃 | 限流 + 排队 | 永远不要让流量直接打到数据库 |
| 体验差 | 虚拟排队 + 进度展示 | 等待可以接受，但要有预期 |
| 数据不一致 | TCC + 对账 | 分布式系统必须有对账机制 |
| 黄牛 | 实名 + 验证码 + 行为分析 | 防刷是持续对抗，没有一劳永逸 |

## 2. SaaS 平台：从单租户到多租户

### 背景

某项目管理 SaaS，从服务单一客户演进到服务 5000+ 租户。

### V0.1 - 单租户版本

**架构**:
```
客户 A → 独立部署（服务器 + 数据库）
客户 B → 独立部署（服务器 + 数据库）
```

**问题**:
- 每增加一个客户 = 重新部署一套环境（成本高）
- 运维复杂（5 个客户 = 5 套环境）
- 无法快速迭代（需要逐个升级）

---

### V1.0 - 共享数据库（Shared Database）

**改进**:
1. **所有租户共享一个数据库**
2. **每张表增加 `tenant_id` 字段**

**架构**:
```
所有客户 → 单一应用 → 单一 MySQL
```

**数据模型**:
```sql
CREATE TABLE projects (
    id BIGINT PRIMARY KEY,
    tenant_id BIGINT,  -- 租户 ID
    name VARCHAR(200),
    created_at TIMESTAMP,
    INDEX idx_tenant (tenant_id)
);

-- 查询时必须带 tenant_id
SELECT * FROM projects WHERE tenant_id = 123;
```

**效果**:
- ✅ 成本降低 90%（1 台服务器服务所有客户）
- ✅ 运维简化（统一升级）
- ❌ 慢查询拖垮所有租户
- ❌ 数据泄露风险（查询忘记加 tenant_id）

---

### V2.0 - 混合模式（Hybrid）

**改进**:
1. **小租户（< 100 用户）**: 共享数据库
2. **大租户（> 100 用户）**: 独立数据库

**架构**:
```
小租户 A, B, C → 共享 MySQL 1
小租户 D, E, F → 共享 MySQL 2
大租户 G → 独立 MySQL 3
大租户 H → 独立 MySQL 4
```

**租户路由表**:
```sql
CREATE TABLE tenant_routing (
    tenant_id BIGINT PRIMARY KEY,
    db_type ENUM('shared', 'isolated'),
    db_shard_id INT,  -- 共享库编号或独立库编号
    created_at TIMESTAMP
);
```

**路由逻辑**:
```python
def get_db_connection(tenant_id):
    routing = db.query("SELECT * FROM tenant_routing WHERE tenant_id = ?", tenant_id)
    if routing.db_type == 'shared':
        return shared_pool[routing.db_shard_id]
    else:
        return isolated_pool[routing.db_shard_id]
```

**效果**:
- ✅ 平衡成本和隔离性
- ✅ 大租户不影响小租户
- ❌ 架构复杂，路由层成为单点

---

### V3.0 - 弹性伸缩

**改进**:
1. **自动升级**: 小租户增长到 100 用户 → 自动迁移到独立库
2. **读写分离**: 大租户独立库增加从库
3. **数据归档**: 超过 1 年的数据归档到 S3

**迁移流程**:
```
1. 创建独立数据库
2. 双写（写共享库 + 写独立库）
3. 数据全量同步
4. 验证数据一致性
5. 切换读流量到独立库
6. 停止写共享库
7. 删除共享库数据
```

**架构**:
```
                租户路由层
                    ↓
    ┌───────────────┴───────────────┐
    │                               │
共享 MySQL 集群            独立 MySQL 集群
    │                               │
（主 + 从）                 （主 + 从，按租户）
```

**效果**:
- ✅ 支持 5000+ 租户
- ✅ 大租户性能隔离
- ✅ 成本可控（小租户共享，大租户独立）

---

### V4.0 - 计费与自助管理

**改进**:
1. **按用户数计费**: $10/用户/月
2. **使用量监控**: 实时统计每租户的存储、API 调用
3. **自助管理**: 租户管理员可自助升级套餐、添加用户

**计费架构**:
```
应用层 → Kafka（事件流）
            ↓
        计费服务（消费事件）
            ↓
        计费数据库（usage_logs）
            ↓
        账单生成（每月 1 号）
```

**效果**:
- ✅ 自助注册转化率提升 200%
- ✅ 人工客服成本降低 50%

---

### 关键教训

| 阶段 | 核心挑战 | 解决方案 | 经验教训 |
|------|----------|----------|----------|
| V0.1 | 成本高 | 共享数据库 | 不要过早优化，先验证商业模式 |
| V1.0 | 大租户拖垮小租户 | 混合模式 | 80/20 法则：大客户需要特殊对待 |
| V2.0 | 架构复杂 | 租户路由层 | 复杂度是必要代价，但要做好抽象 |
| V3.0 | 迁移风险 | 双写 + 灰度 | 数据迁移永远不要一步到位 |
| V4.0 | 运营效率 | 自助管理 | SaaS 的核心是自助服务 |

## 通用演进模式

### 模式 1: 垂直先行（Vertical First）

```
MVP（单机） → 垂直扩展（加配置）→ 水平扩展（分布式）
```

**适用**: 不确定业务是否成功时

### 模式 2: 读写分离

```
单库 → 主从（读写分离）→ 分库分表
```

**适用**: 读多写少场景

### 模式 3: 冷热分离

```
全部热数据 → 热数据（MySQL）+ 冷数据（S3）
```

**适用**: 历史数据访问少

### 模式 4: 同步转异步

```
同步调用 → 消息队列 → 事件驱动
```

**适用**: 长流程、高并发

---

## 架构演进原则

1. **渐进式演进**: 不要重写，小步快跑
2. **数据优先**: 数据迁移比代码重构更难
3. **可回滚**: 每次变更都要有回滚方案
4. **灰度发布**: 先小流量验证，再全量
5. **监控先行**: 没有监控就没有演进

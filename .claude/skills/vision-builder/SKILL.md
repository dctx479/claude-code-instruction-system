# Vision Builder Skill

## 契约定义

### What（输入/输出）

**输入**：
- 用户的模糊需求描述（必需）
- 项目背景信息（可选）
- 约束条件：时间、预算、技术栈（可选）
- 利益相关者信息（可选）

**输出**：
- 结构化愿景文档 `VISION.md`
- 包含 5W1H 分析、SMART 目标、风险评估
- 明确的范围定义（In/Out Scope）

### When Done（验收标准）

- 生成完整的 `VISION.md` 文档
- 5W1H 每个维度都有明确内容
- 至少 3 个 SMART 成功标准
- 识别至少 3 个关键风险
- 范围边界清晰（包含/排除明确）
- 用户确认愿景符合预期

### What NOT（边界约束）

- **禁止技术实现设计**：不设计架构、不选型技术（那是 architect 的职责）
- **禁止编写代码**：只产出文档
- **禁止承诺时间线**：不做具体时间估算
- **禁止资源分配**：不分配人员或预算
- **禁止跳过确认**：必须获得用户对愿景的确认

---

## 何时使用此 Skill

当以下情况时自动激活：
- 用户描述一个新项目或新功能，但需求模糊
- Autopilot Planning 阶段开始
- 用户明确请求"帮我理清需求"或"帮我规划项目"
- 需要在开发前对齐利益相关者期望

## 核心能力

### 1. 需求澄清（引导式提问）

通过结构化问题帮助用户明确需求：

| 问题类型 | 示例问题 |
|----------|----------|
| 目标导向 | "这个项目成功后，最重要的变化是什么？" |
| 用户导向 | "主要用户是谁？他们目前如何解决这个问题？" |
| 约束导向 | "有哪些技术、时间或预算上的限制？" |
| 风险导向 | "如果项目失败，最可能的原因是什么？" |
| 范围导向 | "哪些功能是必须有的？哪些是锦上添花的？" |

### 2. 5W1H 分析框架

| 维度 | 分析要点 | 输出 |
|------|----------|------|
| **Who** | 目标用户、利益相关者、决策者 | 用户画像、角色列表 |
| **What** | 核心功能、交付物、成功定义 | 功能列表、验收标准 |
| **When** | 时间框架、里程碑、截止日期 | 时间线（不做估算） |
| **Where** | 部署环境、使用场景、地理范围 | 环境描述 |
| **Why** | 业务价值、用户痛点、战略目标 | 价值主张 |
| **How** | 高层次方法、关键技术方向 | 方法论概述 |

### 3. SMART 目标设定

确保每个成功标准符合 SMART 原则：

- **S**pecific（具体）：目标清晰明确
- **M**easurable（可衡量）：有量化指标
- **A**chievable（可实现）：在约束内可达成
- **R**elevant（相关）：与业务目标一致
- **T**ime-bound（有时限）：有明确时间框架

### 4. 风险识别矩阵

| 风险类型 | 识别方法 | 输出 |
|----------|----------|------|
| 技术风险 | 技术可行性评估 | 风险描述 + 缓解策略 |
| 业务风险 | 市场/用户分析 | 风险描述 + 缓解策略 |
| 资源风险 | 约束条件分析 | 风险描述 + 缓解策略 |
| 依赖风险 | 外部依赖识别 | 风险描述 + 缓解策略 |

## 工作流程

### 阶段 1：需求收集（Gather）

```
用户输入模糊需求
    ↓
分析需求完整度
    ↓
[完整度 < 60%?]
    ↓
是 → 提出引导性问题（最多 5 轮）
    ↓
收集足够信息后进入下一阶段
```

**引导问题模板**：
```markdown
感谢您的描述！为了帮您构建清晰的项目愿景，我还需要了解：

1. **目标用户**：这个项目主要服务谁？
2. **核心价值**：用户使用后最大的收益是什么？
3. **成功标准**：如何判断项目成功？
4. **已知约束**：有哪些技术、时间、预算限制？
5. **排除范围**：哪些功能明确不在本次范围内？
```

### 阶段 2：需求分析（Analyze）

```
收集到的信息
    ↓
5W1H 结构化分析
    ↓
SMART 目标提炼
    ↓
风险识别与评估
    ↓
范围边界定义
```

### 阶段 3：愿景构建（Build）

```
分析结果
    ↓
生成 VISION.md 文档
    ↓
按模板填充各部分
    ↓
确保完整性检查
```

### 阶段 4：验证确认（Validate）

```
VISION.md 文档
    ↓
展示给用户审阅
    ↓
[用户确认?]
    ↓
是 → 完成，输出最终版本
否 → 根据反馈修改，重新验证
```

## 与系统集成

### Autopilot 集成

```
Autopilot Planning 阶段
    ↓
自动调用 vision-builder
    ↓
生成 VISION.md
    ↓
用户确认后进入 Specification 阶段
    ↓
architect Agent 基于 VISION.md 设计架构
```

### 与其他组件协作

| 组件 | 协作方式 |
|------|----------|
| **architect** | vision-builder 输出 → architect 输入 |
| **spec-writer** | VISION.md 的成功标准 → 验收标准 |
| **plan-review** | 审查实现计划是否符合愿景 |
| **God Committee** | 重大项目愿景需委员会审批 |

## 使用示例

### 示例 1：从模糊需求到清晰愿景

**用户输入**：
> "我想做一个内部知识库系统"

**vision-builder 引导**：
```markdown
感谢您的描述！为了构建清晰愿景，我需要了解：

1. **目标用户**：知识库主要给谁用？（全公司/特定部门/外部客户）
2. **核心问题**：目前知识管理面临什么痛点？
3. **成功标准**：什么情况下算项目成功？
4. **技术约束**：有偏好的技术栈吗？需要与现有系统集成吗？
5. **排除范围**：哪些功能明确不做？
```

**用户补充后生成**：
```markdown
# 项目愿景：内部知识库系统

## 1. 执行摘要
构建一个面向技术团队的内部知识库，解决知识碎片化和搜索困难问题，
提升知识复用效率 50%。

## 3. 5W1H 分析
| 维度 | 内容 |
|------|------|
| Who | 技术团队 50 人 |
| What | 文档管理、全文搜索、知识图谱 |
| When | Q2 完成 MVP |
| Where | 内网部署，Docker 容器化 |
| Why | 减少重复造轮子，加速新人上手 |
| How | Markdown + 向量搜索 |

## 4. 成功标准
| ID | 标准描述 | 衡量指标 | 目标值 |
|----|----------|----------|--------|
| S1 | 知识搜索效率提升 | 平均搜索时间 | < 30 秒 |
| S2 | 知识贡献活跃度 | 月新增文档数 | > 20 篇 |
| S3 | 用户满意度 | NPS 评分 | > 70 |
```

### 示例 2：复杂项目愿景构建

**用户输入**：
> "重构公司的支付系统，支持多种支付方式"

**经过 3 轮引导后生成完整愿景**，包括：
- 详细的利益相关者分析（产品、技术、财务、合规）
- 明确的合规风险和缓解策略
- 分阶段的范围定义（Phase 1/2/3）
- 与现有系统的集成边界

## 质量检查清单

愿景文档生成后自动检查：

- [ ] 执行摘要是否一句话概括核心价值
- [ ] 5W1H 每个维度是否有实质内容
- [ ] 成功标准是否符合 SMART 原则
- [ ] 范围边界是否明确（In/Out/TBD）
- [ ] 风险是否识别至少 3 个
- [ ] 下一步行动是否明确

---

## 注意事项

1. **保持高层次**：愿景是方向性文档，不深入技术细节
2. **用户主导**：愿景必须获得用户确认才算完成
3. **迭代优化**：愿景可以随项目进展更新
4. **链接追踪**：VISION.md 应与后续 SPEC.md、PLAN.md 建立链接

---

**使用提示**：当用户需求模糊时，主动使用此 Skill 帮助澄清。不要假设需求，通过提问获取真实意图。

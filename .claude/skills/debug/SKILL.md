# 结构化调试 (Debug)

> 把"随机尝试"变成"系统排查"：五步工作流确保每次调试都有迹可循，不走回头路。

## 设计动机

调试失败的根因通常不是技术能力不足，而是**流程混乱**：

| 反模式 | 后果 |
|--------|------|
| 直接改代码，不先复现 | 改了不相关的地方，问题还在 |
| 同时改多处 | 不知道哪个改动起了作用 |
| 假设即结论 | 跳过验证，引入新 bug |
| 修完就走 | 同类问题下次再来 |

本 Skill 的核心原则：**一次只改一个变量**，每步都有可验证的输出。

---

## What（输入/输出）

### 输入

| 输入项 | 必需 | 描述 |
|--------|------|------|
| **错误描述/日志** | 是 | 错误信息、堆栈跟踪、异常行为描述 |
| **复现步骤** | 推荐 | 如何触发这个问题 |
| **相关代码/文件** | 推荐 | 问题涉及的代码范围 |
| **环境信息** | 否 | OS、版本、配置（如果可能相关） |

### 输出

| 输出项 | 格式 | 描述 |
|--------|------|------|
| **根因分析** | Markdown | 问题的真正原因（不是表象） |
| **修复方案** | 代码变更 | 最小化修复，不引入额外改动 |
| **验证结果** | 测试输出 | 证明修复有效的证据 |
| **防复发建议** | 列表 | 如何避免同类问题再次出现 |

## How（五步调试工作流）

### Step 1 — 复现（Reproduce）

**目标**：在受控环境中稳定触发问题。

```
检查清单：
□ 能否稳定复现？（偶发 vs 必现）
□ 复现步骤是否最小化？（去掉无关操作）
□ 是否记录了完整错误信息？（包括堆栈）
```

**如果无法复现**：
- 检查日志时间戳，确认问题确实发生过
- 询问用户：上次发生时做了什么不同的操作？
- 考虑环境差异（本地 vs CI vs 生产）

**禁止**：在无法复现的情况下开始修改代码。

### Step 2 — 隔离（Isolate）

**目标**：缩小问题范围到最小可疑单元。

**二分法定位**：
```
整个系统有问题
    → 哪个模块？
        → 哪个函数？
            → 哪一行？
```

**工具**：
- 添加临时日志（`console.log` / `print` / `logger.debug`）
- 注释掉代码块，确认问题消失/保留
- 最小化复现用例（能用 10 行代码复现就不用 100 行）

**输出**：一句话描述问题范围，例如：
> "问题出在 `auth/middleware.ts:47`，`validateToken()` 在 token 过期时没有正确处理 null 返回值"

### Step 3 — 假设（Hypothesize）

**目标**：提出 1-3 个可验证的根因假设，按可能性排序。

**假设格式**：
```
假设 #1（最可能）: [具体原因]
  验证方法: [如何证伪/证实]
  
假设 #2: [备选原因]
  验证方法: [如何证伪/证实]
```

**好假设的标准**：
- 具体到代码行/变量/配置项
- 有明确的验证方法（不是"可能是这个"）
- 能解释所有观察到的症状

**禁止**：同时验证多个假设（会混淆结果）。

### Step 4 — 验证与修复（Verify & Fix）

**目标**：按假设优先级逐一验证，找到根因后最小化修复。

**验证顺序**：
1. 先验证假设 #1（最可能的）
2. 如果证伪 → 验证假设 #2
3. 找到根因 → 实施最小化修复

**最小化修复原则**：
- 只改导致 bug 的那一处
- 不顺手重构周边代码
- 不添加"防御性"代码（除非有明确需要）

**验证修复**：
```bash
# 运行原始复现步骤，确认问题消失
# 运行相关测试，确认没有引入回归
# 如果有测试覆盖，先跑测试再改代码
```

### Step 5 — 防复发（Prevent Recurrence）

**目标**：确保同类问题不再出现。

**三个问题**：
1. **为什么会发生？** — 根因（技术层面）
2. **为什么没被发现？** — 测试/监控缺失
3. **如何防止再次发生？** — 具体行动

**防复发行动**（按成本排序，选最合适的）：
- 添加单元测试覆盖这个 case
- 添加类型检查（TypeScript / mypy）
- 添加运行时断言
- 更新文档/注释说明这个边界情况
- 写入 `memory/lessons-learned.md`（如果是可复用的经验）

## When Done（验收标准）

### 必须满足

1. **根因已确认** — 不是"可能是 X"，而是"确认是 X，因为 Y"
2. **修复已验证** — 原始复现步骤不再触发问题
3. **无回归** — 相关测试通过（或手动验证无回归）

### 建议满足

- 添加了测试覆盖这个 bug case
- 防复发建议已记录（至少口头说明）
- 如果是可复用经验，已写入 lessons-learned.md

---

## What NOT（边界约束）

🚫 **不做的事**：

1. **不跳过复现直接修** — 没有稳定复现就不改代码
2. **不同时改多处** — 一次只验证一个假设
3. **不顺手重构** — 修 bug 时不改无关代码
4. **不假设即结论** — 假设必须经过验证才能成为根因
5. **不忽略防复发** — 修完就走是在给未来埋雷

🚫 **不该触发的场景**：

- 功能需求开发（用 sdd-riper 或 sdd-riper-light）
- 代码审查（用 code-reviewer Agent）
- 性能优化（不是 bug，是优化需求）

## 调用时机

| 场景 | 是否调用 | 说明 |
|------|---------|------|
| 遇到报错/异常行为 | ✅ 强烈建议 | 主用法 |
| 测试失败，原因不明 | ✅ 建议 | 系统排查比随机尝试快 |
| 生产问题排查 | ✅ 建议 | 尤其需要防复发步骤 |
| 已知原因的简单修复 | ❌ 不需要 | 直接修即可 |
| 功能开发中的预期错误 | ❌ 不需要 | 开发过程的正常迭代 |

    ↓

    │

**协同 Skill / Agent**：
- `agents/debugger.md` — 调试专家 Agent，本 Skill 是其方法论的 Skill 化

- `.claude/skills/observability/SKILL.md` — 调试过程中记录决策点
- `memory/error-patterns.md` — 防复发经验的存储目标

## 实例

### 触发：用户说"这个 hooks 脚本不执行，帮我调试"

```
Step 1 复现:
  用户描述: hooks 配置了但没有触发
  复现: 触发 UserPromptSubmit → 确认 hooks 没有执行
  ✅ 稳定复现

Step 2 隔离:
  检查 settings.json hooks 配置
  检查脚本路径是否存在
  检查脚本是否有执行权限
  → 缩小到: settings.json 中 command 字段格式问题

Step 3 假设:
  假设 #1: Windows 上 ./script.sh 不工作，需要绝对路径
    验证方法: 改为 bash "C:\\path\\script.sh" 测试
  假设 #2: 脚本没有执行权限
    验证方法: ls -la 检查权限

Step 4 验证修复:
  验证假设 #1: 改为绝对路径 → hooks 正常触发 ✅
  最小化修复: 只改 command 字段，不动其他配置

Step 5 防复发:
  根因: Windows 不支持 ./相对路径 in hooks
  为什么没发现: 文档没有明确说明 Windows 兼容性
  防复发: 写入 CLAUDE.md hooks 配置规范 + lessons-learned
```

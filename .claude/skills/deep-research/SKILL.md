# Deep Research Skill

## 契约定义

### What（输入/输出）

**输入**：
- 研究主题或问题（自然语言描述）
- 可选：研究深度偏好（快速/标准/深度）
- 可选：输出格式偏好（简报/报告/详尽分析）
- 可选：特定关注领域或排除领域

**输出**：
- 结构化 Markdown 研究报告
- 包含具体数据点（数字、日期、统计、百分比）
- 多角度分析（depth-first）或多子问题覆盖（breadth-first）
- 保存为 `.md` 文件（默认路径：`reports/research-{topic}-{date}.md`）

### When Done（验收标准）

- 报告包含至少 5 个具体数据点（数字/日期/统计）
- 覆盖了主题的核心维度（根据查询类型）
- 所有事实性陈述有可追溯的来源基础
- 报告结构清晰（摘要 → 主体 → 结论）
- Lead Agent 完成了信息综合，而非简单拼接
- 文件已保存到指定路径

### What NOT（边界约束）

- **禁止编造数据**：所有数字必须来自搜索结果
- **禁止委托综合报告撰写**：Lead Agent 必须亲自撰写最终报告
- **禁止单一来源依赖**：重要事实需多来源交叉验证
- **禁止忽略矛盾信息**：发现矛盾时必须在报告中说明
- **禁止主对话污染**：所有搜索在隔离的 Task Agent 中执行
- **暂不添加引用标注**：当前版本不要求严格引用格式（后续可用 citations agent 补充）

## 何时使用此 Skill

当用户请求以下任务时自动激活：
- 调研某个主题/技术/市场/趋势
- 了解某事物的最新进展
- 对比分析多个选项
- 收集某领域的数据和统计
- 生成研究报告或调研报告
- 需要多角度、多来源的信息整合

**触发词示例**：
- "帮我调研一下..."
- "研究一下...的现状"
- "深入了解..."
- "收集...的相关信息"
- "对比分析...和..."
- "调查..."
- "综述..."

## 核心架构

### Lead Agent + Subagent 多轮反思模式

```
┌─────────────────────────────────────────────────────────────┐
│                    Lead Agent (主控)                         │
│  1. 生成研究简报（Research Brief）                           │
│  2. 多轮调度 Subagents，读取摘要后反思是否充分               │
│  3. 综合所有发现，撰写最终报告（不委托）                     │
└───────────────────────┬─────────────────────────────────────┘
                        │
              ┌─────────▼─────────┐
              │  生成研究简报      │  ← 锁定意图边界
              └─────────┬─────────┘
                        │
              ┌─────────▼─────────┐          ┌──────────────────┐
         ┌───►│  本轮调查（Round N）│          │  发现已充分？      │
         │    └─────────┬─────────┘          │  ├─ 是 → 退出     │
         │              │ fan-out            │  └─ 否 → 继续     │
         │    ┌──────────┴──────────┐        └──────────────────┘
         │    ▼          ▼          ▼
         │ Subagent1  Subagent2  SubagentN    ← 独立上下文，ReAct
         │    │          │          │
         │    └──────────┴──────────┘
         │         压缩摘要返回
         │              │
         │    ┌─────────▼─────────┐
         └────┤  Lead Agent 反思   │  ← 还有空白？继续
              │  评估发现充分度    │
              └─────────┬─────────┘
                        │ 充分 / 达到 max_rounds
                        ▼
              ┌─────────────────┐
              │   综合撰写报告   │
              └─────────────────┘
```

**双层终止条件**（同 Ralph 设计哲学）：
- **质量驱动**：Lead Agent 判断发现已充分，主动退出循环
- **资源驱动**：达到 `max_rounds`（默认 3）强制退出，防止无限调查

### Token 隔离模式

**关键原则**：所有搜索操作在独立的 Task Agent 中执行，避免主对话上下文污染

```
主对话上下文
    │
    ├─→ Task Agent 1 (独立上下文)
    │       ├─ web_search("query 1")
    │       ├─ web_fetch(url)
    │       └─ 返回精炼后的要点 (≤500 tokens)
    │
    ├─→ Task Agent 2 (独立上下文)
    │       ├─ web_search("query 2")
    │       ├─ web_fetch(url)
    │       └─ 返回精炼后的要点 (≤500 tokens)
    │
    └─→ 主对话仅接收精炼结果，Token 开销可控
```

## 工作流程

### 阶段 0: 研究简报生成（Research Brief）

**Lead Agent 职责**：在派出任何 Subagent 之前，先生成结构化研究简报，锁定意图边界。

```
研究简报格式：
- 核心研究问题：[用户真正想知道什么，而非原始措辞]
- 调查维度：[需要覆盖的子问题列表]
- 约束条件：[时间范围、地域、已知前提]
- 开放维度：[用户未指定但研究必需的维度，标记为"待调查"]
- 成功标准：[什么情况下视为"发现充分"可退出循环]
```

> 简报的作用：锁定意图（防止调查偏移）+ 定义退出条件（质量驱动终止）

1. **分析研究主题**
   - 理解用户意图和期望深度
   - 识别核心问题和边界

2. **确定查询类型**（三选一）

   | 类型 | 描述 | Subagent 数量 | 适用场景 |
   |------|------|--------------|---------|
   | **depth-first** | 一个问题，多角度深入 | 3-6 | 复杂主题、需要深度分析 |
   | **breadth-first** | 多个独立子问题 | 2-4 | 主题宽泛、需要覆盖面 |
   | **straightforward** | 直接事实查询 | 1-2 | 简单问题、快速获取 |

3. **分解初轮子任务**
   - 为每个 Subagent 定义明确的搜索目标
   - 确保子任务之间互补而非重复

### 阶段 1: 多轮调查（含反思循环）

**调查循环**（最多 `max_rounds` 轮，默认 3）：

```
Round N：
  1. Lead Agent 派发 Subagents → 并行搜索执行
  2. Subagents 返回压缩摘要
  3. Lead Agent 反思：
     - 对照研究简报的"成功标准"
     - 已发现了什么？还有哪些维度未覆盖？
     - 新发现是否指向需要追加调查的线索？
  4. 退出判断：
     - 发现充分 → 退出循环（质量驱动）
     - 达到 max_rounds → 退出循环（资源驱动）
     - 仍有关键空白 → Round N+1，只针对空白维度补充调查
```

**Subagent 职责**（在 Task Agent 中执行）：

1. **执行搜索**
   ```
   使用 web_search 获取初步结果
       ↓
   使用 web_fetch 获取详细内容（如需）
       ↓
   如果信息不足，使用 Playwright MCP（动态内容）
   ```

2. **信息精炼**
   - 提取关键数据点（数字、日期、统计）
   - 识别核心观点和事实
   - 标注是否存在矛盾或需追查的新线索
   - 压缩为精炼摘要（≤500 tokens）

3. **返回结果**
   - 结构化的发现列表
   - 标注信息可信度
   - 标注是否有矛盾信息
   - 标注发现的新线索（供 Lead Agent 反思时参考）

### 阶段 2: 综合与报告撰写

**Lead Agent 职责**（不可委托）：

1. **整合 Subagent 结果**
   - 合并相同/相似信息
   - 识别和处理矛盾信息
   - 构建完整的信息图谱

2. **撰写结构化报告**
   ```markdown
   # 研究报告：[主题]

   ## 执行摘要
   [3-5 句核心发现]

   ## 研究方法
   - 查询类型：[depth-first/breadth-first/straightforward]
   - 搜索维度：[列出各 Subagent 负责的维度]

   ## 核心发现
   ### 发现 1: [标题]
   - 关键数据：[具体数字/日期/统计]
   - 详细说明：...

   ### 发现 2: [标题]
   ...

   ## 数据汇总
   | 指标 | 数值 | 说明 |
   |------|------|------|
   | ... | ... | ... |

   ## 矛盾与不确定性
   [如有矛盾信息，在此说明]

   ## 结论与建议
   [基于发现的总结性结论]

   ## 后续研究方向
   [建议的深入研究点]

   ---
   生成时间：[时间戳]
   使用 Skill：deep-research v1.0.0
   ```

3. **保存报告**
   - 默认路径：`reports/research-{topic}-{YYYY-MM-DD}.md`
   - 可指定自定义路径

## 工具集成

### 必需工具

| 工具 | 用途 | 使用时机 |
|------|------|---------|
| **web_search** | 网络搜索 | 所有搜索任务的起点 |
| **web_fetch** | 获取页面内容 | 需要详细信息时 |
| **Task Agent** | 隔离上下文 | 所有 Subagent 搜索 |

### 可选工具（增强能力）

| 工具 | 用途 | 使用时机 |
|------|------|---------|
| **Playwright MCP** | 浏览器自动化 | 动态内容（SPA、登录墙） |
| **Exa** | 高级搜索 | web_search 结果不足时 |

### 升级策略

```
web_search 返回结果
    ↓
[结果充足?]
    ├─ 是 → 使用 web_fetch 获取详情
    └─ 否 → [需要动态内容?]
              ├─ 是 → 升级到 Playwright MCP
              └─ 否 → 调整搜索词重试
```

## 使用示例

### 示例 1: 技术调研（depth-first）

**用户**: 帮我深入研究一下 WebAssembly 在 2025 年的发展现状

**执行流程**:
1. Lead Agent 判断为 depth-first 类型
2. 分配 4 个 Subagent：
   - Subagent 1: 技术演进和规范更新
   - Subagent 2: 浏览器支持和性能数据
   - Subagent 3: 企业采用案例
   - Subagent 4: 生态工具和框架
3. 并行执行搜索
4. Lead Agent 综合撰写报告

### 示例 2: 市场对比（breadth-first）

**用户**: 对比分析目前主流的 AI 编程助手

**执行流程**:
1. Lead Agent 判断为 breadth-first 类型
2. 分配 3 个 Subagent：
   - Subagent 1: GitHub Copilot 功能和定价
   - Subagent 2: Claude Code 功能和定价
   - Subagent 3: Cursor 功能和定价
3. 并行执行搜索
4. Lead Agent 综合撰写对比报告

### 示例 3: 快速事实查询（straightforward）

**用户**: 查一下 OpenAI GPT-4o 的发布日期和主要参数

**执行流程**:
1. Lead Agent 判断为 straightforward 类型
2. 分配 1 个 Subagent 执行搜索
3. 快速返回事实性结果

## 限制与注意事项

### 当前限制

1. **暂不支持引用标注**：后续版本将集成 citations agent
2. **动态内容依赖 Playwright**：SPA 站点需要配置 Playwright MCP
3. **搜索结果受限**：部分网站可能有访问限制

### 最佳实践

1. **明确研究范围**：提供清晰的主题边界
2. **指定关注点**：告知最关心的方面
3. **适当深度选择**：避免过度或不足
4. **验证关键数据**：重要决策前人工复核

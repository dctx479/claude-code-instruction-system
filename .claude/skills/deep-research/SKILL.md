# Deep Research Skill

## 契约定义

### What（输入/输出）

**输入**：
- 研究主题或问题（自然语言描述）
- 可选：研究深度偏好（快速/标准/深度）
- 可选：输出格式偏好（简报/报告/详尽分析）
- 可选：特定关注领域或排除领域
- 可选：创新发现目标（如研究方向探索、跨领域组合、方案 shortlist）

**输出**：
- 结构化 Markdown 研究报告
- 包含具体数据点（数字、日期、统计、百分比）
- 多角度分析（depth-first）或多子问题覆盖（breadth-first）
- 在创新模式下输出：已有发现 → 可迁移机制 → 组合假设 → 风险/验证点 → shortlist
- 保存为 `.md` 文件（默认路径：`reports/research-{topic}-{date}.md`）

### When Done（验收标准）

- 报告包含至少 5 个具体数据点（数字/日期/统计）
- 覆盖了主题的核心维度（根据查询类型）
- 所有事实性陈述有可追溯的来源基础
- 报告结构清晰（摘要 → 主体 → 结论）
- Lead Agent 完成了信息综合，而非简单拼接
- 文件已保存到指定路径

### What NOT（边界约束）

- **禁止编造数据**：所有数字必须来自搜索结果
- **禁止委托综合报告撰写**：Lead Agent 必须亲自撰写最终报告
- **禁止单一来源依赖**：重要事实需多来源交叉验证
- **禁止忽略矛盾信息**：发现矛盾时必须在报告中说明
- **禁止主对话污染**：所有搜索在隔离的 Task Agent 中执行
- **暂不添加引用标注**：当前版本不要求严格引用格式（后续可用 citations agent 补充）

## 何时使用此 Skill

当用户请求以下任务时自动激活：
- 调研某个主题/技术/市场/趋势
- 了解某事物的最新进展
- 对比分析多个选项
- 收集某领域的数据和统计
- 生成研究报告或调研报告
- 需要多角度、多来源的信息整合

**触发词示例**：
- "帮我调研一下..."
- "研究一下...的现状"
- "深入了解..."
- "收集...的相关信息"
- "对比分析...和..."
- "调查..."
- "综述..."
- "帮我找 10 个可验证的新方向"
- "把 A 和 B 组合成可落地的 idea"
- "从多篇论文里筛选靠谱的创新方案"

## 核心架构

### Lead Agent + Subagent 多轮反思模式

```
┌─────────────────────────────────────────────────────────────┐
│                    Lead Agent (主控)                         │
│  1. 生成研究简报（Research Brief）                           │
│  2. 多轮调度 Subagents，读取摘要后反思是否充分               │
│  3. 综合所有发现，撰写最终报告（不委托）                     │
└───────────────────────┬─────────────────────────────────────┘
                        │
              ┌─────────▼─────────┐
              │  生成研究简报      │  ← 锁定意图边界
              └─────────┬─────────┘
                        │
              ┌─────────▼─────────┐          ┌──────────────────┐
         ┌───►│  本轮调查（Round N）│          │  发现已充分？      │
         │    └─────────┬─────────┘          │  ├─ 是 → 退出     │
         │              │ fan-out            │  └─ 否 → 继续     │
         │    ┌──────────┴──────────┐        └──────────────────┘
         │    ▼          ▼          ▼
         │ Subagent1  Subagent2  SubagentN    ← 独立上下文，ReAct
         │    │          │          │
         │    └──────────┴──────────┘
         │         压缩摘要返回
         │              │
         │    ┌─────────▼─────────┐
         └────┤  Lead Agent 反思   │  ← 还有空白？继续
              │  评估发现充分度    │
              └─────────┬─────────┘
                        │ 充分 / 达到 max_rounds
                        ▼
              ┌─────────────────┐
              │   综合撰写报告   │
              └─────────────────┘
```

**双层终止条件**（同 Ralph 设计哲学）：
- **质量驱动**：Lead Agent 判断发现已充分，主动退出循环
- **资源驱动**：达到 `max_rounds`（默认 3）强制退出，防止无限调查

### Token 隔离模式

**关键原则**：所有搜索操作在独立的 Task Agent 中执行，避免主对话上下文污染

```
主对话上下文
    │
    ├─→ Task Agent 1 (独立上下文)
    │       ├─ web_search("query 1")
    │       ├─ web_fetch(url)
    │       └─ 返回精炼后的要点 (≤500 tokens)
    │
    ├─→ Task Agent 2 (独立上下文)
    │       ├─ web_search("query 2")
    │       ├─ web_fetch(url)
    │       └─ 返回精炼后的要点 (≤500 tokens)
    │
    └─→ 主对话仅接收精炼结果，Token 开销可控
```

## 工作流程

### 阶段 0: 研究简报生成（Research Brief）

#### Question Refiner 集成（可选前置）

在生成研究简报之前，Lead Agent 可选择调用 `question-refiner` Skill 对查询进行预处理：

**触发条件**（满足任一即触发）：
- 查询长度 <10 字且无法判断研究类型
- 查询范围过于宽泛，无法直接分解为子任务
- 用户显式请求帮助澄清研究方向

**不触发条件**（满足任一即跳过）：
- 查询已包含明确主题、范围和期望输出
- 用户已指定研究深度（快速/标准/深度）
- 查询为直接事实型问题（straightforward 类型）

**集成流程**：
1. Lead Agent 评估查询清晰度
2. 若需精炼 → 调用 question-refiner 的 7 项验证清单
3. question-refiner 输出 Structured Research Prompt
4. Lead Agent 基于精炼后的 Prompt 生成研究简报（替代原始查询）

> 此步骤完全可选，已清晰的查询将直接进入研究简报生成，不增加任何交互开销。

**Lead Agent 职责**：在派出任何 Subagent 之前，先生成结构化研究简报，锁定意图边界。

```
研究简报格式：
- 核心研究问题：[用户真正想知道什么，而非原始措辞]
- 调查维度：[需要覆盖的子问题列表]
- 约束条件：[时间范围、地域、已知前提]
- 开放维度：[用户未指定但研究必需的维度，标记为"待调查"]
- 成功标准：[什么情况下视为"发现充分"可退出循环]
```

> 简报的作用：锁定意图（防止调查偏移）+ 定义退出条件（质量驱动终止）

1. **分析研究主题**
   - 理解用户意图和期望深度
   - 识别核心问题和边界

2. **确定查询类型**（三选一）

   | 类型 | 描述 | Subagent 数量 | 适用场景 |
   |------|------|--------------|---------|
   | **depth-first** | 一个问题，多角度深入 | 3-6 | 复杂主题、需要深度分析 |
   | **breadth-first** | 多个独立子问题 | 2-4 | 主题宽泛、需要覆盖面 |
   | **straightforward** | 直接事实查询 | 1-2 | 简单问题、快速获取 |

3. **分解初轮子任务**
   - 为每个 Subagent 定义明确的搜索目标
   - 确保子任务之间互补而非重复

### 阶段 0.5: 创新组合模式（可选）

> **启用条件**：当用户目标是研究方向探索、跨领域迁移、组合式方案生成、研究空白发现或 shortlist 筛选时启用。

**Lead Agent 职责**：在常规调研之外，将发现转化为可检验的候选方案。

1. **建立候选池**
   - 收集可复用的论文、方法、机制、案例或产品模式
   - 保留最新、优质、可验证的候选项，避免把低可信度材料直接纳入组合

2. **抽取可迁移机制**
   - 为每个候选项提炼核心机制、适用前提、关键资源和失败边界
   - 优先提炼能够跨领域迁移的抽象模式，而不是停留在表面术语

3. **生成组合假设**
   - 对候选机制做两两或多维组合
   - 排除自我组合和明显冲突的组合
   - 将组合表达为“已有发现 → 可迁移机制 → 组合假设”

4. **快速验证与筛选**
   - 通过补充检索、逻辑约束、开源实现线索或小规模实验设想验证可行性
   - 标注每个假设的风险点、依赖条件和验证入口
   - 最终输出 shortlist，而不是平铺所有组合
### 阶段 1: 多轮调查（含反思循环）

**调查循环**（最多 `max_rounds` 轮，默认 3）：

```
Round N：
  1. Lead Agent 派发 Subagents → 并行搜索执行
  2. Subagents 返回压缩摘要
  3. Lead Agent 反思：
     - 对照研究简报的"成功标准"
     - 已发现了什么？还有哪些维度未覆盖？
     - 新发现是否指向需要追加调查的线索？
  4. 退出判断：
     - 发现充分 → 退出循环（质量驱动）
     - 达到 max_rounds → 退出循环（资源驱动）
     - 仍有关键空白 → Round N+1，只针对空白维度补充调查
```

**Subagent 职责**（在 Task Agent 中执行）：

1. **执行搜索**
   ```
   使用 web_search 获取初步结果
       ↓
   使用 web_fetch 获取详细内容（如需）
       ↓
   如果信息不足，使用 Playwright MCP（动态内容）
   ```

2. **信息精炼**
   - 提取关键数据点（数字、日期、统计）
   - 识别核心观点和事实
   - 标注是否存在矛盾或需追查的新线索
   - 压缩为精炼摘要（≤500 tokens）

3. **返回结果**
   - 结构化的发现列表
   - 标注信息可信度
   - 标注是否有矛盾信息
   - 标注发现的新线索（供 Lead Agent 反思时参考）

### 阶段 2: 综合与报告撰写

**Lead Agent 职责**（不可委托）：

1. **整合 Subagent 结果**
   - 合并相同/相似信息
   - 识别和处理矛盾信息
   - 构建完整的信息图谱

2. **交叉引用验证**
   - 对关键数值型发现，检查多个 Subagent 的数据是否一致
   - 数值差异 >20% 的发现标记为 ⚠️ 需进一步验证
   - 仅单一来源支持的关键事实标注为"待确认"

3. **矛盾解决**
   - 当多个来源信息矛盾时，按以下优先级判定：
     | 优先级 | 来源类型 | 说明 |
     |--------|---------|------|
     | 1 | 官方/一手数据 | 年报、政府统计、学术论文 |
     | 2 | 权威二手来源 | 知名咨询机构、行业报告 |
     | 3 | 媒体报道 | 主流财经/科技媒体 |
     | 4 | 社区/自媒体 | 论坛、博客、社交媒体 |
   - 无法判定时，在报告中并列呈现并注明差异

4. **发现置信度评分**
   - 为每个核心发现标注置信度等级：
     | 等级 | 条件 | 标记 |
     |------|------|------|
     | **高** | ≥2 个独立来源一致 | ✅ |
     | **中** | 单一权威来源或多来源部分一致 | ➖ |
     | **低** | 单一非权威来源或来源间矛盾 | ⚠️ |

2. **撰写结构化报告**
   ```markdown
   # 研究报告：[主题]

   ## 执行摘要
   [3-5 句核心发现]

   ## 研究方法
   - 查询类型：[depth-first/breadth-first/straightforward]
   - 搜索维度：[列出各 Subagent 负责的维度]

   ## 核心发现
   ### 发现 1: [标题]
   - 关键数据：[具体数字/日期/统计]
   - 详细说明：...

   ### 发现 2: [标题]
   ...

   ## 创新组合 Shortlist（可选）
   | 组合假设 | 来源机制 | 可行性信号 | 风险/验证点 |
   |----------|----------|------------|-------------|
   | [假设 1] | [A+B] | [已有证据] | [关键风险] |

   ## 发现置信度概览
   | 发现 | 置信度 | 来源数 | 说明 |
   |------|--------|--------|------|
   | [发现 1] | ✅ 高 | 3 | 多来源一致 |
   | [发现 2] | ➖ 中 | 1 | 单一权威来源 |
   | [发现 3] | ⚠️ 低 | 1 | 来源间矛盾 |

   ## 数据汇总
   | 指标 | 数值 | 说明 |
   |------|------|------|
   | ... | ... | ... |

   ## 矛盾与不确定性
   [如有矛盾信息，在此说明]

   ## 结论与建议
   [基于发现的总结性结论]

   ## 后续研究方向
   [建议的深入研究点]

   ---
   生成时间：[时间戳]
   使用 Skill：deep-research v1.2.0
   ```

3. **保存报告**
   - 默认路径：`reports/research-{topic}-{YYYY-MM-DD}.md`
   - 可指定自定义路径

### 阶段 2.5: 引用验证（可选）

> **启用条件**：仅在**深度模式**或用户**显式请求引用验证**时启用。Quick/Standard 模式跳过此阶段。

在报告撰写完成后，对关键发现的来源进行可信度评估：

**来源可信度 5 级评分**：

| 等级 | 来源类型 | 示例 |
|------|---------|------|
| ★★★★★ | 一手/官方数据 | 政府统计、公司年报、学术论文、SEC 文件 |
| ★★★★☆ | 权威二手来源 | 知名咨询机构报告、行业白皮书 |
| ★★★☆☆ | 专业媒体 | Reuters、Bloomberg、专业行业媒体 |
| ★★☆☆☆ | 一般媒体 | 新闻网站、科技博客 |
| ★☆☆☆☆ | 用户生成内容 | 社交媒体、论坛、个人博客 |

**交叉验证矩阵**：

对每个核心发现，记录：
- 支持来源数量和最高可信度等级
- 是否有矛盾来源
- 最终置信度评级（高/中/低）

**低置信度标注**：

当核心发现的置信度为"低"时，在报告中以 ⚠️ 标注，并附注：
```
⚠️ 此发现置信度较低：[原因说明，如"仅有单一非权威来源"或"来源间存在矛盾"]
```

## 工具集成

### 必需工具

| 工具 | 用途 | 使用时机 |
|------|------|---------|
| **web_search** | 网络搜索 | 所有搜索任务的起点 |
| **web_fetch** | 获取页面内容 | 需要详细信息时 |
| **Task Agent** | 隔离上下文 | 所有 Subagent 搜索 |

### 可选工具（增强能力）

| 工具 | 用途 | 使用时机 |
|------|------|---------|
| **Playwright MCP** | 浏览器自动化 | 动态内容（SPA、登录墙） |
| **Exa** | 高级搜索 | web_search 结果不足时 |

### 升级策略

```
web_search 返回结果
    ↓
[结果充足?]
    ├─ 是 → 使用 web_fetch 获取详情
    └─ 否 → [需要动态内容?]
              ├─ 是 → 升级到 Playwright MCP
              └─ 否 → 调整搜索词重试
```

## 使用示例

### 示例 1: 技术调研（depth-first）

**用户**: 帮我深入研究一下 WebAssembly 在 2025 年的发展现状

**执行流程**:
1. Lead Agent 判断为 depth-first 类型
2. 分配 4 个 Subagent：
   - Subagent 1: 技术演进和规范更新
   - Subagent 2: 浏览器支持和性能数据
   - Subagent 3: 企业采用案例
   - Subagent 4: 生态工具和框架
3. 并行执行搜索
4. Lead Agent 综合撰写报告

### 示例 2: 市场对比（breadth-first）

**用户**: 对比分析目前主流的 AI 编程助手

**执行流程**:
1. Lead Agent 判断为 breadth-first 类型
2. 分配 3 个 Subagent：
   - Subagent 1: GitHub Copilot 功能和定价
   - Subagent 2: Claude Code 功能和定价
   - Subagent 3: Cursor 功能和定价
3. 并行执行搜索
4. Lead Agent 综合撰写对比报告

### 示例 3: 快速事实查询（straightforward）

**用户**: 查一下 OpenAI GPT-4o 的发布日期和主要参数

**执行流程**:
1. Lead Agent 判断为 straightforward 类型
2. 分配 1 个 Subagent 执行搜索
3. 快速返回事实性结果

## 质量控制

### 数据点验证清单

- [ ] 报告包含至少 5 个具体数据点
- [ ] 数字数据有上下文说明（来源、时间、范围）
- [ ] 矛盾信息已标注并说明
- [ ] 无编造或推测的数据

### 报告质量清单

- [ ] 执行摘要简明扼要
- [ ] 结构层次清晰
- [ ] 发现有数据支撑
- [ ] 结论基于发现推导

## 配置选项

### 配置选项

### 深度级别

| 级别 | Subagent 数量 | max_rounds | 搜索深度 | 报告长度 | 适用场景 |
|------|--------------|-----------|---------|---------|---------|
| **快速** | 1-2 | 1 | 浅 | 500-1000 词 | 快速了解、事实查询 |
| **标准** | 3-4 | 2 | 中 | 1500-2500 词 | 常规调研 |
| **深度** | 5-6 | 3 | 深 | 3000-5000 词 | 深度分析、决策支持 |

### 输出格式

| 格式 | 说明 | 适用场景 |
|------|------|---------|
| **简报** | 要点列表，精简 | 快速汇报 |
| **报告** | 完整结构，标准 | 常规使用 |
| **详尽分析** | 深度论述，详细 | 重要决策 |

## 限制与注意事项

### 当前限制

1. **引用验证为可选**：深度模式下可启用 Citation Validation，但非默认行为
2. **动态内容依赖 Playwright**：SPA 站点需要配置 Playwright MCP
3. **搜索结果受限**：部分网站可能有访问限制

### 最佳实践

1. **明确研究范围**：提供清晰的主题边界
2. **指定关注点**：告知最关心的方面
3. **适当深度选择**：避免过度或不足
4. **验证关键数据**：重要决策前人工复核

## 领域扩展协议

### 设计理念

deep-research 作为通用研究基座，通过**领域扩展**机制支持特定领域的深度增强。扩展以叠加方式工作，不修改基座行为，确保向后兼容。

### 目录结构

```
.claude/skills/deep-research/
├── SKILL.md           # 基座 Skill（本文件）
├── EXTENSIONS.md      # 扩展指南和模板
└── extensions/        # 领域扩展文件目录
    └── {domain}-research.md
```

### 扩展文件格式

每个扩展包含 4 个部分，均为叠加式：

| 部分 | 说明 | 约束 |
|------|------|------|
| 领域问题集 | 追加在 question-refiner 7 项之后 | ≤5 个，编号从 8 开始 |
| 分析维度 | 追加在基座调查维度之后 | ≤5 个，不与基座重复 |
| 报告模板扩展 | 插入"核心发现"和"结论"之间 | ≤3 个追加章节 |
| 搜索策略 | 指导 Subagent 搜索行为 | 含可信度分级信息源 |

### 加载协议

1. Lead Agent 在阶段 0 识别研究领域
2. 检查 `extensions/` 目录是否有匹配的 `{domain}-research.md`
3. 如有匹配 → 读取扩展文件，将 4 部分内容叠加到对应阶段
4. 如无匹配 → 使用纯基座能力执行（不影响任何现有行为）

### 不变保证

- 无扩展文件时，行为与 v1.1.0 完全一致
- 扩展仅叠加，不覆盖或替换基座任何部分
- 每次研究最多加载 1 个领域扩展
- 扩展加载失败时静默降级为基座模式

> 详细扩展模板和完整示例见 `EXTENSIONS.md`

## 版本历史

### v1.2.0 (2026-03-08)
- 新增 Question Refiner 集成（可选前置）：模糊查询自动精炼，清晰查询零摩擦跳过
- 新增综合增强：交叉引用验证（差异 >20% 标记）+ 矛盾解决矩阵（4 级优先级）+ 发现置信度评分（高/中/低）
- 新增阶段 2.5 引用验证（可选）：来源 5 级可信度评分，仅深度模式启用
- 新增领域扩展协议：基座 + 扩展叠加架构，支持特定领域深度增强
- 报告模板新增"发现置信度概览"表格章节
- 完全向后兼容：Quick/Standard 模式行为与 v1.1.0 一致

### v1.1.0 (2026-03-06)
- 新增 Research Brief 生成步骤（阶段 0）：显式锁定研究意图和退出标准
- 升级为多轮反思循环架构：Lead Agent 每轮评估发现充分度，不足则针对性补充
- 新增双层终止机制：质量驱动（发现充分）+ 资源驱动（max_rounds）
- 新增 `max_rounds` 配置参数，各深度级别有对应默认值
- Subagent 返回摘要新增"新线索标注"字段

### v1.0.0 (2026-02-04)
- 初始版本

- 三种查询类型支持
- Token 隔离模式
- 基础报告生成

# Paper Revision Skill

## 契约定义

### What（输入/输出）

**输入**：
- 中文原文（技术性或学术性描述）
- 可选：目标风格偏好（解释性/精简/口语化）

**输出**：
- 修改后的文本，符合目标风格
- 字数与原文基本相当（±10%）
- 保持所有技术术语准确

### When Done（验收标准）

- 输出字数与原文相当
- 所有技术术语、代码、路径保持原样
- 风格转换符合预期（更解释性/通俗化）
- 核心技术逻辑和因果关系未改变

### What NOT（边界约束）

- **禁止修改技术术语**：Django, RESTful API, MySQL 等必须保持原样
- **禁止修改代码片段**：views.py, settings.py 等不得改动
- **禁止过于口语化**：不使用"xxx呢"等过度口语表达
- **禁止使用第一人称**：不使用"我"、"我们"
- **禁止增加过多字数**：输出不应显著多于原文
- **禁止添加 emoji**：除非用户明确要求
- **禁止附带说明**：只输出修改后的文本，不附加"我修改了XXX"等解释
- **禁止破坏表格结构**：表格必须保持原有行列格式，不得改写为段落
- **禁止改变公式**：LaTeX 公式、数学符号、统计量（α/β/p/F值）原样保留
- **禁止统一改变一词多义词**：如"性能"在不同语境含义不同，需按语境判别后再改

### 输出纯净性规则（强制）

修改完成后：
- **只输出修改后的文本**
- 不输出任何解释、修改理由或评注
- 不输出"修改前/修改后"对比（除非用户明确要求对比模式）
- 若有不确定的术语，保留原文并在该词后加 `【待确认】` 标记

## 何时使用此 Skill

当用户请求以下任务时自动激活：
- 修改论文/技术文档的表述风格
- 将简洁描述改写为更解释性的版本
- 学术文本通俗化（保持专业底线）
- 统一文档的措辞风格

## 核心修改手法

### 1. 增加冗余与解释性（Verbose Elaboration）

**动词短语扩展**：

| 原文 | 修改后 |
|------|--------|
| 管理 | 开展...的管理工作 / 进行管理 |
| 交互 | 进行交互 / 开展交互 |
| 配置 | 进行配置 |
| 处理 | 去处理...工作 |
| 恢复 | 进行恢复 |
| 实现 | 得以实现 / 来实现 |
| 提供功能 | 有...功能 / 拥有...功能 |

**增加辅助词**：
- 适当增加：了、的、地、所、会、可以、这个、方面、当中

### 2. 系统性词汇替换

**动词/介词/连词**：

| 原文 | 替换为 |
|------|--------|
| 采用 / 使用 | 运用 / 选用 / 把...当作...来使用 |
| 基于 | 鉴于 / 基于...来开展 |
| 利用 | 借助 / 运用 / 凭借 |
| 通过 | 借助 / 依靠 / 凭借 |
| 和 / 及 / 与 | 以及（列举时） |
| 并 | 并且 / 还 / 同时 |
| 其 | 它 / 其（根据语境） |

**名词/形容词**：

| 原文 | 替换为 |
|------|--------|
| 原因 | 缘由 / 主要原因囊括... |
| 符合 | 契合 |
| 适合 | 适宜 |
| 特点 | 特性 |
| 极大(地) | 极大程度(上) |
| 立即 | 马上 |

### 3. 括号内容处理

**解释性括号**（优先整合）：

| 原文 | 修改后 |
|------|--------|
| ORM（对象关系映射） | 对象关系映射即ORM / ORM也就是对象关系映射 |
| 功能（如ORM、Admin） | 功能，比如ORM、Admin / 功能，像ORM、Admin等 |

**代码/标识符旁括号**（直接移除）：

| 原文 | 修改后 |
|------|--------|
| 视图 (views.py) 中 | 视图也就是views.py中 |
| 权限类 (admin_panel.permissions) | 权限类 admin_panel.permissions |

### 4. 句式微调

**使用"把"字句**：
- "会将对象移动" → "会把对象移动"

**条件句式转换**：
- "若...，则..." → "要是...，那就..." / "如果...，就..."

**增加连接词**：
- 在句首或句中添加"那么"、"这样"、"同时"等

## 绝对禁止修改清单

以下内容**必须保持原样**：

| 类型 | 示例 |
|------|------|
| 技术术语 | Django, RESTful API, Ceph, RGW, S3, JWT, ORM, MySQL |
| 代码片段 | views.py, settings.py, accounts.CustomUser, .folder_marker |
| 库名 | Boto3, djangorestframework-simplejwt |
| 配置项 | CEPH_STORAGE, DATABASES |
| API 路径 | /accounts/api/token/refresh/ |
| 数值和单位 | 100MB, 3.14, 1024 |
| LaTeX 公式 | $\alpha$, $p < 0.05$, $F(2,97)=3.14$ |
| 引用编号 | [12], [Smith et al., 2023], 图1, 表3 |
| 表格结构 | 保留所有行列，不转写为段落 |

## 用户可配置术语表

> **使用方法**：用户可在调用时提供领域术语映射，Skill 会在全文保持一致替换。
> 若未提供，使用下方通用规则。

### 声明格式（用户在 prompt 中附加）

```
[术语表]
采用 → 运用
使用 → 借助
基于 → 基于...来开展
利用 → 凭借
通过 → 借助
[/术语表]
```

### 术语一致性规则

- 同一术语在全文**只使用一种替换方式**，不随机变化
- 首次出现的替换词确定后，后续所有出现必须保持相同
- 一词多义情况：先判断语义，在各自语义范围内保持一致
  - 例："性能"作为 performance 时 → "性能表现"；作为 characteristic 时 → "特性"
  - 例："优化"作为 optimization 时保留；作为 improve 时 → "改进"

## 使用示例

### 示例 1: 基本修改

**原文**：
> 系统采用 Django 框架，通过 ORM 管理数据库，并利用 JWT 进行用户认证。

**修改后**：
> 系统运用 Django 框架，借助 ORM 来开展数据库的管理工作，同时凭借 JWT 进行用户认证。

### 示例 2: 括号处理

**原文**：
> 视图层 (views.py) 负责处理 HTTP 请求，使用装饰器（如 @api_view）标记 API 端点。

**修改后**：
> 视图层也就是 views.py 负责去处理 HTTP 请求的工作，运用装饰器比如 @api_view 来标记 API 端点。

### 示例 3: 条件句式

**原文**：
> 若用户未登录，则系统将返回 401 错误。

**修改后**：
> 如果用户未登录，那么系统就会返回 401 错误。

## 质量检查清单

修改完成后自动检查：

- [ ] 字数是否与原文相当（±10%）
- [ ] 技术术语是否保持原样
- [ ] 代码片段是否未被修改
- [ ] 是否有"xxx呢"等过度口语化表达
- [ ] 是否使用了第一人称
- [ ] 核心逻辑是否保持一致

## 与写作系统集成

此 Skill 与太一元系统的科研支持系统配合使用：

- **paper-writing-assistant Agent**: 生成初稿后用此 Skill 调整风格
- **literature-review 命令**: 综述写作的风格统一
- **qa-reviewer Agent**: 审查修改后的文本质量

## 注意事项

1. **保持专业底线**：通俗化不等于不专业
2. **尊重原文逻辑**：修改措辞，不改技术内容
3. **适度修改**：不要过度添加冗余
4. **风格一致**：整篇文档的修改风格要统一

---

**使用提示**：提供原文时，可以指定目标风格偏好（如"更解释性"、"略微通俗"、"学术正式"）。

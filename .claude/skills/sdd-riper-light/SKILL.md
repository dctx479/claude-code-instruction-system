# SDD-RIPER Light

## 核心理念

**大道至简**：保留 RIPER 的核心价值（质量可控、过程可追溯），去除繁重的流程负担。

## 适用场景

- ✅ 熟练用户（已理解 RIPER 流程）
- ✅ 简单任务（预计 <500 行代码或涉及 <3 个文件）
- ✅ 快速迭代（需要敏捷响应）
- ❌ 新手用户（建议使用标准版 sdd-riper）
- ❌ 复杂架构设计（建议使用标准版）

## 与标准版的区别

| 维度 | 标准版 (sdd-riper) | 轻量版 (sdd-riper-light) |
|------|-------------------|-------------------------|
| Spec 文档 | 完整 8 章节 | Micro-Spec（1-2 段） |
| Research | 必须 | 可选（自动判断） |
| Innovate | 2-3 个方案 | 1 个方案（快速决策） |
| Plan | 原子级拆解 | 高层级步骤 |
| Plan Approved | 必须等待 | 可选（默认自动批准） |
| Execute Log | 详细记录 | 简要摘要 |
| Review | 三角验证 | 快速检查 |
| 知识沉淀 | 自动归档 | 可选 |

## 快速启动

```bash
# 方式 1: 自动判断（推荐）
"修复登录页面的空指针异常"
# → Claude 自动判断为简单任务，使用 Light 版

# 方式 2: 显式启动

```

## 流程概览

```
┌─────────────┐
│  Research   │ 快速扫描（可选）
│  (可选)      │ - 定位问题代码
└──────┬──────┘
       │
┌──────▼──────┐
│  Innovate   │ 单方案（快速决策）
│  (简化)      │ - 直接给出推荐方案
└──────┬──────┘
       │
┌──────▼──────┐
│    Plan     │ 高层级步骤
│  (简化)      │ - 3-5 个步骤
│             │ - 默认自动批准
└──────┬──────┘
       │
┌──────▼──────┐
│  Execute    │ 快速实现
│  (标准)      │ - 按步骤执行
└──────┬──────┘
       │
┌──────▼──────┐
│   Review    │ 快速检查
│  (简化)      │ - 验证核心功能
└─────────────┘
```

## Micro-Spec 格式

```markdown
# Micro-Spec: [任务名称]

## 目标
[1-2 句话描述要做什么]

## 范围
- ✅ [包含的功能点]
- ❌ [不包含的功能点]

## 约束
- [关键约束条件]

## 验收
- [ ] [验收标准 1]
- [ ] [验收标准 2]
```

## 示例：修复 Bug

### 输入
```
修复登录页面的空指针异常，用户点击"忘记密码"时崩溃
```

### 输出

#### Micro-Spec
```markdown
# Micro-Spec: 修复登录页面空指针异常

## 目标
修复用户点击"忘记密码"按钮时的空指针异常

## 范围
- ✅ 修复 LoginPage.tsx 的空指针问题
- ❌ 不重构整个登录流程

## 约束
- 不能影响现有的登录功能
- 必须向后兼容

## 验收
- [ ] 点击"忘记密码"不再崩溃
- [ ] 现有登录功能正常
- [ ] 单元测试通过
```

#### Plan（简化版）
```markdown
1. 定位空指针来源（LoginPage.tsx:L45）
2. 添加空值检查
3. 更新单元测试
```

#### Execute Summary
```markdown
✅ 已修复 LoginPage.tsx:L45 的空指针问题
✅ 添加了 `user?.email` 可选链
✅ 更新了 LoginPage.test.tsx
```

#### Review
```markdown
✅ 验收通过
- 点击"忘记密码"功能正常
- 现有登录功能未受影响
- 测试覆盖率保持 92%
```

## 示例：小功能开发

```
在用户个人资料页面添加"导出数据"按钮
```

```markdown

## 目标

## 范围
- ✅ 添加 UI 按钮

- [ ] 文件包含用户的所有数据
```

```markdown
✅ 实现了 exportUserData() 函数（utils/export.ts）
```

## 自动判断逻辑

Claude 会根据以下特征自动选择 Light 版或标准版：

| 特征 | Light 版 | 标准版 |
|------|---------|--------|
| 需求描述长度 | <100 字 | >100 字 |
| 预计文件数 | <3 个 | >3 个 |
| 是否需要架构设计 | 否 | 是 |
| 是否需要方案对比 | 否 | 是 |
| 用户熟练度 | 熟练 | 新手 |

## 何时升级到标准版

如果在 Light 版执行过程中发现以下情况，会自动建议升级：

- ⚠️ 需求边界不清晰（需要更详细的 Research）
- ⚠️ 发现多个可行方案（需要 Innovate 对比）
- ⚠️ 任务复杂度超出预期（涉及 >5 个文件）
- ⚠️ 需要架构级别的决策

## 最佳实践

### 1. 清晰的需求描述
```
✅ 好："修复登录页面的空指针异常，用户点击'忘记密码'时崩溃"
❌ 差："登录有问题"
```

### 2. 明确的验收标准
```
✅ 好："点击按钮后下载 JSON 文件"
❌ 差："导出功能正常"
```

### 3. 合理的范围控制
```
✅ 好："只修复这个 Bug，不重构整个模块"
❌ 差："顺便把整个登录流程都优化一下"
```

## 常见问题

### Q1: Light 版会牺牲质量吗？
A: 不会。Light 版保留了核心的质量保障机制（Plan + Execute + Review），只是简化了流程。

### Q2: 什么时候应该用 Light 版？
A: 当你已经熟悉 RIPER 流程，且任务边界清晰、复杂度低时。

### Q3: Light 版支持 Plan Approved 门禁吗？
A: 支持，但默认自动批准。如果需要人工审批，可以设置 `manual_approval: true`。

### Q4: Light 版的 Spec 可以复用吗？
A: 可以。Micro-Spec 虽然简洁，但包含了核心信息，可以作为知识沉淀。

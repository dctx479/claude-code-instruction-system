# Task Decompose Skill

## 契约定义

### What（输入/输出）

**输入**：
- 模糊需求描述（自然语言）
- 可选：技术栈约束
- 可选：时间/人力约束
- 可选：已知依赖关系

**输出**：
- 原子任务列表（每个任务 ≤ 4 小时）
- 依赖关系图（哪些任务可并行）
- 关键路径标注
- 风险任务标记（技术不确定性高）

### When Done（验收标准）

每个原子任务必须满足：
1. **可独立执行**：不依赖未完成的任务（或依赖已明确标注）
2. **可验收**：有明确的完成标准（Done Criteria）
3. **有边界**：输入和输出明确，不模糊
4. **原子性**：不能再拆分为更小的有意义单元
5. **估时合理**：单任务 ≤ 4 小时（超过则继续拆分）

### What NOT（边界约束）

- **禁止过度拆分**：不把"写一个函数"拆成"写函数签名"+"写函数体"
- **禁止忽略依赖**：有依赖关系的任务必须在图中标注，不能假装独立
- **禁止遗漏测试任务**：每个功能任务必须有对应的测试任务
- **禁止跳过风险评估**：技术不确定性高的任务必须标记 ⚠️

## 三步分解流程

### Step 1: 需求澄清

在分解前，先确认以下问题（如用户未提供）：

```
1. 这个功能的核心用户价值是什么？（一句话）
2. 完成的标准是什么？（可测试的验收条件）
3. 有哪些已知的技术约束？
4. 有哪些不在范围内的事项？（Out of Scope）
```

如果需求过于模糊，先输出澄清问题，等用户回答后再分解。

### Step 2: 分层拆解

**三层结构**：

```
Epic（史诗）
└── Story（用户故事）
    └── Task（原子任务）
```

**拆解规则**：
- Epic → Story：按用户价值维度拆分（登录、注册、找回密码）
- Story → Task：按技术实现维度拆分（API、UI、测试、文档）
- Task 粒度：≤ 4 小时，有明确输入/输出

**任务类型标签**：
- `[BE]` 后端实现
- `[FE]` 前端实现
- `[DB]` 数据库/Schema
- `[TEST]` 测试编写
- `[DOC]` 文档更新
- `[INFRA]` 基础设施/配置

### Step 3: 依赖分析与并行化

**依赖图格式**：

```
[DB] 设计 users 表 Schema
    ↓
[BE] 实现用户注册 API ──────────────────┐
    ↓                                   ↓
[TEST] 单元测试注册逻辑          [FE] 注册表单 UI
    ↓                                   ↓
[TEST] 集成测试注册流程 ←──────────────┘
    ↓
[DOC] 更新 API 文档
```

**并行化建议**：
- 标注哪些任务可以同时进行（无依赖关系）
- 标注关键路径（最长依赖链）
- 建议人力分配（如有多人协作）

## 输出格式

### 任务卡片格式

```markdown
### T-001 [BE] 实现用户注册 API

**输入**: 用户邮箱、密码、用户名
**输出**: 用户 ID、JWT token
**依赖**: T-000（DB Schema 完成）
**估时**: 3h
**风险**: 低
**Done Criteria**:
- POST /api/auth/register 返回 201
- 密码已 bcrypt 加密存储
- 重复邮箱返回 409
- 单元测试覆盖率 ≥ 80%
```

### 完整输出示例

```markdown
## 任务分解：用户认证系统

**总估时**: 24h | **关键路径**: 16h | **可并行**: 3 条线

### 依赖图
[DB] Schema → [BE] API → [TEST] 集成测试 → [DOC]
                ↑
[FE] 表单 UI ──┘（可与 BE 并行）

### 任务列表

| ID | 类型 | 任务 | 估时 | 依赖 | 风险 |
|----|------|------|------|------|------|
| T-001 | DB | 设计 users/sessions 表 | 1h | - | 低 |
| T-002 | BE | 注册 API | 3h | T-001 | 低 |
| T-003 | BE | 登录 API + JWT | 3h | T-001 | 中 ⚠️ |
| T-004 | FE | 注册/登录表单 | 4h | - | 低 |
| T-005 | TEST | 单元测试 BE | 2h | T-002,T-003 | 低 |
| T-006 | TEST | E2E 测试登录流程 | 2h | T-004,T-005 | 低 |
| T-007 | DOC | 更新 API 文档 | 1h | T-005 | 低 |

⚠️ T-003 风险说明：JWT 刷新策略需要决策（短期 token vs 长期 token），建议先与团队确认方案。
```

## 与其他 Skill 的协作

- **sdd-riper**: 分解后的任务列表可直接作为 SDD-RIPER 的输入规范
- **spec-first**: 每个 Story 可触发 spec-first 流程生成详细规范
- **parallel-explore**: 并行任务可用 worktree 隔离开发
- **handoff**: 任务分解结果可作为交接文档的核心内容

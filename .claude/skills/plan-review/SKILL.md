# 计划审查 Skill (Plan Review)

> 基于 Aha-Loop 的计划审查模式，对实现计划进行系统化、多维度评估

## What (输入/输出)

### 输入

| 输入项 | 必需 | 描述 |
|--------|------|------|
| **实现计划文档** | 是 | PLAN.md 或类似格式的计划文档 |
| **项目愿景** | 否 | 项目的愿景文档或 README，用于验证对齐 |
| **技术约束** | 否 | 技术栈限制、性能要求、安全标准等 |
| **历史经验** | 否 | memory/lessons-learned.md 中的相关经验 |

### 输出

| 输出项 | 格式 | 描述 |
|--------|------|------|
| **审查报告** | PLAN-REVIEW.md | 完整的多维度评估报告 |
| **评分摘要** | JSON | 机器可读的评分数据 |
| **改进清单** | Markdown | 可操作的改进建议列表 |

---

## When Done (验收标准)

### 必须满足

- [ ] 生成完整的 PLAN-REVIEW.md 审查报告
- [ ] 10 个评审维度均有明确评分（1-10 分）
- [ ] 计算加权总分（满分 100）
- [ ] 每个评审维度都有具体说明
- [ ] 识别所有严重问题（红色）和中等问题（黄色）
- [ ] 提供可操作的改进建议

### 通过标准

| 总分 | 结论 | 后续行动 |
|------|------|----------|
| ≥80 | ✅ 通过 | 可直接进入开发阶段 |
| 70-79 | ⚠️ 有条件通过 | 修复黄色问题后可继续 |
| <70 | ❌ 需要修改 | 必须修复红色问题，重新审查 |

---

## What NOT (边界约束)

### 禁止行为

- ❌ **不修改原始计划** - 只提出建议，不直接编辑 PLAN.md
- ❌ **不执行计划任务** - 只评估计划质量，不做开发工作
- ❌ **不做技术决策** - 只评估现有决策的合理性，不替代架构师
- ❌ **不替代人工审批** - 提供审查报告，最终决策权在人

### 范围限制

- 评审仅针对计划文档本身，不深入代码实现细节
- 风险识别基于计划描述，不进行代码静态分析
- 不评估人员能力和资源分配（那是项目管理范畴）

## How (执行步骤)

### 阶段 1：结构检查（Structure Validation）

检查计划文档是否包含必要元素：

```markdown
必要元素清单:
□ 项目概述 - 背景、目标、范围
□ 技术方案 - 架构、数据模型、API 设计
□ 任务拆分 - 任务列表、依赖关系
□ 时间估算 - 工作量、里程碑
□ 验收标准 - 功能、性能、安全
□ 风险清单 - 已识别的风险和对策
```

**缺失必要元素将自动扣分**

### 阶段 2：深度评估（Deep Evaluation）

对 10 个维度进行打分（每维度 1-10 分）：

| 维度 | 权重 | 评审要点 | 评分标准 |
|------|------|----------|----------|
| **完整性** | 15% | 是否覆盖所有需求 | 10=全覆盖; 7=覆盖80%+; 4=有明显遗漏; 1=严重不完整 |
| **可行性** | 15% | 技术上是否可实现 | 10=完全可行; 7=有小风险; 4=有中等风险; 1=不可行 |
| **清晰度** | 10% | 描述是否清晰无歧义 | 10=非常清晰; 7=基本清晰; 4=部分模糊; 1=大量歧义 |
| **依赖性** | 10% | 依赖关系是否明确 | 10=完全明确; 7=大部分明确; 4=部分不清; 1=混乱 |
| **风险性** | 15% | 风险识别是否充分 | 10=全面识别; 7=覆盖主要; 4=遗漏重要; 1=无风险意识 |
| **测试性** | 10% | 是否可测试验证 | 10=可完全测试; 7=大部分可测; 4=部分难测; 1=难以测试 |
| **可维护性** | 10% | 长期维护成本 | 10=易维护; 7=一般; 4=维护成本高; 1=维护噩梦 |
| **扩展性** | 5% | 未来扩展空间 | 10=充分考虑; 7=有基本考虑; 4=考虑不足; 1=无扩展性 |
| **安全性** | 5% | 安全考虑是否充分 | 10=安全第一; 7=基本安全; 4=有漏洞; 1=安全隐患大 |
| **文档性** | 5% | 文档是否完整 | 10=完整详尽; 7=基本够用; 4=不够详细; 1=缺乏文档 |

### 阶段 3：综合建议（Synthesis）

1. **优点提炼** - 识别计划的亮点（3-5 条）
2. **问题分类** - 按严重程度分类（红/黄/绿）
3. **改进路径** - 提供具体可操作的改进建议
4. **下一步行动** - 明确审查后的行动计划

## 系统集成

### 与 Autopilot 集成

```
Autopilot Planning 阶段
    ↓
生成 PLAN.md
    ↓
自动调用 plan-review Skill ← 这里
    ↓
生成 PLAN-REVIEW.md
    ↓
[总分 ≥70?]
    ↓
是 → 进入 Specification 阶段
否 → 返回 Planning 阶段修改
```

### 与 spec-writer 协作

- **输入衔接**: spec-writer 生成的规范文档可作为 plan-review 的参考
- **输出传递**: plan-review 的问题清单可指导 spec-writer 完善规范

### 与 qa-reviewer 协作

- **基准传递**: PLAN-REVIEW.md 中的验收标准作为 QA 审查的基准
- **风险追踪**: 已识别的风险在 QA 阶段重点验证

### 与 God Committee 协作

```markdown
当满足以下条件时，需要 God Committee 审批：

1. 总分 < 70（需要重大修改）
2. 存在 3 个以上红色问题
3. 涉及安全或合规风险
4. 计划影响多个系统模块
5. 估算工期超过 2 周
```

## 使用示例

### 示例 1：简单功能计划审查

**场景**: 审查一个用户头像上传功能的实现计划

**输入**:
```markdown
PLAN.md - 用户头像上传功能计划
- 后端 API 设计
- 图片处理逻辑
- 存储方案（本地/OSS）
- 前端集成
```

**执行**:
```markdown
"请使用 plan-review Skill 审查 PLAN.md"
```

**输出摘要**:
```markdown
## 审查概要
- 总评分: 82/100
- 结论: ✅ 通过

## 主要发现
- ✅ 任务拆分清晰，依赖关系明确
- ⚠️ 缺少图片格式验证的安全考虑
- ⚠️ 未考虑大文件上传的超时处理

## 建议
- P1: 添加文件类型和大小验证
- P2: 考虑分片上传方案
```

### 示例 2：复杂系统计划审查

**场景**: 审查微服务架构重构的实现计划

**输入**:
```markdown
PLAN.md - 微服务架构重构计划
- 服务拆分方案（8 个微服务）
- 通信机制（REST/gRPC）
- 数据库拆分策略
- 部署方案（K8s）
- 迁移计划（3 阶段）
```

**执行**:
```markdown
"请使用 plan-review Skill 对 PLAN.md 进行全面审查，
  参考项目愿景 VISION.md 和技术约束 CONSTRAINTS.md"
```

**输出摘要**:
```markdown
## 审查概要
- 总评分: 68/100
- 结论: ❌ 需要修改

## 严重问题 (🔴)
1. 数据一致性方案缺失（分布式事务处理）
2. 服务间依赖形成环形依赖
3. 回滚方案不完整

## 中等问题 (🟡)
1. 性能测试计划不足
2. 监控和告警方案简略
3. 文档更新计划缺失

## 必须修复
1. 添加 Saga 模式处理分布式事务
2. 重新设计服务边界，消除环形依赖
3. 完善每阶段的回滚方案和验证点

## 下一步
需要返回 Planning 阶段修改后重新审查。
建议提交 God Committee 讨论架构设计。
```

## 最佳实践

### 提高审查质量

1. **提供充分上下文** - 附带项目愿景、技术约束等背景资料
2. **明确审查重点** - 如果有特别关注的维度，可以指定重点审查
3. **结合历史经验** - 引用 memory/lessons-learned.md 中的相关经验

### 审查后行动

1. **记录经验** - 重要的审查发现沉淀到 lessons-learned.md
2. **更新计划** - 根据审查报告修改 PLAN.md
3. **追踪问题** - 将识别的风险加入风险跟踪

### 避免的误区

- ❌ 期望审查能发现所有问题（审查基于文档，不能替代测试）
- ❌ 忽视低分维度（即使总分通过，低分维度也需要关注）
- ❌ 机械执行审查（需要结合具体项目情况灵活评估）

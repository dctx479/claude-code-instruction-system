# 横纵分析法 (Horizontal-Vertical Analysis) - Enhanced

## 契约定义

### What（输入/输出）

**输入**：
- 待分析的问题/决策/方案（必需）
- 已有的初步思路（可选）
- 分析深度要求：快速(2维度) / 标准(4维度) / 深度(6+维度)（可选，默认标准）
- 约束条件：时间、成本、技术栈等（可选）

**输出**：
- 横向分析矩阵：不同领域/视角的解读
- 纵向分析矩阵：不同抽象层级的解读
- 矛盾框架集：2-4 个相互冲突的观点/方案
- 盲点清单：用户可能忽略的角度
- 决策支持建议（但不给出"唯一答案"）

### When Done（验收标准）

- [ ] 生成完整的 HV-ANALYSIS.md 分析报告
- [ ] 至少提供 2 个相互矛盾的框架/观点
- [ ] 横向维度覆盖 ≥3 个不同领域视角
- [ ] 纵向维度覆盖 ≥3 个抽象层级
- [ ] 识别至少 3 个潜在盲点
- [ ] 明确说明"这不是唯一答案，请根据实际情况选择"

### What NOT（边界约束）

- **禁止给出单一"最优解"** - 这是延迟收敛的核心，必须提供多种选择
- **禁止隐藏矛盾** - 如果两个观点冲突，必须显式标出，不能试图"调和"
- **禁止替代用户决策** - 只提供分析框架，决策权在用户
- **禁止实施方案** - 只做分析，不写代码或生成配置
- **禁止忽略用户初步思路** - 如果用户已有想法，必须作为其中一个框架纳入分析

## 核心理念

### 延迟收敛 (Delayed Convergence)

**传统 AI 协作**：
```
用户问题 → AI 分析 → 给出"最优"答案 → 用户接受/调整
问题：AI 过早收敛到一个方案，用户失去探索空间
```

**横纵分析法**：
```
用户问题 → AI 生成多个矛盾框架 → 展示冲突点 → 用户选择/综合 → AI 协助实施
优势：用户在充分理解权衡后做决策，避免盲从
```

### 对抗性思维 (Adversarial Thinking)

不是寻找"一致性"，而是故意寻找"不一致性"：

- 如果一个框架认为"应该微服务化"，另一个框架必须论证"为什么单体更好"
- 如果一个视角强调"性能优先"，另一个视角必须论证"可维护性优先"
- 冲突不是 bug，冲突是特性 — 它帮助用户看到不同权衡

## How（执行步骤）

### 阶段 1：问题理解与框架构造

```
用户问题
    ↓
分析问题类型（技术选型/架构设计/需求澄清/方案评估）
    ↓
确定分析深度（快速/标准/深度）
    ↓
构造横纵分析矩阵
```

#### 横向维度（跨领域视角）

| 视角类型 | 关注点 | 适用场景 |
|----------|--------|----------|
| **技术视角** | 可行性、性能、技术债 | 技术选型、架构设计 |
| **业务视角** | 价值、ROI、用户体验 | 需求分析、产品决策 |
| **工程视角** | 可维护性、测试性、部署复杂度 | 实现方案、重构决策 |
| **安全视角** | 攻击面、合规性、数据保护 | 敏感系统、对外服务 |
| **成本视角** | 开发成本、运维成本、迁移成本 | 资源受限、商业决策 |
| **人员视角** | 团队技能、学习曲线、协作效率 | 技术栈选择、流程设计 |

#### 纵向维度（抽象层级）

| 层级 | 关注点 | 时间尺度 |
|------|--------|----------|
| **战略层** | 长期愿景、行业趋势、竞争格局 | 3-5 年 |
| **战术层** | 架构决策、技术路线、资源分配 | 6-12 月 |
| **操作层** | 具体实现、工具选择、代码设计 | 周/月 |

### 阶段 2：矛盾框架生成

**生成规则**：
1. 至少生成 2 个框架，最多 4 个（超过 4 个用户无法比较）
2. 框架间必须有明显冲突（不是"A 好"和"A 更好"，而是"A 好"vs"B 好"）
3. 每个框架都必须有合理的论据支撑
4. 明确标注框架的适用场景和限制

**矛盾框架模板**：
```markdown
### 框架 1: [名称]（如"激进微服务化"）

**核心观点**: [一句话主张]

**论据**:
1. [支持点 1]
2. [支持点 2]
3. [支持点 3]

**适用场景**:
- [场景 1]
- [场景 2]

**风险与代价**:
- [风险 1]
- [风险 2]

---

### 框架 2: [名称]（如"保守单体优先"）

**核心观点**: [与框架 1 冲突的主张]

**论据**:
1. [支持点 1]
2. [支持点 2]

**与框架 1 的冲突点**:
- 冲突维度 1: 框架 1 认为 X，框架 2 认为 Y
- 冲突维度 2: ...

**适用场景**:
- [场景 1]
- [场景 2]

**风险与代价**:
- [风险 1]
- [风险 2]
```

### 阶段 3：盲点识别

系统化检查用户可能忽略的角度：

**检查清单**：
```markdown
横向盲点:
□ 是否考虑了安全视角?
□ 是否评估了长期维护成本?
□ 是否分析了团队技能匹配度?
□ 是否考虑了合规和法律风险?

纵向盲点:
□ 是否从战略层思考（3 年后的技术趋势）?
□ 是否从操作层验证（实际开发的可行性）?

时间盲点:
□ 是否考虑了迁移期间的兼容性?
□ 是否评估了技术过时的风险?

人员盲点:
□ 是否考虑了团队的学习曲线?
□ 是否评估了离职风险（关键技能依赖单人）?
```

### 阶段 4：决策支持（不替代决策）

输出格式：
```markdown
## 决策支持建议

我提供了 [N] 个不同的框架，它们在以下维度上存在冲突：
- 冲突点 1: [描述]
- 冲突点 2: [描述]

**如何选择**：
- 如果你的场景是 [场景 A]，倾向于框架 [X]
- 如果你的场景是 [场景 B]，倾向于框架 [Y]
- 如果你不确定，可以考虑先做 [小范围实验]

**我不建议**：
- 不建议试图"综合所有框架" — 它们本质上是矛盾的，强行融合会失去各自优势
- 不建议选择"看起来最安全的"方案 — 安全往往意味着保守，可能错失机会

**下一步行动**：
1. [ ] 明确你的核心约束（时间/成本/团队技能）
2. [ ] 选择一个框架作为主导方向
3. [ ] 用其他框架的视角检查选定方案的盲点
```

## 使用示例

### 示例 1: 技术选型决策

**用户问题**:
> "我们的单体应用用户量增长很快，要不要拆成微服务？"

**HV 分析输出摘要**:

```markdown
## 矛盾框架集

### 框架 1: 激进微服务化

**核心观点**: 应该尽快拆分，否则后续成本更高

**论据**:
1. 用户增长快，单体扩展性差
2. 团队规模扩大，单体协作效率低
3. 微服务可独立部署，降低发布风险

**适用场景**:
- 团队 >20 人
- 需要频繁发布（周级）
- 有 DevOps 基础设施

**风险与代价**:
- 运维复杂度 10x 增长
- 分布式事务难处理
- 需要 6-12 月迁移期

---

### 框架 2: 保守单体优先

**核心观点**: 单体优化 + 模块化，不急于拆分

**论据**:
1. 微服务成本高（需要服务发现、网关、监控）
2. 团队经验不足，容易拆错边界
3. 单体优化空间大（缓存、异步、读写分离）

**与框架 1 的冲突点**:
- **扩展性**: 框架 1 认为必须拆，框架 2 认为单体优化够用
- **团队协作**: 框架 1 认为微服务提升效率，框架 2 认为增加复杂度

**适用场景**:
- 团队 <15 人
- 业务边界不清晰
- 运维能力不足

**风险与代价**:
- 技术债累积
- 后续拆分成本更高
- 单点故障风险

---

### 框架 3: 混合架构（中间路线）

**核心观点**: 核心功能模块化，非核心保持单体

**论据**:
1. 只拆分高并发/独立演化的模块
2. 降低微服务管理成本
3. 渐进式迁移，风险可控

**与框架 1/2 的关系**:
- 不如框架 1 彻底，但比框架 2 有扩展性
- 需要清晰的模块边界设计

**适用场景**:
- 部分功能有明确瓶颈
- 团队有一定微服务经验
- 可以分阶段实施

---

## 盲点识别

1. **时间盲点**: 是否考虑了迁移期间的双写成本？
2. **人员盲点**: 团队是否有人熟悉 Kubernetes/服务网格？
3. **业务盲点**: 业务是否允许 6-12 月的技术重构期？

---

## 决策支持建议

我提供了 3 个框架，核心冲突在：**现在拆 vs 优化后拆 vs 部分拆**

### 选择指引

- 如果团队 >20 人 + 有 DevOps 基础 + 业务允许停止新功能 6 月 → 倾向框架 1
- 如果团队 <15 人 + 运维能力弱 + 业务需求紧 → 倾向框架 2
- 如果有明确瓶颈（如支付/搜索）+ 团队有一定经验 → 倾向框架 3

### 我不建议

- ❌ 不建议盲目跟风微服务（大部分公司不需要 Netflix 规模的架构）
- ❌ 不建议"先拆了再说" — 拆错了回滚成本极高

### 下一步

1. [ ] 评估当前单体的性能瓶颈（是否真的卡在架构）
2. [ ] 评估团队 DevOps 能力（是否能驾驭微服务）
3. [ ] 选择一个框架试验（例如用框架 3 拆一个模块验证）
```

### 示例 2: 需求澄清场景

**用户问题**:
> "老板让我做一个数据分析平台，不知道从哪开始"

**HV 分析输出摘要**:

```markdown
## 矛盾框架集

### 框架 1: MVP 先行（最小可用产品）

**核心观点**: 快速上线一个能用的版本，迭代优化

**论据**:
1. 需求不明确，做太多会浪费
2. 用户用了才知道要什么
3. 2 周上线，边用边改

**适用场景**: 内部工具、试验性项目

---

### 框架 2: 完整规划（一步到位）

**核心观点**: 先做充分调研，设计完整架构

**论据**:
1. 数据平台涉及多系统集成，架构错了难改
2. 性能和安全需要前期考虑
3. 避免后续推倒重来

**适用场景**: 对外服务、核心系统

**与框架 1 的冲突**: MVP 快但可能技术债重，完整规划慢但更稳健

---

## 盲点识别

1. **需求盲点**: 老板说"数据分析"，具体是 BI 报表/实时监控/机器学习？
2. **人员盲点**: 团队有数据工程师吗？还是只有后端开发？

---

## 决策支持

如果需求不明确 → 先用框架 1 做 MVP，验证需求后再补架构
如果需求明确 + 涉及多部门 → 框架 2，前期投入时间做规划
```

## 系统集成

### 与其他 Skills 的协作

| Skill | 协作方式 | 典型场景 |
|-------|----------|----------|
| **vision-builder** | HV 分析前先用 vision-builder 澄清需求 | 用户问题模糊时 |
| **plan-review** | HV 分析后，用 plan-review 评估选定方案 | 决策后验证可行性 |
| **architecture-copilot** | HV 分析确定方向后，用 architecture-copilot 深入设计 | 技术选型确定后 |
| **spec-writer** | 框架选定后，用 spec-writer 编写规范 | 进入实施阶段 |

### 典型工作流

```
用户模糊问题
    ↓
[1] vision-builder（如果需求极度模糊）
    ↓
[2] hv-analysis-enhanced（生成矛盾框架）
    ↓
[3] 用户选择框架
    ↓
[4] plan-review（评估选定方案）
    ↓
[5] architecture-copilot / spec-writer（实施）
```

## 质量检查清单

生成分析报告后自动检查：

- [ ] 是否生成了 ≥2 个矛盾框架
- [ ] 框架间是否有明确的冲突点标注
- [ ] 是否识别了 ≥3 个潜在盲点
- [ ] 是否明确说明"不是唯一答案"
- [ ] 是否避免了给出"最优解"的表述
- [ ] 横向维度是否覆盖 ≥3 个领域视角
- [ ] 纵向维度是否覆盖 ≥3 个抽象层级

## 最佳实践

### 何时使用此 Skill

**适合场景**:
- 技术选型（选框架、选架构模式、选数据库）
- 需求澄清（用户需求模糊，需要探索可能性）
- 方案评估（已有初步方案，想看看有无盲点）
- 架构决策（重构 vs 优化、单体 vs 微服务）

**不适合场景**:
- 明确的 bug 修复（直接用 debugger Agent）
- 执行性任务（已经决策完毕，只需实施）
- 纯技术问题（如"这段代码怎么写"）

### 提高分析质量的技巧

1. **提供充分上下文** - 团队情况、技术栈、业务约束
2. **明确分析重点** - 如果特别关注某个维度（如成本），可以指定
3. **接受矛盾** - 不要期望分析给出"完美方案"，矛盾本身就是价值

### 避免的误区

- ❌ 期望 HV 分析给出"正确答案" — 它只提供视角，不替代决策
- ❌ 忽视盲点清单 — 盲点往往是决策失败的根源
- ❌ 试图综合所有框架 — 矛盾的框架无法简单融合

---

## 注意事项

1. **延迟收敛是核心** - 不要过早收敛到单一方案，给用户充分的选择空间
2. **矛盾是特性不是 bug** - 冲突的框架帮助用户看到权衡
3. **不替代用户决策** - 提供分析框架，但决策权始终在用户手中
4. **结合其他 Skills** - HV 分析是决策前的探索工具，决策后需要其他 Skills 执行

## Agent 编排最佳实践

### BP-001: 任务分解原则

**实践描述**:
复杂任务分解时遵循 MECE 原则 (相互独立、完全穷尽)

**适用场景**:
所有需要分解的复杂任务

**具体做法**:
```
1. 识别任务的主要组成部分
2. 确保各部分相互独立 (无重叠)
3. 确保所有部分覆盖完整任务 (无遗漏)
4. 评估各部分的依赖关系
5. 确定可并行的部分
```

**效果**:
- 减少任务冲突
- 提高并行度
- 更清晰的进度跟踪

### BP-002: 策略选择决策树

**实践描述**:
使用决策树快速选择最优编排策略

**决策流程**:
```
任务是否可分解?
├── 否 → 单Agent执行
└── 是 → 子任务间有依赖?
         ├── 完全独立 → PARALLEL
         ├── 链式依赖 → SEQUENTIAL
         └── 部分依赖 → 需要专家决策?
                        ├── 是 → HIERARCHICAL
                        └── 否 → 需要多视角?
                                 ├── 是 → COLLABORATIVE
                                 └── 否 → PARALLEL (分组)
```

### BP-004: 并行任务优化

**实践描述**:
最大化并行效率同时避免资源冲突

**具体做法**:
```
1. 分析任务资源依赖
2. 将任务分组:
   - 独立任务 → 立即并行
   - 冲突任务 → 串行队列
3. 设置合理的并行度 (通常3-5个)
4. 监控执行进度
5. 动态调整并行度
```

**注意事项**:
- 过高并行度可能导致资源竞争
- 文件修改任务需特别注意冲突

### BP-006: 错误恢复策略

**实践描述**:
优雅处理Agent失败

**恢复流程**:
```
Agent失败
├── 可重试错误 (超时、临时错误)
│   └── 重试 (最多3次，指数退避)
├── 可降级错误 (复杂度过高)
│   └── 分解任务或使用更强模型
└── 不可恢复错误
    ├── 保存已完成的工作
    ├── 记录错误详情
    └── 通知用户
```

### BP-009: Skill UX 设计原则

**来源**: x-ai-topic-selector v1.0→v1.4 迭代经验

**核心原则**:

1. **并行输入 > 串行确认**
   - ❌ 5 个参数逐步确认，每次执行点 5 次
   - ✅ 一屏显示所有参数，一次选完再执行
   - 效果: 操作从 5 步缩到 1-2 步，配置自动保存供下次复用

2. **根据用户心智模型分离处理流**
   - 书签 = 已人工筛选过的内容 → 适合"分析 + 建议"流程
   - 信息流 = 海量原始内容 → 适合"过滤 + 排序"流程
   - 将两者混用同一逻辑是错误的；识别数据的心智模型，设计对应处理流

3. **输出质量与功能同等重要**
   - 功能跑通只是起点，报告不好读 = 用着别扭
   - 每次输出加"今日看点"摘要，一眼判断值不值得深读
   - 可视化（词云、关键词统计）让信号更清晰

4. **保存用户偏好，减少重复操作**
   - 自动记录上次配置，下次选"使用上次配置"一秒启动
   - 需要调整时展示上次值，方便对比修改

**迭代方法论**: 先跑通 → 实际使用 → 发现不爽 → 修复。不要坐着想"我应该加什么功能"，从自己用着的痛点出发才有价值。

### BP-011: 多模型任务分工原则

**来源**: AI 科研自动化实践经验

**核心洞察**: 没有万能模型，不同认知任务有最优模型分配：

| 任务类型 | 推荐模型 | 原因 |
|----------|---------|------|
| 论文写作/长文生成 | claude-opus-4-6 | 大上下文窗口，表达流畅 |
| 逻辑推理/数学 | claude-sonnet-4-6 | 推理能力强，性价比高 |
| 创意发散/Idea 生成 | grok 系列 | 发散思维强，出奇制胜 |
| 代码实现 | GPT 系列 / claude-sonnet | 代码基准分高 |
| 文献检索引用 | API 查询（非生成） | AI 幻觉问题，必须用真实数据源 |

**反模式（避免）**:
- ❌ 让单一模型包揽所有任务（质量下降 + 成本暴增）
- ❌ 用 LLM 自由生成文献引用（幻觉导致学术不端风险）
- ❌ 用低端模型做顶刊级创新（创新度不足）

**成本控制**: AI 科研全流程非常耗 token，单日千美元量级可能。建议：
- 先用小模型草稿，再用大模型精修
- 文献工作用 Zotero MCP 等工具替代生成
- 科研工作流需要多人协作，单人 ROI 有限

### BP-012: Anthropic 官方 Skills 设计最佳实践

**来源**: Anthropic Thariq (Claude Code 团队) 2026-03 公开分享，Anthropic 内部数百个 Skill 实践总结

**Skills 九大分类体系**:

| 分类 | 说明 | 示例 |
|------|------|------|
| **Library & API Reference** | 教模型正确使用库/CLI/SDK | billing-lib, internal-platform-cli |
| **Product Verification** | 测试/验证代码是否正常工作 | signup-flow-driver, checkout-verifier |
| **Data Fetching & Analysis** | 连接数据和监控系统 | funnel-query, grafana |
| **Business Process & Team Automation** | 重复性工作流自动化 | standup-post, weekly-recap |
| **Code Scaffolding & Templates** | 生成框架模板代码 | new-migration, create-app |
| **Code Quality & Review** | 推行代码质量标准 | adversarial-review, testing-practices |
| **CI/CD & Deployment** | 代码推送和部署 | babysit-pr, deploy-service, cherry-pick-prod |
| **Runbooks** | 多工具排查流程 | service-debugging, oncall-runner |
| **Infrastructure Operations** | 日常运维 | resource-orphans, cost-investigation |

**当前项目覆盖情况**:
- ✅ 已覆盖: Library/API (pytorch, pandas), Data Fetching (data-analysis, amazon-analyse), Code Quality (QA 系统)
- ⚠️ 部分覆盖: Business Automation (有 commit/standup 但不完善), Code Scaffolding (parallel-explore)
- ❌ 未覆盖: Product Verification, CI/CD & Deployment, Runbooks, Infrastructure Operations

**10 条 Skill 设计原则**:

1. **只写 Claude 不知道的东西** — Claude 对代码库已有了解，聚焦于能让它跳出常规思维的信息。示例: frontend-design Skill 专门避免"AI 味"（Inter 字体 + 紫色渐变）

2. **认真写 Gotchas 部分** — 任何 Skill 中信噪比最高的内容。从 Claude 反复踩的坑中积累，随时间持续更新

3. **文件系统 = 渐进式上下文** — Skill 是文件夹不只是 Markdown。用 `references/api.md`、`assets/template.md`、`scripts/` 实现按需加载

4. **给出信息但留出灵活度** — Skills 高度可复用，别把指令写得太死。给 Claude 需要的信息，也给它灵活调整的空间

5. **用 config.json 做初始化设置** — 需要用户上下文时，存到 Skill 目录下的 config.json。未配置时用 AskUserQuestion 主动问

6. **Description 是触发器不是摘要** — Claude 靠 description 判断"这个请求有没有对应的 Skill"。写法要像触发条件

7. **让 Skill 拥有记忆** — 用追加写入的日志文件或 JSON。`${CLAUDE_PLUGIN_DATA}` 是稳定数据目录，Skill 目录可能被升级覆盖

8. **给 Claude 可直接用的代码** — 提供脚本和函数库，让 Claude 花精力在组合和决策上而非从零重写

9. **善用 Skill 级别 Hooks** — Skills 可包含只在该 Skill 调用时激活的 hooks。适合放"平时不想开但有时很有用"的强约束（如 `/careful` 拦截危险命令、`/freeze` 锁定编辑范围）

10. **分享策略** — 小团队直接提交到 `.claude/skills/`；规模扩大用 plugin 市场分发。过多 Skills 增加上下文负担

**验证方法**: 对照现有 Skill 逐条审查，识别改进点（如缺少 Gotchas、Description 不够触发式等）

**案例引用**: Anthropic 内部数百个 Skill 实践，frontend-design Skill 消除 AI 味的迭代

### BP-013: 并行研究架构中的压缩与隔离权衡

**来源**: ODR (Open Deep Research) 架构分析，与 deep-research Skill 的 Lead Agent + Subagent 模式直接相关

**核心架构模式**:
```
Supervisor (Plan-and-Execute)
  ├─ think_tool → 反思/规划
  ├─ ConductResearch → 派发任务（×N 并行）
  └─ ResearchComplete → 判断信息足够
        ↓
Researcher (ReAct 循环)
  ├─ search → 搜索
  ├─ think_tool → 搜后反思
  └─ ResearchComplete → 信息足够退出
        ↓
compress_research → 保真去重，交回摘要
```

**关键设计洞察**:

1. **两层用不同模式**: Supervisor 用 Plan-and-Execute（可控性），Researcher 用 ReAct（灵活性）。搜索本质是探索性的，ReAct 适配
2. **"空工具"塑造角色**: think_tool 和 ResearchComplete 都是空壳（输入原样返回/终止信号），但定义了"想-做-停"的完整行为空间
3. **两个出口并存**: 模型信号（ResearchComplete）是正常出口，资源上限（max_tool_calls）是安全网
4. **搜-想节奏**: 每次搜索后必须 think_tool 反思（"搜到什么？还缺什么？够了吗？"），不允许连续搜索

**压缩的两层机制**:
- 第一层: 原始网页 → 预处理摘要（~25% 体量），保留 who/what/when/where/why
- 第二层: Researcher 工作历史 → compress_research 结构化摘要（保真去重，非总结）

**并行隔离的收益与代价**:

| 收益 | 代价 |
|------|------|
| 速度: 耗时≈最慢的单条线索 | 跨线索关联在压缩时不可见 |
| 聚焦: 每个 Researcher 上下文干净 | 做判断的模型不知道下游需要什么 |
| 容错: 单条线质量波动不传染 | 压缩是有损的 |
| 上下文效率: 最终能一次综合 | 关联细节可能被丢弃 |

**缓解机制**: 压缩的保真原则、Supervisor 多轮反思、Synthesizer 全局视野、prompt 显式告知隔离存在。每一层都依赖模型判断，是概率性缓解而非结构性保证。

**应用到 deep-research Skill**: 当研究问题本质是因果链而非独立线索时，需要格外关注 Subagent 间的信息传递完整性

**案例引用**: Anthropic Open Deep Research 开源项目架构

### BP-014: 极简 Skill 设计哲学

**来源**: liangdabiao simple-review-analyzer 项目，LinuxDo 开源社区

**核心理念**: "以至简达至繁" — 纯 Markdown 指令，不写 Python，不教 AI 做事，让 AI 按指导原则自主解决

**两种 Skill 设计风格对比**:

| 维度 | 结构化风格（当前项目） | 极简风格 |
|------|----------------------|---------|
| 实现方式 | 详细步骤 + 模板 + 脚本 | 纯 Markdown 指导原则 |
| 控制力 | 强（输出格式精确可控） | 弱（依赖 AI 自主判断） |
| 可维护性 | 需要维护脚本和模板 | 几乎零维护 |
| 一致性 | 高（模板保证） | 中（AI 每次可能不同） |
| 灵活性 | 低（模板限制） | 高（AI 自适应） |
| 适合场景 | 生产级、需要稳定输出 | 个人使用、快速迭代 |

**实践建议**: 不必二选一。核心生产 Skill 用结构化风格保证稳定输出；探索性/个人 Skill 可用极简风格快速验证。关键在于明确输入/输出契约和验收标准。

**案例引用**: simple-review-analyzer（GitHub: liangdabiao），buluslan/review-analyzer-skill（22 维度分析的结构化对照）

### BP-015: Context Hygiene — 上下文卫生实践

**来源**: Token 优化文章 + Everything Claude Code 上下文腐化分析 + CLAUDE.md 超限事件（lessons-learned #011）

**核心理念**: 上下文工程的核心指标是信噪比 (SNR)，而非总信息量。

**5 习惯**:

1. **精准加载，按需注入** — INDEX → 按需 SKILL.md，不一次性加载所有内容
   - Before: 启动时加载 50K tokens 的全部 Agent 定义
   - After: 加载 600 tokens INDEX，按需加载 3K tokens 单个 Agent

2. **压缩终端输出** — 工具调用后只保留对下一步有用的信息
   - Before: 500 行 npm test 输出全部注入上下文
   - After: "3 tests failed: auth.test.ts:45, user.test.ts:78, api.test.ts:112"

3. **单一事实源** — 每个概念只在一处权威定义，其他地方引用
   - Before: CLAUDE.md 和 docs/ 各写一份完整规则 → 修改时漏改一处 → 矛盾
   - After: CLAUDE.md 写 2-3 行摘要 + 指向 docs/X.md 的链接

4. **定期审计固定上下文税** — 每月审计 CLAUDE.md，目标 ≤39,500 chars
   - 触发条件: CLAUDE.md 接近 39K 或每月例行

5. **避免内容重复** — 重复 = 增大固定税 + 矛盾风险
   - 具体案例: lessons-learned #011 中 Bash 规则 section 嵌入已有 StatusLine 内容导致超限

**两大失效模式**:
- **上下文投毒**: 过时信息残留 → 产出偏差（检测: AI 引用废弃规则）
- **注意力漂移**: 批量任务 >30 项后遗忘约束（检测: 格式漂移/跳步）

**验证方法**: `wc -m CLAUDE.md` < 40000; 新增内容前检查是否已有重复

**案例引用**: lessons-learned.md #011（CLAUDE.md 超限）、#015（批量任务上下文丢失）、`docs/CONTEXT-ENGINEERING-GUIDE.md`

### BP-016: Phase-Gated Execution — 阶段门禁执行

**来源**: Superpowers 方法论 + SDD 阶段门禁规则 + 127 只股票批量分析中的格式漂移经验

**核心理念**: 每个阶段结束时都有可验证的验收标准，禁止"跳过规划直接实现"（除非小任务）。

**阶段序列**:

```
brainstorm → spec → plan → execute → review → finish
  (发散)    (定义)  (规划)  (实现)   (验证)   (收尾)
```

**任务规模-流程深度匹配表**:

| 规模 | 预估工时 | 流程 | 可跳过 |
|------|---------|------|--------|
| **小** | ≤1h | brainstorm → execute → finish | spec, plan, review |
| **中** | 1-4h | brainstorm → spec → execute → review → finish | 独立 plan |
| **大** | >4h | 全流程（不可跳过任何阶段） | 无 |

**每步验收标准示例**:

| 阶段 | Before（不合格） | After（合格） |
|------|-----------------|--------------|
| brainstorm | "可以用几种方案做" | "方案 A: JWT+Redis, 方案 B: Session+DB, 推荐 A 因为..." |
| spec | "实现用户认证" | SPEC-user-auth.md 包含 API 列表、数据模型、验收标准 |
| plan | "先做后端再做前端" | 任务拆分为 12 个子任务，有依赖图和时间估算 |
| execute | "代码写完了" | 所有测试通过 + lint 无错误 + 功能演示通过 |
| review | "看了一下没问题" | QA 评分 ≥80，P0/P1 问题清零 |

**禁止行为**:
- ❌ 大任务不经 spec 直接实现
- ❌ 跳过 review 直接发布
- ❌ brainstorm 只列方案不给推荐和理由

**验证方法**: 编排器在任务开始时评估规模 → 自动匹配流程深度 → 在每个门禁检查验收标准

**案例引用**: Superpowers 方法论、spec-writer Agent 的阶段门禁规则、批量股票分析中的格式漂移问题

### BP-017: AI 执行安全反模式

**来源**: Vibe Skills 项目实践 + 社区踩坑反馈

**核心洞察**: AI 在执行中容易陷入两类高风险行为，需通过硬性约束（guardrails）而非柔性建议来防治。

**反模式 1: 批量删除**

```
❌ AI 执行 rm *.log / rm -rf temp/ — 批量删除文件
✅ 只允许逐个文件删除，每次删除前确认文件内容
```

**根因**: AI 对文件系统的影响力不对称 — 一次批量删除可瞬间销毁大量数据，而恢复成本极高。AI 在理解"哪些文件可以删"上容易出错（误判临时文件、忽略备份依赖）。

**防治规则**:
- 禁止 `rm *`、`rm -rf`、`find -delete` 等批量删除命令
- 删除前必须逐个确认文件路径和内容摘要
- 特别警惕"清理临时文件"类任务中的误删

**反模式 2: 静默兜底机制**

```
❌ try: result = main_logic() except: result = default_value  # 静默吞掉错误
✅ try: result = main_logic() except Exception as e:
       logger.error(f"⚠️ 兜底机制触发: {e}")  # 必须显式警告
       result = default_value
```

**根因**: AI 倾向于写"保险"的兜底逻辑来确保程序不崩溃，但静默兜底会掩盖功能实现度不足的问题。AI 可能自信地报告"功能已完成"，实际上全是兜底在生效，核心逻辑未真正工作。

**防治规则**:
- 所有兜底机制必须包含 `⚠️ WARNING` 级别的日志输出
- 代码审查时重点检查 catch-all 异常处理是否隐藏了真实错误
- 验收测试必须验证主逻辑路径是否真正执行（而非只走 fallback）

**验证方法**: 对 AI 生成的代码做 grep 检查 — `grep -r "except:" --include="*.py"` 和 `grep -r "catch (" --include="*.ts"` 查找静默异常处理

**案例引用**: Vibe Skills 项目社区反馈 — "AI 喜欢写静默的兜底机制，然后早早的自信满满的给你说做好了，实际上全是兜底机制在发力，主要功能实现度很差"

### BP-018: 子代理并行时的 Skills 加载机制

**来源**: Vibe Skills 项目多代理并发编排经验 + 太一元系统编排指南缺失补充

**核心问题**: 当多个子代理并行执行时，每个子代理是否能正确加载所需的 Skills？当前编排指南只描述了 Agent 层级架构和渐进式披露，但完全没有子代理级别的 Skills 加载规范。

**问题场景**:

```
Orchestrator 分解任务为 3 个并行子任务:
├─ Worker-1: 前端开发 → 需要 frontend-design Skill
├─ Worker-2: 数据分析 → 需要 data-analysis Skill
└─ Worker-3: 安全审计 → 需要 security-audit Skill

如果每个 Worker 独立上下文:
- Worker-1 是否会自动读 INDEX.md 并匹配 frontend-design？
- 还是只会盲目执行，丢失专业指导？
```

**规范设计**:

| 层级 | Skills 加载策略 | 说明 |
|------|----------------|------|
| **Orchestrator** | 全量 INDEX.md | 需要全局视野来分配任务和 Skills |
| **Specialist** | 按任务匹配 INDEX + 加载对应 SKILL.md | 在任务分配时由 Orchestrator 指定所需 Skills |
| **Worker** | 继承 Specialist 指定的 SKILL.md | 只加载与当前任务直接相关的 Skills |

**编排器职责增强**:

1. **任务分解时**: 在子代理 prompt 中明确指定需要加载的 Skills（如 "你需要先读取 `.claude/skills/frontend-design/SKILL.md`"）
2. **Skills 传递**: 子代理 prompt 应包含 Skills 的关键约束摘要（即使子代理未读 SKILL.md，也能遵循核心规则）
3. **负向路由**: 明确告知子代理**不需要**加载的 Skills（减少上下文浪费）

**具体做法**:

```
# Orchestrator 分发任务时的 prompt 模板
你负责 {task_description}。
执行前，请先读取以下 Skills:
1. {skill_path_1} — 原因: {why_needed}
2. {skill_path_2} — 原因: {why_needed}

你不需要读取: {excluded_skills}（与当前任务无关）
```

**与上下文工程的协同**: 子代理天然获得独立上下文窗口（见 CONTEXT-ENGINEERING-GUIDE.md 会话管理框架），Skills 加载在独立上下文中进行，不污染父上下文。

**验证方法**: 在下次执行 PARALLEL/HIERARCHICAL 编排时，检查子代理 prompt 是否包含 Skills 指引

**案例引用**: Vibe Skills 项目 "在并行分发任务后，子代理能并行拉起各自任务的skills，实现多代理并发时，skills也会被正确拉起"

### BP-019: 长上下文下 Skills 路由失效防护

**来源**: Vibe Skills 项目 4.15 补丁更新 + 太一元系统上下文衰减分析

**核心问题**: 当上下文超过 300-400k tokens（见 CONTEXT-ENGINEERING-GUIDE.md 衰减阈值），Skills 路由匹配的约束力下降 — AI 可能不再按照 Skills 的指导工作，而是退化到"默认行为"。

**失效模式**:

```
正常: 用户请求 → INDEX.md 匹配 → 加载 SKILL.md → 按指导执行
失效: 用户请求 → AI 忽略 INDEX.md → 不加载 Skill → 自由发挥（质量下降）
```

**根因**: Skills 路由依赖 AI 在长上下文中仍然"注意到"路由规则的存在。上下文衰减导致 AI 注意力从系统约束转移到对话内容，路由规则被"挤出"注意力窗口。

**防护策略（三层防御）**:

| 层级 | 策略 | 实现 |
|------|------|------|
| **L1: 主动压缩** | 在 ~250k tokens 时 `/compact` 并附上未来方向 | 见会话管理决策框架 |
| **L2: 路由强化** | 在关键约束中重申 Skills 路由规则 | 关键约束写入 CLAUDE.md（固定上下文） |
| **L3: 执行验证** | 任务完成后检查是否调用了正确的 Skills | 在 review 阶段验证 |

**具体防护规则**:

1. **/compact 指令**: `/compact 保持 Skills 路由规则，保留待办任务列表和当前阶段`
2. **CLAUDE.md 中的路由强化**: 在固定上下文税中保留简短的路由提醒（如 "执行任务前先检查 INDEX.md 匹配 Skill"）
3. **Subagent 隔离**: 大量中间输出的工作交给 Subagent，避免撑大主上下文导致路由失效

**与现有机制的协同**:
- BP-015 Context Hygiene 的习惯 4（审计固定上下文税）— 确保路由提醒在固定税中
- BP-016 Phase-Gated 的 review 阶段 — 增加 Skills 调用验证
- CONTEXT-ENGINEERING-GUIDE.md 的上下文衰减防治 — 量化阈值 + 主动压缩

**验证方法**: 在长会话（>250k tokens）后检查 AI 是否仍然正确匹配和调用 Skills

**案例引用**: Vibe Skills 项目 "防止在超长上下文下，vibe的调用路由因为约束不足而失效的问题"

### BP-020: 文件目录语义治理规范

**来源**: Vibe Skills 项目文件语义治理实践 + 太一元系统目录约定散乱现状

**核心问题**: 当前目录结构约定散见于各文件，缺乏统一的语义命名规范。新对话的 AI 需要重新理解"什么目录存什么"，容易遗漏细节导致工作衔接问题。

**Vibe Skills 的四层语义模型**:

| 目录 | 语义层 | 存什么 | 不存什么 |
|------|--------|--------|---------|
| `docs/` | 解释层 | 说明文档、指南、架构设计 | 运行时产物、临时文件 |
| `config/` | 合同层 | 配置文件、路由规则、Agent 定义 | 脚本、代码 |
| `scripts/` | 执行层 | 可执行脚本、工具、自动化 | 文档、配置 |
| `references/` | 长期资产层 | 模板、参考数据、知识库 | 运行时状态 |
| `outputs/runtime/` | 运行证据层 | 日志、执行记录、临时输出 | 源代码、配置 |

**太一元系统现有目录对照**:

| 语义层 | Vibe 目录 | 太一现有目录 | 差距 |
|--------|----------|-------------|------|
| 解释层 | `docs/` | `docs/` | ✅ 一致 |
| 合同层 | `config/` | `config/` + `.claude/settings.json` | ⚠️ Agent 定义在 `agents/`，分散 |
| 执行层 | `scripts/` | `hooks/` + `scripts/` | ✅ 基本一致 |
| 长期资产层 | `references/` | `memory/` + `.claude/skills/` | ⚠️ memory 含运行时状态 |
| 运行证据层 | `outputs/runtime/` | 无统一目录 | ❌ 散落各处 |

**建议改进（低优先级，避免破坏现有结构）**:

1. **运行时产物与长期资产分离**: Vibe 项目的做法 — 工作区运行时文件放到 `.vibeskills/`（或 `.taiyi/runtime/`），和核心项目根目录文档分离
2. **新增目录时的语义审批**: 确认新目录属于哪一语义层，避免层间混用
3. **AI 接手工作区时的快速理解**: 在项目根目录的 README 或 CLAUDE.md 中用 5 行说明目录语义

**关键原则**: 文件目录语义治理的核心目标是让**新的 AI 对话**能快速理解工作区结构，减少"重新理解"的成本和遗漏。

**案例引用**: Vibe Skills 项目 "按固定化的架构存储文件，让下一个新的对话的AI明白什么什么目录下存储什么什么文件" + "修复工作区记忆和核心区的文件杂糅的问题"

### BP-021: Agent 协作协议与任务地图

**来源**: Hermes SOUL.md 协作协议思想 + claude-init 三层文档架构 + 太一元系统 Auto-Dispatch/编排实践

**核心问题**: 复杂任务中，AI 如果只按单轮指令执行，容易出现三类问题：任务边界不清、过早顺从错误方向、完成后没有责任闭环。仅靠零散提示词无法稳定约束这些行为。

**协作协议五要素**:

| 要素 | 作用 | 建议落点 |
|------|------|----------|
| **身份定义** | 明确当前 Agent 的职责、专长和不可做事项 | `agents/*.md` frontmatter + 角色说明 |
| **任务地图** | 把目标拆成当前阶段、下一步、阻塞点和完成标准 | Task 工具 / plan / handoff |
| **反驳规则** | 当用户方案有明显风险或与目标冲突时，必须指出并给替代方案 | `CLAUDE.md` 或 Agent 行为准则 |
| **授权边界** | 区分可自主执行、需确认、绝对禁止的动作 | `CLAUDE.md` 自主决策授权 |
| **责任闭环** | 完成后验证、同步知识、说明未验证项 | `/neat`、QA、git diff 检查 |

**具体做法**:

1. **任务开始**: 先识别当前 Agent/Skill/阶段，不把所有规则一次性塞进上下文。
2. **任务推进**: 对复杂任务维护任务地图，标出已完成、进行中、阻塞和验收标准。
3. **遇到风险**: 不盲从用户方案；用“风险 → 证据 → 替代方案”的格式反驳。
4. **任务结束**: 执行验证和收尾同步，避免“做完代码但文档/记忆/索引失同步”。

**反模式（避免）**:

- ❌ 把 SOUL/角色协议写成大段人格化宣言，增加固定上下文税但不提供可执行约束。
- ❌ 每个 Agent 复制一份相同协作规则，导致多处维护和上下文投毒。
- ❌ 用强压式或 PUA 式话术驱动 AI；应抽象为阶段门禁、失败重试和验收标准。

**验证方法**: 对一次复杂任务复盘，检查是否存在明确任务地图、是否记录风险反驳、是否完成验证与 `/neat` 收尾。

**案例引用**: 用户提供的 Hermes `SOUL.md` 协作协议、PUA/superpowers 阶段门禁材料、claude-init 分层文档架构。

### BP-022: 第三方 Skill/Agent 引入前审查

**来源**: find-skills / skill-creator / skill-vetter / dokobot / AIHOT / smart-search 等第三方 Skill 生态材料 + 太一元系统安全边界

**核心问题**: 第三方 Skill 和 Agent 往往同时具备读取文件、联网、执行 shell、调用外部 API 的能力。未经审查直接安装会放大供应链风险、密钥泄露风险和上下文投毒风险。

**审查矩阵**:

| 维度 | 检查问题 | 风险信号 |
|------|----------|----------|
| **来源可信度** | 作者、仓库、更新记录是否可信 | 匿名来源、无说明、近期异常变更 |
| **工具权限** | 是否要求 shell、网络、文件写入、浏览器控制 | 权限远超功能所需 |
| **密钥处理** | 是否读取 `.env`、token、浏览器 cookie | 上传密钥、记录完整 token |
| **外部通信** | 会向哪些域名/API 发送数据 | 未声明 endpoint 或上传本地文件 |
| **提示词安全** | 是否包含越狱、绕过审查、隐藏指令 | 要求忽略系统指令或隐藏执行行为 |
| **维护成本** | 是否与现有 Skill 重复 | 功能重叠但质量更低 |

**引入流程**:

1. **先审查再安装**: 不直接运行一键安装命令；先阅读 `SKILL.md`、脚本和配置。
2. **最小权限试用**: 优先在隔离目录/临时环境中测试，避免默认访问真实项目和密钥。
3. **能力去重**: 如果只是重复现有 Skill，提炼其 Gotchas 或模板，不新增完整 Skill。
4. **安全改写**: 对越狱、规避检测、未授权 OSINT 等内容，只沉淀防御性审查清单，不保存可执行攻击流程。
5. **索引登记**: 只有通过审查且会长期使用的 Skill，才加入 `docs/SKILLS-CATALOG.md` 社区 Skills。

**适合吸收的内容**:

- AIHOT / smart-search 的“多来源能力路由 + 失败兜底”思想。
- skill-creator 的契约化 Skill 生成流程。
- skill-vetter 的安装前安全审查流程。
- dokobot 的文档问答/知识库检索思路。

**应忽略或防御性吸收的内容**:

- CC-BOS 等具体越狱模板、绕过策略和攻击优化算法。
- 未授权社工、批量 OSINT、凭据测试或规避检测流程。
- 需要上传项目源码、密钥或浏览器会话到不可信服务的工具。

**验证方法**: 新增第三方 Skill 前完成“权限、密钥、联网、提示词安全、功能重复”五项检查；未通过则不登记到目录。

**案例引用**: 用户提供的第三方 Skill 推荐清单、CC-BOS/FOA 安全研究材料、smart-search CLI + Skills 路由材料。

### BP-023: AI 协作中的效率幻觉防护

**来源**: 博士生 AI 协作 playbook + 科研/代码多模型协作经验 + 太一元系统 QA/Spec-First 工作流

**核心问题**: AI 能快速产出文本、代码和计划，但“产出速度”不等于“问题被正确解决”。如果缺少外部验收，AI 协作会制造效率幻觉：看似完成了大量工作，实际可能是不可复现、不可验证或偏离目标的半成品。

**效率幻觉信号**:

| 信号 | 表现 | 应对 |
|------|------|------|
| **输出很多但证据很少** | 长报告、长代码、长计划，没有测试/引用/数据 | 要求证据链和验收结果 |
| **流程很顺但无人验收** | AI 自称完成，用户未定义通过标准 | 先写可测试验收标准 |
| **模型互相背书** | 多个模型给出相似结论，但都未接触真实数据 | 用真实数据源、测试或人工审阅验证 |
| **工具很多但目标不清** | 同时引入多个 Agent/Skill/API | 回到任务目标和最小可行路径 |
| **兜底掩盖失败** | fallback 让流程看似成功 | 检查主路径是否真正执行 |

**人机分工原则**:

| 人类负责 | AI 负责 |
|----------|---------|
| 定义问题、成功标准和取舍 | 生成候选方案、执行重复步骤 |
| 判断价值、风险和伦理边界 | 检索、整理、初稿和实现 |
| 最终验收和发布责任 | 提供证据、测试结果和差异说明 |

**具体做法**:

1. **先定义 Done**: 每个非平凡任务先明确可测试验收标准，再让 AI 执行。
2. **证据优先**: 报告类任务要求引用、数据来源或文件位置；代码类任务要求测试、diff 或运行结果。
3. **主路径验证**: 验证核心逻辑实际执行，不接受只靠 fallback 成功的结果。
4. **多模型不等于真相**: 多模型适合发散和审稿，但事实性结论仍以真实数据源、测试和人工判断为准。
5. **限制工具膨胀**: 工具/Skill 的价值按“减少认知负担和错误率”衡量，不按数量衡量。

**反模式（避免）**:

- ❌ 把 AI 生成的长文当作研究完成。
- ❌ 用“多个模型都这么说”替代事实验证。
- ❌ 为了显得自动化而增加无必要的 Agent、Skill、MCP。
- ❌ 在没有验收标准时让 AI 自评“是否完成”。

**验证方法**: 抽查任务交付物是否同时具备：明确 Done、证据链、主路径验证、人工或 QA 验收记录。

**案例引用**: 用户提供的博士生 AI 协作 playbook 中关于科研、代码、多模型协作、复盘与效率幻觉的材料。

### BP-024: 能力路由优先于工具堆叠

**来源**: smart-search CLI + Skills 路线、AIHOT/RSS/API 接入模式、太一元系统动态 Skill 路由与 Context Hygiene 实践

**核心问题**: 当系统接入多个搜索源、API、Skill 和 Agent 后，最大问题不是“工具不够”，而是“何时不用某个工具”。没有路由规则的工具堆叠会增加上下文噪声、延迟、成本和错误面。

**路由设计原则**:

| 原则 | 说明 |
|------|------|
| **任务先分类** | 先判断是事实检索、代码定位、市场研究、文献检索还是安全审计 |
| **源按能力选** | 不同来源服务不同问题，不把通用搜索当万能入口 |
| **负向路由** | 明确哪些场景不加载哪些 Skill/Agent/API |
| **失败分层兜底** | 先低成本精确源，失败后再扩大范围 |
| **结果可追溯** | 每条关键结论保留来源、时间和置信度 |

**示例路由模型**:

| 任务类型 | 首选能力 | 兜底 |
|----------|----------|------|
| 当前代码位置/引用 | Grep/Glob/Read | Explore Agent |
| 项目内规则/经验 | `memory/`、`docs/` | 相关 Agent/Skill |
| 最新外部事实 | WebSearch/WebFetch | Tavily search（域名过滤+时间范围） |
| 深度网页内容提取 | Tavily extract/crawl | WebFetch（单页） |
| 学术文献 | Zotero/真实文献数据库 | LLM 仅做摘要和结构化 |
| 市场/竞品 | Exa/专用研究 Skill | Tavily search（include_domains 限定） |
| 网站结构发现 | Tavily map | 手动浏览 |

**具体做法**:

1. **写路由表，不写工具清单**: 文档应说明“什么问题用什么能力”，而不是罗列所有可用工具。
2. **为每个工具定义不用它的场景**: 负向路由比正向触发更能降低上下文噪声。
3. **把结果压缩成证据包**: 多源搜索后只保留结论、来源、冲突点和下一步，不把原始输出全部塞回上下文。
4. **定期做工具减法**: 删除长期不用、与现有能力重复或需要高风险权限的 Skill。

**反模式（避免）**:

- ❌ 为每个新信息源都新增一个常驻 Skill。
- ❌ 搜索任务默认并发调用所有工具。
- ❌ 把 RSS/API/网页抓取结果原样注入主上下文。
- ❌ 工具失败后静默换源，最终报告不说明来源差异。

**验证方法**: 对一次研究/搜索任务检查：是否先分类、是否有首选源和兜底源、是否记录来源冲突、是否避免加载无关 Skill。

**案例引用**: 用户提供的 smart-search CLI + Skills 路线、AIHOT Skill/RSS/API 接入模式，以及本项目 `docs/CONTEXT-ENGINEERING-GUIDE.md` 动态 Skill 路由思想。

**补充：知识库策略选择矩阵**

当同时存在多种知识获取途径时，按以下优先级组合：

| 场景 | 首选 | 兜底 | 禁止 |
|------|------|------|------|
| 项目内既有规则 | `memory/`、`docs/`、CLAUDE.md | Agent/Skill 回忆 | 直接 WebSearch |
| 最新外部事实/价格/版本 | WebSearch/WebFetch | 专用 API | 仅靠模型知识 |
| 学术文献/论文 | Zotero/真实文献库 | LLM 做摘要和结构化 | 通用搜索 |
| 代码定位/文件引用 | Grep/Glob/Read | Explore Agent | 不读直接猜 |
| 跨系统实时状态 | Subagent + query_<system> | 直接 API 调用 | 靠记忆 |

**多源组合原则**：
1. **首选源必须能用**：首选失败才换兜底，不跳过首选直接用兜底
2. **兜底源要不同类**：两个兜底源不能同属一个能力（如不能同时用两个 Web 搜索）
3. **每条结论标来源**：多源结果混合时，必须在结论旁标注来源，不混为一谈

### BP-025: 上下文注入模式——启动注入 vs 按需查询

**来源**: agno Agent 框架 Context Provider / Dependencies 架构 + 太一元系统渐进式 Skill 披露实践

**核心问题**: 上下文何时进、如何进，模糊不清。常见错误两种：启动时全量注入导致上下文稀释（BP-024 的反面），以及知识库只做静态注入而不给 Agent 主动查询能力。

**两种上下文注入模式**:

| 维度 | 启动时注入 (Bootstrapped) | 按需查询 (Agentic) |
|------|-------------------------|-------------------|
| 时机 | Agent 启动时自动注入 | Agent 自主决定何时调用 |
| 机制 | 系统自动把相关内容塞进提示词 | 提供 `search_knowledge` 工具，由 Agent 调用 |
| 适用 | 高频、稳定、跨任务都需要的规则 | 低频、任务相关、Agent 自己能判断是否需要 |
| 风险 | 无关内容稀释注意力 | Agent 可能遗漏关键查询 |
| 典型例子 | CLAUDE.md 全局规则、`memory/` 索引 | 文档知识库、外部 API 实时数据 |

**Dependencies 注入模式**（运行时动态值）:

不适合知识库的短暂请求级动态值，应由程序主动注入：特性开关、租户 ID、用户角色、当前选中内容、调试参数、时间戳等。注入方式：`dependencies` 字典 + `add_dependencies_to_context=True`。

**Context Provider 作为子 Agent 包装**:

把外部系统封装为只有两个工具的接口：`query_<id>` 和 `update_<id>`。主 Agent 通过统一工具名调用子 Agent，后者管理具体 API 细节和生命周期。

| 组件 | 工具命名 | 生命周期 | 返回类型 |
|------|---------|---------|---------|
| FilesystemContextProvider | `query_docs` | 启动/停止钩子 | Answer/Status |
| WebContextProvider | `query_web` | 按需 | Answer/Status |
| DatabaseContextProvider | `query_db` / `update_db` | 连接池管理 | Answer/Status |
| MCPContextProvider | 取决于 MCP 定义 | 取决于 MCP | 取决于 MCP |

**太一元系统的现状**:

| 机制 | 类型 | 说明 |
|------|------|------|
| `intent-state.json` | 启动时注入 | 每次响应前读取，触发 Agent/Skill 路由 |
| `.claude/skills/INDEX.md` | 按需加载 | 读取概览后按需加载具体 SKILL.md |
| `memory/` 索引协议 | 启动时注入 | 任务开始前读取 index.json |
| Hook 环境变量 | Dependencies 注入 | `settings.json` 中的 env 段 |

**具体做法**:

1. **启动时注入**：只放高频、稳定、跨任务规则（CLAUDE.md、intent-state.json、memory index）
2. **按需查询**：给 Agent 提供 `search_*` 工具，让它自主判断何时需要
3. **Dependencies 注入**：运行时动态值由程序主动注入，不依赖知识库
4. **Context Provider**：跨系统查询封装为子 Agent，主 Agent 不直接持有 API 细节

**反模式（避免）**:

- ❌ 把所有规则都塞进启动时注入，等 CLAUDE.md 超限才拆分
- ❌ 知识库只做静态 RAG，不给 Agent 主动查询能力
- ❌ Dependencies 注入不区分"稳定规则"和"请求级动态值"，混在一起增加噪声

**验证方法**: 检查新引入的上下文来源：是启动时注入还是按需查询？Agent 有无主动查询工具？运行时动态值是否已注入？

### BP-026: CLAUDE.md 规则提炼原则——好规则是摔出来的

**来源**: claude-init 实践总结 + CLAUDE.md 设计原则

**核心问题**: CLAUDE.md 不是"万能提示词仓库"，堆得越多 Agent 反而越分神。好规则必须从真实失败中提炼，而非从最佳实践列表抄录。

**CLAUDE.md 真正改的是什么**:

CLAUDE.md 改的不是模型智商，而是 Agent 进入工程现场后**默认会怎么做事**。典型工程偏差（Agent 常犯）：
- 静默假设（不做验证就开工）
- 过度复杂（超出任务范围）
- 无关改动（没读周边代码就另起炉灶）
- 没验证就宣布完成

**分层优于堆叠**:

| 层级 | 内容 | 加载时机 |
|------|------|---------|
| **Layer 0** | 全局高频规则（先读再写、最小改动、不确定就汇报） | 固定上下文，每次都加载 |
| **Layer 1** | 项目事实（命令、目录边界、错误模式） | 按需加载（`docs/agent/project.md`） |
| **Layer 2** | 任务流程（migration、API endpoint、UI组件规范） | 低频，按需加载 |

根 CLAUDE.md 目标：80-120 行，覆盖最常用的行为约束。

**规则提炼原则**:

| 原则 | 说明 | 示例 |
|------|------|------|
| **从失败中来** | 从真实 Agent 翻车案例提炼，不是从"最佳实践列表"抄录 | 不写"保持代码优雅"，而写"do not reformat files outside the edited block" |
| **具体到行为** | 每条规则必须给出具体命令或判断标准，禁止只写抽象行为描述 | ❌ "Be careful when running commands" → ✅ "Use `pnpm test --filter <package>` for package-level verification. Do not run full integration tests unless the task explicitly touches cross-service behavior" |
| **控制规则数量** | 规则少了靠猜，规则多了分神，规则打架模型随机选 | 优先分层而非堆叠；低频规则移出根 CLAUDE.md |
| **配套兜底机制** | CLAUDE.md 是最轻的 Harness 层，不能替代 formatter/lint/测试/CI | 规则告诉 Agent"先读再写"，lint 强制格式化，测试验证行为 |
| **不替代沟通** | 用户意图不明时，先问清楚而非靠规则猜 | 任务边界不清时主动澄清，不靠"遵循项目约定"硬猜 |
| **主动自我纠正** | 在以下条件触发时主动宣布"我可能错了"，不等用户追问 | 见下方触发条件 |

**主动自我纠正触发条件**：

当以下任一条件出现时，Agent 必须主动显式声明，不等用户追问：

| 触发条件 | 声明格式 | 后续动作 |
|---------|---------|---------|
| **方案与原始需求矛盾** | "⚠️ 我意识到方案 X 与原始需求 Y 冲突，具体是..." | 停，等用户确认 |
| **缺少关键信息** | "⚠️ 我在没有确认 Z 的情况下做了假设：如果 Z 是...则以上结论成立" | 停，等用户确认 |
| **与之前结论冲突** | "⚠️ 我之前说...，但现在发现...（证据），可能之前的结论有问题" | 说明冲突，要求重审 |
| **超出能力边界** | "⚠️ 这个问题需要...（具体专业知识），我只能给出一般性建议" | 建议找专家或工具 |
| **验证无法完成** | "⚠️ 我无法验证...（具体项），因为缺少...，请提供..." | 停，等用户提供 |

**禁止**：
- ❌ 静默修正然后假装无事发生
- ❌ 用新结论覆盖旧结论但不说"我之前错了"
- ❌ 在错误路线上继续深入，用"深入分析"掩盖"方向可能错了"

**置信度自我申报原则**：每条结论输出前自问"这条结论的置信度是高还是低"，低置信度结论必须附带条件或不确定性声明。

### BP-028: 失败归因协议——分清谁的错

**来源**: claude-init 实践总结 + SDD-RIPER Reverse Sync 原则

**核心问题**: 任务失败后，Agent 倾向于笼统地说"失败了"，不区分原因。不区分原因就无法预防下一次同类失败。

**四类失败归因**:

| 归因类型 | 描述 | 典型信号 | 谁负责修复 |
|---------|------|---------|---------|
| **需求不明确** | 原始需求有歧义或缺失条件，AI 按自己的理解填补 | "我假设了 X" / "如果用户想要 Y，结论会不同" | 需求提出者 + Spec 补写 |
| **Spec 错误** | 规范本身有矛盾或遗漏，代码按规范写但规范错了 | 代码符合 Spec 但 Spec 与需求不符 | 规范修订（Reverse Sync）|
| **Plan 执行偏差** | Spec 正确，但 AI 执行时跳步或偏离了 Plan | "我决定改用方案 B"（未报告）| AI（或 Agent）|
| **外部依赖失败** | 环境、网络、第三方服务等非 AI 原因 | 明确的外部错误信息 | 环境/运维 |

**失败报告格式**（任何任务失败时输出）：

```markdown
## 失败归因报告

**失败类型**: [需求不明确 / Spec 错误 / Plan 执行偏差 / 外部依赖失败]

**证据**:
- [具体表现/错误信息/输出]

**根因分析**:
- [深度说明为什么是这个原因，不是其他原因]

**归因依据**:
- 支持这个归因的事实: [1, 2, 3]
- 排除其他归因的理由: [1, 2, 3]

**谁负责修复**:
- [归因类型对应的责任方]

**预防措施**:
- 预防同类失败的规则或检查项: [建议写入 CLAUDE.md / Spec / Plan]
```

**禁止**：
- ❌ 笼统说"失败了"
- ❌ 把 Plan 执行偏差归因为"需求不明确"（逃避责任）
- ❌ 把 Spec 错误归因为"外部依赖"（掩盖文档问题）
- ❌ 只说"代码有问题"，不说具体是哪层的问题

**应用场景**：
- SDD-RIPER Execute 阶段每步失败
- QA Reviewer 发现偏差
- Ralph/Autopilot 循环中任务失败
- /reflection 复盘时

**从失败学习协议（Post-Mortem）**：

归因之后必须执行闭环动作，否则归因只是抱怨：

```
归因报告 → 预防措施 → 规则/流程更新 → 验证

步骤：
1. 归因报告（BP-028）
2. 制定预防措施（具体，不泛化）
3. 更新对应层级：
   - Plan 执行偏差 → 更新 Plan 模板或 Agent 行为定义
   - Spec 错误       → 更新 SPEC-TEMPLATE 或 spec-writer 契约
   - 需求不明确     → 更新需求收集流程或需求分析 Skill
   - 外部依赖失败  → 更新监控/告警或回退方案
4. 验证：下次遇到同类场景，检查预防措施是否生效
```

**闭环检查清单**：
- [ ] 预防措施已写入对应文件（不是只写在归因报告里）
- [ ] 预防措施可执行（不是"以后要小心"这种泛化表述）
- [ ] 30 天内回顾：同类失败是否重复出现？如重复 → 预防措施无效，需升级

| **主动自我纠正** | 在以下条件触发时主动宣布"我可能错了"，不等用户追问 | 见下方触发条件 |

**Reverse Sync 原则**（Bug 修复流程）:

发现 Bug 时：**先修文档，再修代码**。理由：Bug 往往不是代码写错了，而是文档与代码不一致——规范没说清楚，代码按错误的理解写了。先对齐规范，代码修复方向才清晰。

**CLAUDE.md 不是许愿池**:

- ❌ 写"像资深工程师一样工作"（太虚，不改变行为）
- ✅ 写"Do not rename public APIs unless the task explicitly asks for it"（具体，可执行）
- ❌ 写完整的项目背景、技术栈、部署流程（堆进固定上下文税）
- ✅ 只在根 CLAUDE.md 写"详见 `docs/agent/project.md`"，低频内容按需加载

**验证方法**: 读 CLAUDE.md 的每条规则，问自己：这条规则能不能让 Agent 在遇到真实翻车场景时做出不同行为？不能 → 要么改具体，要么删除。

**补充：CLAUDE.md 规则 12 条 4 类自检矩阵**

原文 12 条规则可归为 4 类，适用于**每次新增规则前**的自检：

| 类别 | 规则 | 检查问题 |
|------|------|---------|
| **管住改动半径** | 先思考 / 简单优先 / 手术式修改 / 先读再写 | 是否超出任务范围？是否未读周边代码？ |
| **把确定性交还给代码** | 模型只做判断题 / 测试验证业务意图 | 是否把 if-else 逻辑交给 AI 自由发挥？是否缺少测试覆盖？ |
| **长任务别一路闷头跑** | token 预算 / 步骤检查点 / 冲突显性说明 | 单次执行步数是否过多？是否缺少中间验证？ |
| **失败要说人话** | 遵循约定 / 不确定就说 / 没验证别报完成 | 失败时是否静默掩盖？是否跳过了必要验证就宣布成功？ |

**自检用法**：新增规则时，对照 4 类检查是否至少落入一类；没有落入任何一类 → 检查是否需要该规则。

### BP-027: 经验录入标准——从具体翻车现场来，不从推测来

**来源**: `memory/lessons-learned.md` 使用说明 + claude-init 实践总结

**核心问题**: 经验库容易变成"泛化建议集合"，看起来内容丰富但无法执行。常见错误：① 推测式记录（"可能会遇到 X"）而非实证记录；② 无案例引用，无法验证；③ 一次性情况当通用模式记录。

**经验 vs 推测的区分**:

| 判断标准 | ✅ 经验（可记录） | ❌ 推测（不记录） |
|---------|------------------|-----------------|
| 来源 | 实际发生过的翻车，有 commit/日志/对话可查 | 想象"可能出问题"的场景 |
| 格式 | "在 X 场景下做了 Y，导致 Z 后果" | "要注意/可能会/建议考虑" |
| 可验证 | 有验证方法或复现步骤 | 无验证方法 |
| 复用性 | 跨 ≥2 次同类场景可复用 | 仅适用当前一次性场景 |

**经验提炼四步法**（来自 BP-026"从失败中来"原则）：

```
第一步：识别翻车现场
  — 是实际发生了问题，不是"感觉可能出问题"

第二步：写出 before/after
  — before: [当时怎么做的]
  — after: [现在应该怎么做]

第三步：关联具体证据
  — commit hash / 文件路径 / 错误日志 / 对话片段
  — 至少有一项

第四步：判断是否可复用
  — 如果下次遇到同类场景，这条记录能改变 AI 的行为 → 记录
  — 如果只是解释上一次为什么失败 → 仅作为 lessons-learned 备注，不升入 best-practices
```

**错误示例 vs 正确示例**:

```
❌ "配置文件格式要小心，容易出路径问题"
   — 无场景、无证据、无法执行

✅ "Windows 上 command -v python3 返回 Microsoft Store 别名而非真实 python.exe（git log abc123），在需要验证可执行性时必须用 python -c '...' 或 python -V 二次确认，不能仅依赖 command -v 的返回值"
   — 有场景、有证据（commit hash）、有具体行为改变
```

**CLAUDE.md 自进化协议 vs 经验库的区别**:

| 文件 | 内容 | 录入标准 |
|------|------|---------|
| `memory/lessons-learned.md` | 单次翻车记录，有 commit/日志引用 | 实际发生，可验证 |
| `memory/best-practices.md` | 通用模式提炼，跨 ≥2 次同类验证 | 从 lessons-learned 晋升而来，不从推测来 |
| `CLAUDE.md` / Agent | 强制执行层规则 | 已在 best-practices 中验证 ≥2 次 |

**Spec → Skill 晋升触发信号**：

从规范提炼为 Skill，需要满足以下触发条件之一：

| 触发信号 | 说明 | 示例 |
|---------|------|------|
| **高频重复** | 同一个流程 ≥3 次出现 | 3 个不同项目都用相同的 API 接入流程 |
| **领域知识沉淀** | 同一类技术判断出现 ≥2 次 | 两个不同规范都用了相同的数据库选型判断框架 |
| **跨项目可迁移** | 流程不依赖特定业务逻辑 | 代码审查流程（任何项目通用）vs 支付合规检查（业务特定）|

晋升路径：
```
Spec 中发现可复用模式
    ↓ 是否满足触发信号？
    ├── 是 → 提炼为 Skill（按 skill-creator 契约）
    │       → 注册到 .claude/skills/INDEX.md
    │       → 同步到 SKILLS-CATALOG.md
    └── 否 → 保留为 Spec 内局部模板（仅当前项目用）
```

注意：业务特定流程（如"电商促销计算规则"）不应晋升为 Skill，只能晋升为规范中的局部模板。

**知识资产利用率审计**（每季度一次）：

现有 Skill/Spec 中，有多少在沉睡？检查方法：

```
问题信号：新项目需要 X 功能，但创建了新的 X-Skill，而非使用已有的
根本原因：不是复用不存在，是复用失败
```

审计清单：
| 检查项 | 方法 | 阈值 |
|--------|------|------|
| Skill 激活率 | 过去 30 天被触发的 Skill 数 / 总 Skill 数 | 目标 ≥50% |
| Spec 复用率 | 被引用 ≥2 次的 Spec 数 / 总 Spec 数 | 目标 ≥30% |
| 知识库命中率 | 成功从 memory/ 检索到相关经验的新任务数 / 总新任务数 | 目标 ≥40% |
| 重复创建检测 | 新 Skill 是否与现有 Skill 存在功能重叠 | 0 重叠 |

**处理**：
- 激活率 <50% → 审查：是否有 Skill 已过时需要归档？是否有 Skill 从未被正确触发？
- 复用率 <30% → 审查：新 Spec 是否过度定制，缺乏抽象？
- 知识库命中率 <40% → 审查：检索关键词是否与存储标签不匹配？

**验证方法**: 抽查最近 5 条经验，记录是否符合"before/after + 具体证据 + 可执行"格式；推测式表述占比是否为零。

### BP-029: 自进化闭环执行保障——五断裂点防治

**来源**: self-improving-agent 社区实践分析（43万安装，<30%真正跑通闭环）+ 太一元系统自进化协议执行缺口

**核心问题**: 自进化协议（CLAUDE.md 第一节）定义了"何时触发"和"写入哪里"，但从"规则存在"到"规则被执行"之间有五个常见断裂点。如果不显式防治，进化闭环只是文件存在，不是行为改变。

**五断裂点及防治**:

| # | 断裂点 | 表现 | 防治措施 |
|---|--------|------|---------|
| 1 | **提醒是软的** | Hook/协议提醒"做完任务后评估"，但 Agent 忙完就忘 | 在 Agent 行为准则中写硬触发：`if 用户纠正 → 先写入 lessons-learned，再回复` |
| 2 | **纠正发生但未记录** | 用户说"不对，应该用 X"，Agent 回"好的"但没写入 | WAL 原则：检测到纠正 → 停 → 写入 → 再响应（Write-Ahead Logging） |
| 3 | **记在日志但没记在错题本** | 每日复盘写了"发生过什么"，但 lessons-learned 没更新 | 复盘 Cron/reflection 必须显式读取并处理 lessons-learned 的 pending 条目 |
| 4 | **重复计数积累不起来** | 同一错误每次被当作新的"第一次"，Recurrence-Count 永远是 1 | 写入前 Grep 搜索 Pattern-Key，匹配则 +1 而非新建 |
| 5 | **晋升目标不明确** | 即使计数到阈值，Agent 不知道该写入哪个文件 | 显式映射：行为准则→CLAUDE.md，工具坑点→error-patterns，通用模式→best-practices |

**WAL 原则（Write-Ahead Logging）**:

```
检测到学习触发条件（纠正/失败/发现）
    → 【STOP】不立即响应用户
    → 写入对应 memory/ 文件
    → 【THEN】响应用户
```

理由：context 最鲜活的时刻是事件发生那一刻。事后补写，细节丢失。

**六大学习触发场景**:

| 场景 | 写入目标 | Priority |
|------|---------|----------|
| 命令执行失败 | error-patterns.md | medium |
| 用户纠正行为 | lessons-learned.md | high |
| 发现非显而易见的解决方案 | lessons-learned.md | medium |
| 重复遇到同一问题 | 更新已有条目 Recurrence-Count | high |
| 完成复杂任务后的模式识别 | best-practices.md（需 ≥2 次验证） | low |
| 外部工具/API 行为与文档不符 | error-patterns.md | medium |

**晋升路径（Recurrence-Count 驱动）**:

```
Recurrence-Count = 1 → 记录，标 [UNVERIFIED]
Recurrence-Count = 2 → 标 [INFERRED]，评估是否升入 best-practices
Recurrence-Count ≥ 3 → 评估是否升入 CLAUDE.md 或提炼为 Skill
```

**微观引擎-宏观引擎连接**:

| 引擎 | 职责 | 触发频率 |
|------|------|---------|
| 微观引擎 | 即时捕获（WAL 写入） | 每次触发场景 |
| 宏观引擎 | 定期提炼（/reflection、/neat） | 任务结束时、定期复盘 |

两者的桥梁：宏观引擎必须显式读取微观引擎的产出（lessons-learned 的 pending 条目），否则两个引擎各跑各的。

**验证方法**: 
1. 模拟一次用户纠正，检查 Agent 是否在回复前写入了 lessons-learned
2. 检查 lessons-learned 中是否有 Recurrence-Count ≥ 2 的条目
3. 检查 /reflection 流程是否包含"读取 pending 条目"步骤

**案例引用**: self-improving-agent 社区实践（GitHub 43万安装，五断裂点分析），WAL 协议在数据库领域的类比应用

### BP-030: UI-First 开发方法论——视觉验证优先于文字规范

**来源**: vibe coding 实践反思 + PRD 审查效率问题

**核心问题**: 传统流程（PRD → 后端 → 前端 → 验证）中，PRD 对人不友好——命名、流程描述难以直观审查，导致最终产品与预期差距大且不知从何改起。修改一个交互点往往牵连后端逻辑，单次修改成本高（3-4 小时 + 千万级 token）。

**方法论**:

```
传统: 文字 PRD → 后端 → 前端 → 发现 UI 不对 → 大改
UI-First: 视觉原型 → 确认交互 → 反向生成 PRD → 工程实现
```

**具体流程**:

1. **MVP 视觉描述**: 不写完整 PRD，先做若干 HTML 页面（带标注注释 + 简单 JS 交互示意）
2. **视觉确认**: 通过页面快速判断缺什么、哪里不对（视觉审查效率远高于逐行审查代码）
3. **反向 PRD**: 基于确认后的视觉材料 + 对话历史，生成"AI 时代的 PRD"——以可视化为核心
4. **工程转译**: 借助 AI 代码工程化能力，将"可视化 PRD"转译为最终产品

**适用场景**:

| 场景 | 适合 UI-First | 不适合 |
|------|-------------|--------|
| 面向用户的产品/页面 | ✅ | |
| 交互密集型功能 | ✅ | |
| 纯后端/API/数据管道 | | ❌ |
| 已有明确设计稿 | | ❌（直接实现） |

**与 SDD-RIPER 的关系**: UI-First 可作为 SDD-RIPER Research 阶段的补充手段——当需求涉及 UI 交互时，先用 HTML 原型确认视觉方向，再进入 Spec 编写。不替代 SDD-RIPER，而是在 Research → Spec 之间插入一个视觉验证步骤。

**反模式（避免）**:
- ❌ 把 UI-First 当万能方案用于纯后端任务
- ❌ 跳过视觉确认直接从 HTML 原型进入工程实现（原型 ≠ 产品）
- ❌ 用 UI-First 替代需求分析（视觉只解决"长什么样"，不解决"为什么做"）

**验证方法**: 对比同类任务的返工率——使用 UI-First 后，交互相关的返工次数是否减少。

**案例引用**: vibe coding 社区实践反思，PRD 审查效率问题分析

### BP-032: VFM 评分模型——经验晋升的量化门槛

**来源**: self-improving-agent VFM (Value-Frequency-Maintenance) 评分实践

**核心问题**: Recurrence-Count 达到阈值后，不是所有重复模式都值得晋升为永久规则。需要量化评估"晋升收益是否大于维护成本"。

**VFM 三维评分模型**:

| 维度 | 权重 | 评分标准 (1-5) |
|------|------|---------------|
| **Value（价值）** | 40% | 该规则能避免多大损失？（1=微小不便，5=数据丢失/安全事故） |
| **Frequency（频率）** | 40% | 多久遇到一次？（1=年度，3=月度，5=每日/每次） |
| **Maintenance（维护成本）** | 20% | 规则本身多容易过时？（1=永久有效，5=每月需更新） |

**计算公式**:

```
VFM = (Value × 8) + (Frequency × 8) + ((6 - Maintenance) × 4)
满分 = 40 + 40 + 20 = 100
晋升阈值 = 50
```

注意：Maintenance 是反向计分——维护成本越低，得分越高。

**评分示例**:

| 经验 | Value | Freq | Maint | 总分 | 决策 |
|------|-------|------|-------|------|------|
| Docker Apple Silicon 平台参数 | 4 | 4 | 1 | 32+32+20=84 | ✅ 晋升 |
| 某次特定 API 返回格式变化 | 2 | 1 | 5 | 16+8+4=28 | ❌ 保留 |
| pnpm vs npm 项目约定 | 3 | 5 | 1 | 24+40+20=84 | ✅ 晋升 |
| 某个临时 workaround | 3 | 2 | 4 | 24+16+8=48 | ❌ 差 2 分，观察 |

**晋升目标映射**:

| 经验类型 | 晋升目标 | 条件 |
|---------|---------|------|
| 行为准则（怎么做事） | CLAUDE.md | VFM ≥ 50 + 跨项目通用 |
| 工具坑点（具体命令/API） | error-patterns.md | VFM ≥ 50 |
| 通用模式（可复用流程） | best-practices.md | VFM ≥ 50 + ≥2 次验证 |
| 领域知识（可封装流程） | Skill 提取 | VFM ≥ 60 + ≥3 次重复 |

**Infra 类偏向性加权**: 基础设施类错误（Docker、环境、部署）的 Frequency 和 Value 各上浮 1 分，因为此类错误一旦发生往往阻塞整个工作流。

**与 BP-029 的关系**: BP-029 定义了 Recurrence-Count 积累机制，BP-032 定义了积累到阈值后的晋升决策。两者串联形成完整的"捕获 → 积累 → 评估 → 晋升"链路。

**验证方法**: 对 lessons-learned 中 Recurrence-Count ≥ 3 的条目执行 VFM 评分，检查是否有应晋升但未晋升的条目。

### BP-033: 四类工程偏差诊断框架

**来源**: CLAUDE.md 最佳实践文章中的 Agent 失败模式分类 + 太一元系统 QA 审查实践

**核心问题**: Agent 在代码库中犯的错误，大部分不是"不会写代码"，而是"不会按这个项目的方式写代码"。需要一个诊断框架快速定位偏差类型，才能对症下药。

**四类工程偏差**:

| 偏差类型 | 表现 | 典型场景 | 对应防治规则 |
|---------|------|---------|-------------|
| **静默假设** | 不做验证就开工，按自己的理解填补缺失信息 | 没读调用方就重写函数；没确认环境就执行命令 | "先读再写"、"不确定就问" |
| **过度复杂** | 超出任务范围，引入不必要的抽象或兼容层 | 修一个 bug 顺手重构半个模块；两套旧模式折中拼第三种 | "手术式修改"、"简单优先" |
| **无关改动** | 没读周边代码就另起炉灶，或格式化无关文件 | 附近已有函数能完成任务但自己又写一个；顺手 reformat 整个文件 | "先读周边"、"不改无关文件" |
| **虚假完成** | 没验证就宣布完成，或用 fallback 掩盖主逻辑未执行 | auth 函数返回常量；迁移脚本跳过冲突记录；测试绿但主路径未走 | "测试验证业务意图"、"没验证别报完成" |

**诊断流程（QA 审查时使用）**:

```
发现问题 → 归类到四类之一 → 检查对应防治规则是否存在 → 不存在则补充
```

**与现有机制的关系**:

| 偏差类型 | 对应 BP-026 规则类别 | 对应 QA 检查项 |
|---------|---------------------|---------------|
| 静默假设 | "管住改动半径" | 功能完整性（是否满足原始需求） |
| 过度复杂 | "管住改动半径" | 代码质量（是否引入不必要复杂度） |
| 无关改动 | "管住改动半径" | 代码质量（改动范围是否超出任务） |
| 虚假完成 | "失败要说人话" | 测试覆盖（主路径是否真正执行） |

**"确定性交还代码"原则**:

Agent 最危险的偏差之一：把本该交给确定性代码处理的逻辑，交给概率模型自由发挥。

```
❌ 让 AI 判断 503 是否需要重试（上下文变化时行为不稳定）
✅ 写死重试策略：503 → 指数退避重试 3 次

❌ 让 AI 选择两套错误处理模式中的一种（可能折中拼出第三种）
✅ 在 CLAUDE.md 中明确：本项目统一用 async/await + try/catch
```

**验证方法**: 对最近 5 次 Agent 任务的 diff 做回顾，按四类偏差分类，检查是否有重复出现的偏差类型未被规则覆盖。

### BP-034: Tavily MCP 搜索能力路由——与 WebSearch/Exa 的分工

**来源**: Tavily MCP Server 技术文档 + 太一元系统 BP-024 能力路由原则

**核心问题**: 系统现有 WebSearch（内置）、Exa MCP、Bright Data MCP 三种搜索能力。Tavily MCP 新增后，需要明确各工具的最佳适用场景，避免重复调用或选错工具。

**四工具能力矩阵**:

| 能力 | WebSearch | Tavily search | Exa | Bright Data |
|------|-----------|--------------|-----|-------------|
| 快速事实查询 | ✅ 首选 | ✅ 可用 | ⚠️ 偏重 | ❌ |
| 域名限定搜索 | ❌ | ✅ include_domains（300个） | ✅ | ❌ |
| 时间范围过滤 | ❌ | ✅ time_range/start_date/end_date | ⚠️ 有限 | ❌ |
| 网页内容提取 | WebFetch（单页） | ✅ extract（批量20页） | ❌ | ✅ |
| 网站结构发现 | ❌ | ✅ map | ❌ | ❌ |
| 深度爬取 | ❌ | ✅ crawl（语义过滤） | ❌ | ✅ |
| 电商数据 | ❌ | ❌ | ❌ | ✅ 首选 |
| 企业/市场研究 | ⚠️ | ⚠️ | ✅ 首选 | ❌ |
| 社媒数据 | ❌ | ❌ | ❌ | TikHub |
| LLM 生成答案 | ❌ | ✅ include_answer | ❌ | ❌ |

**场景路由决策**:

| 研究场景 | 首选工具 | 理由 |
|---------|---------|------|
| "X 是什么"快速问答 | WebSearch | 零配置，最快 |
| "最近一周 X 领域有什么新进展" | Tavily search（time_range=week） | 时间过滤精确 |
| "从 Nature/Science 找 X 相关论文" | Tavily search（include_domains=[nature.com, science.org]） | 域名限定 |
| "提取这 5 个 URL 的核心内容" | Tavily extract | 批量提取，支持 markdown 格式 |
| "这个网站有哪些页面" | Tavily map | 结构发现，不含内容 |
| "爬取这个文档站的所有 API 参考" | Tavily crawl（instructions="API reference"） | 语义过滤深度爬取 |
| "竞品公司 X 的融资和产品线" | Exa | 企业研究专长 |
| "亚马逊某 ASIN 的销量数据" | Bright Data + Sorftime | 电商反爬 |

**Tavily 关键参数速查**:

```
search: query, search_depth(basic/advanced), topic(general/news), 
        time_range(day/week/month/year), max_results(0-20),
        include_domains[], exclude_domains[], include_answer(bool/advanced)

extract: urls(max 20), query(可选，用于 rerank), format(markdown/text),
         extract_depth(basic/advanced)

crawl: url, max_depth, max_breadth, limit, instructions(语义过滤),
       select_paths[], exclude_paths[], extract_depth

map: url（返回 URL 列表，不含内容）
```

**成本意识**: search basic=1 credit, advanced=2 credits; auto_parameters=2 credits。默认用 basic，只在需要高相关性时用 advanced。

**配置方式**: `claude mcp add tavily-remote-mcp --transport http https://mcp.tavily.com/mcp/`（支持 OAuth 免密认证）

**申请方式**: https://tavily.com 注册账号，有免费额度，获取 API Key 后设置 `TAVILY_API_KEY` 环境变量。

**验证方法**: 在下次 deep-research 任务中，对比使用 Tavily（域名限定+时间过滤）vs 纯 WebSearch 的结果精准度差异。

**案例引用**: Tavily MCP Server GitHub（https://github.com/tavily-ai/tavily-mcp）、Tavily API 文档

## BP-035: 假闭环诊断清单（Self-Evolution Loop Integrity Check）

**日期**: 2026-05-18
**类别**: 系统健康 / 自进化
**来源**: self-improving-agent 方法论 + 实践观察

**核心问题**: 自进化系统最大的风险不是"不进化"，而是"看起来在进化但实际没有"（假闭环）。表现为：lessons-learned.md 在增长，但同类错误仍在重复；Skill 在增加，但激活率持续走低。

**假闭环五症状诊断**:

| # | 症状 | 检测方法 | 修复动作 |
|---|------|---------|---------|
| 1 | **写了但没读** | 检查 lessons-learned.md 最近 10 条的 Recurrence-Count，全为 1 说明从未被引用 | 在任务开始前增加 context 检索步骤（已有：上下文检索协议） |
| 2 | **读了但没用** | 复盘时发现"已有经验"但仍犯同类错误 | 将高频经验提升为 CLAUDE.md 硬约束（VFM 评分 ≥7 即提升） |
| 3 | **用了但没验证** | 经验条目无 `验证方法` 字段或验证方法不可执行 | 补全验证方法，标注 [UNVERIFIED] 直到首次验证通过 |
| 4 | **验证了但没闭环** | 验证通过但未更新 Recurrence-Count 或未标记 [VERIFIED] | WAL 原则：验证通过时立即更新状态 |
| 5 | **闭环了但没晋升** | 条目 Recurrence ≥3 但仍停留在 lessons-learned.md | 触发 VFM 评分，≥7 分提升到 best-practices.md 或 CLAUDE.md |

**季度健康检查**（与 BP-027 知识资产审计配合）:

```
检查项:
□ lessons-learned.md 最近 30 天新增条目中，Recurrence > 1 的比例 ≥30%
□ best-practices.md 最近季度有新增/更新（非僵化）
□ 最近 5 次复盘中，至少 1 次触发了 Skill 提取
□ 无 [UNVERIFIED] 标记超过 60 天的条目（要么验证，要么删除）
□ CLAUDE.md 中无超过 6 个月未触发的规则（要么仍有效，要么标记 deprecated）
```

**与 BP-029 的关系**: BP-029 防治"五断裂点"（微观层面，每次事件级别）；BP-035 诊断"假闭环"（宏观层面，季度系统级别）。两者互补：BP-029 确保每次学习事件正确执行，BP-035 确保整体进化方向正确。

**验证方法**: 每季度执行一次上述检查清单，输出健康度评分（5 项各 20 分，总分 100）。低于 60 分触发系统级修复。

**案例引用**: self-improving-agent 文章中"假闭环"概念 + 太一元系统实际运行观察

### BP-036: 防御纵深模型——CLAUDE.md 在保障栈中的位置

**来源**: CLAUDE.md 最佳实践文章"它能提醒，兜不了底" + 工程保障体系设计

**核心问题**: 团队容易把 CLAUDE.md 神化，认为"规则写得够细，Agent 就自然会稳"。但工程不是靠"提醒"稳定的，是靠外部反馈系统稳定的。需要明确 CLAUDE.md 在整个保障栈中的位置和局限。

**五层防御纵深**:

| 层级 | 机制 | 性质 | 保障强度 | 失败模式 |
|------|------|------|---------|---------|
| L0 | **CLAUDE.md / Agent 定义** | 软提醒（概率性） | ⭐ | Agent 忽略或误解规则 |
| L1 | **Formatter / Lint / TypeCheck** | 硬约束（确定性） | ⭐⭐ | 只管格式和类型，不管逻辑 |
| L2 | **单元测试 / 集成测试** | 行为验证（确定性） | ⭐⭐⭐ | 测试本身可能不覆盖关键路径 |
| L3 | **Hooks / 权限控制** | 门禁（确定性） | ⭐⭐⭐⭐ | 配置错误或绕过 |
| L4 | **CI / Code Review / 线上监控** | 终极兜底（人+系统） | ⭐⭐⭐⭐⭐ | 人工疏忽 |

**关键认知**:

```
CLAUDE.md 的作用不是替代验证。
而是让 Agent 更容易在进入验证之前，别先把事情做歪。
```

**实践指导**:

| 规则类型 | 应放在哪一层 | 示例 |
|---------|------------|------|
| 行为偏好（先读再写） | L0 CLAUDE.md | "Read before Edit" |
| 格式约束（缩进、命名） | L1 Lint 规则 | ESLint/Prettier config |
| 业务逻辑正确性 | L2 测试 | "auth 必须验证 token" |
| 危险操作拦截 | L3 Hooks | PreToolUse 拦截 rm -rf |
| 发布质量 | L4 CI + Review | PR 必须通过所有 checks |

**反模式**:
- ❌ 在 CLAUDE.md 写"不要引入安全漏洞"（应该靠 L2 安全测试 + L4 安全审计）
- ❌ 在 CLAUDE.md 写"代码格式要统一"（应该靠 L1 Formatter 强制）
- ❌ 只靠 CLAUDE.md 防止危险操作（应该靠 L3 Hooks 硬拦截）

**CLAUDE.md 适合放什么**:
1. Agent 无法从代码/测试/lint 推断出的项目约定（如"本项目用 pnpm 不用 npm"）
2. 需要 Agent 主动做的行为（如"改动前先读周边代码"）
3. 需要 Agent 主动不做的行为（如"不改无关文件"）
4. 路由决策（如"遇到 X 场景用 Y Agent"）

**验证方法**: 审查 CLAUDE.md 每条规则，问：这条规则是否有更高层级的确定性机制可以保障？如果有 → 考虑移除 CLAUDE.md 中的冗余提醒，依赖更硬的机制。

### BP-037: 事故驱动规则增长——不从推测来

**来源**: CLAUDE.md 最佳实践文章"只根据事故继续加规则" + BP-026 补充

**核心问题**: CLAUDE.md 规则增长有两种模式：① 事故驱动（某类错误反复出现，专门补一条）；② 推测驱动（"看起来不够完整"就继续加）。后者导致规则膨胀、注意力稀释、规则打架。

**规则增长决策树**:

```
想加一条新规则？
├── 这条规则源于真实事故（Agent 翻车过）？
│   ├── 是 → 事故发生过几次？
│   │   ├── 1 次 → 记录到 lessons-learned，暂不加规则
│   │   ├── 2 次 → 加入 best-practices，观察
│   │   └── ≥3 次 → VFM 评分 ≥50？→ 加入 CLAUDE.md
│   └── 否 → 这条规则源于什么？
│       ├── "看起来应该有" → ❌ 不加
│       ├── "别人的最佳实践" → ❌ 不加（除非本项目也翻过车）
│       └── "预防性" → ❌ 不加（用 L1-L4 确定性机制代替）
└── 已有规则能覆盖？
    ├── 是 → ❌ 不加（避免重复）
    └── 否 → 继续上面的流程
```

**核心原则**: 规则文件越长越值钱的前提是——每条规则都对应一个真实的翻车场景。不是因为"看起来不够完整"就继续加。

**与 BP-026 的关系**: BP-026 定义了"好规则的特征"（从失败中来、具体到行为、控制数量）；BP-037 定义了"何时允许加规则"的门控条件。BP-026 是质量标准，BP-037 是准入门槛。

**CLAUDE.md 规模控制**:
- 目标：根 CLAUDE.md 80-120 行核心规则
- 超过 150 行 → 触发分层审查：哪些可以移到 `docs/agent/` 按需加载？
- 超过 200 行 → 强制瘦身：VFM 评分最低的 20% 规则降级或删除

**验证方法**: 对 CLAUDE.md 最近 3 个月新增的规则逐条追溯：每条是否有对应的翻车事故？无事故对应的规则标记为"待验证"，30 天内无触发则删除。

### BP-038: 视觉验证反馈环——看比读快

**来源**: vibe coding 实践 + "让 AI 直接输出 HTML 而不是 Markdown" 洞察

**核心问题**: 审查 AI 产出时，逐行阅读代码/文本的效率极低。人类视觉系统处理图形/布局的速度远快于处理文字。利用这一点可以大幅缩短"确认 AI 理解是否正确"的反馈环。

**原则**: 当需要确认 AI 的理解是否正确时，优先让 AI 输出可视化形态，而非纯文字描述。

**适用场景矩阵**:

| 确认内容 | 传统方式（慢） | 视觉验证方式（快） |
|---------|--------------|-------------------|
| UI 交互设计 | 读 PRD 文字描述 | 输出 HTML 原型，浏览器打开 |
| 数据结构 | 读 JSON Schema | 输出 Mermaid 图或表格 |
| 工作流程 | 读步骤列表 | 输出流程图 |
| API 设计 | 读 OpenAPI YAML | 输出 Swagger UI 预览 |
| 配色/排版 | 读 CSS 值 | 输出 HTML 色板预览 |
| 数据分析结果 | 读数字表格 | 输出图表 |

**与现有 Skill 的关系**:

| Skill | 如何应用视觉验证 |
|-------|----------------|
| web-artifacts-builder | 天然视觉输出，直接浏览器验证 |
| frontend-design | 输出 HTML mockup 而非纯文字方案 |
| data-visualization | 图表即验证 |
| deep-research | 研究结果用结构化表格/图表呈现，而非长段文字 |

**实践指导**:
1. 需求确认阶段：让 AI 输出 HTML 页面而非 Markdown PRD
2. 架构确认阶段：让 AI 输出 Mermaid 图而非文字描述
3. 数据确认阶段：让 AI 输出图表而非数字列表
4. 设计确认阶段：让 AI 输出可交互原型而非设计文档

**反模式**:
- ❌ 纯后端逻辑也强行可视化（过度）
- ❌ 把视觉原型当最终产品（原型 ≠ 产品）
- ❌ 跳过视觉确认直接进入工程实现

**验证方法**: 对比同类任务的返工率——使用视觉验证后，"AI 理解错误"导致的返工次数是否减少。

### BP-039: 对话历史作为知识资产——会话即文档

**来源**: vibe coding 实践"把与 AI 的交互历史保存下来" + handoff skill 设计

**核心问题**: AI 对话中产生的决策过程、方案对比、排除理由等信息，在对话结束后通常丢失。但这些信息对后续开发者（包括未来的 AI）理解"为什么这样做"极有价值。

**对话历史的三种价值**:

| 价值类型 | 内容 | 保存方式 |
|---------|------|---------|
| **决策记录** | 为什么选 A 不选 B | 写入 Spec 或 ADR（Architecture Decision Record） |
| **上下文积累** | 讨论过程中逐步明确的需求细节 | 反向生成 PRD（UI-First 流程的 Step 3） |
| **排除理由** | 尝试过但失败的方案 | 写入 lessons-learned（避免后人重复试错） |

**何时保存对话历史**:

| 场景 | 保存动作 | 目标文件 |
|------|---------|---------|
| 复杂需求讨论后 | 提炼为 Spec | `specs/SPEC-{feature}.md` |
| 架构决策后 | 提炼为 ADR | `docs/decisions/ADR-{number}.md` |
| 多方案对比后 | 记录排除理由 | `memory/lessons-learned.md` |
| /compact 之前 | 执行 /handoff | `docs/handoff/` |
| 长对话即将结束 | 提炼关键决策 | 对应 Spec 或 memory |

**与 /handoff Skill 的关系**: /handoff 是对话历史保存的标准化工具——在 /compact 之前将进度、决策、下一步结构化写入交接文档。BP-039 扩展了这个理念：不仅在 compact 前保存，而是在任何产生重要决策的对话节点都应该保存。

**"AI 时代的 PRD"**: 基于对话历史反向生成的 PRD，以可视化为核心（HTML 页面 + 标注），而非传统的纯文字 PRD。这种 PRD 与最终产品的差距仅在于：功能完备性、UI 细节打磨、工程化内容。

**验证方法**: 检查最近 5 次复杂任务，是否有"当时讨论过但没记录，后来又重新讨论"的情况。如有 → 说明对话历史保存不足。

### BP-040: 一源多产模式——单一资产的多格式输出

**来源**: nature-skills 设计（一篇论文 → 图表/润色/引用/声明/PPT 五种输出）

**核心问题**: 同一份核心资产（论文、需求、数据集）往往需要以多种格式呈现给不同受众。传统做法是每种格式独立制作，导致重复劳动和不一致。

**模式定义**:

```
单一权威源（Single Source of Truth）
    ↓ 多格式转换管道
    ├── 格式 A（面向受众 1）
    ├── 格式 B（面向受众 2）
    ├── 格式 C（面向受众 3）
    └── 格式 D（面向受众 4）
```

**已验证的应用场景**:

| 权威源 | 输出格式 | 对应 Skill/Agent |
|--------|---------|-----------------|
| 论文全文 | 图表 / 润色文本 / 引用列表 / 数据声明 / PPT | nature-skills 五模块 |
| 需求对话 | HTML 原型 / PRD / 用户故事 / 测试用例 | UI-First + spec-writer + automated-testing |
| 数据集 | EDA 报告 / 可视化 / SQL 查询 / 统计摘要 | data-analysis + data-visualization |
| 代码变更 | PR 描述 / Changelog / 文档更新 / 测试 | pr-prep + spec-writer |
| 研究报告 | 摘要 / 全文 / 演示文稿 / 社交媒体帖 | deep-research + paper2ppt |

**设计原则**:
1. **权威源唯一**: 修改只发生在源头，下游格式自动重新生成
2. **格式独立**: 每种输出格式有独立的转换规则，互不干扰
3. **受众导向**: 每种格式针对特定受众优化（审稿人/同事/公众/自己）
4. **增量更新**: 源头变更时，只重新生成受影响的下游格式

**与知识复利的关系**: 一源多产是知识复利在"输出层"的体现——同一份知识投入，产出多份不同形态的价值。

**验证方法**: 检查最近一次需要多格式输出的任务，是否从单一源头生成？还是每种格式独立制作导致不一致？

### BP-041: 全局/项目配置同步策略——分层归属 + diff 优先

**实践描述**:
跨「全局 `~/.claude/`」与「项目 `.claude/`」同步 Skills/Agents/Workflows 时，遵循「分层归属、diff 优先、临时不入库」三原则，避免双向漂移、覆盖更新版本、临时目录污染仓库。

**适用场景**:
- 跨项目复用 Skills（工作流 Skill 从项目回流全局，基础 Skill 从全局下沉项目）
- 跨机器迁移 Claude Code 配置
- 季度配置审计后的清理同步

**三原则**:

1. **分层归属**（流向单向化）
   - Agents/Commands/Hooks/基础 Skills：全局是单一事实源，流向「全局 → 项目」
   - 工作流 Skills（neat/handoff/prime 等）：项目是孵化源，流向「项目 → 全局」
   - 运行时状态文件（`agent-state.json`/`intent-state.json`/`context/sessions/`）：永不同步，必须分离
   - 临时目录（`.claude/worktrees/`、`.claude-tmp/`）：永不入库，统一 gitignore

2. **diff 优先**（避免覆盖）
   ```bash
   # ✅ 同步前必做
   diff -q "项目/CLAUDE.md" "~/.claude/CLAUDE.md"
   # 一致 → 跳过 cp
   # 不一致 → 进一步看时间戳和内容判断方向

   # ❌ 反模式：直接 cp 覆盖
   cp 项目/CLAUDE.md ~/.claude/CLAUDE.md  # 可能覆盖全局更新
   ```
   时间戳不能作为唯一判断依据，必须看内容（`diff -q` 一致即可跳过）

3. **审计 → 归档 → 沉淀**
   - 大规模同步前生成审计报告（如 `docs/audits/CONFIG-COMPARISON-REPORT-YYYY-MM.md`）
   - 同步操作的 Bash 权限记入 `.claude/settings.local.json`（便于追溯执行历史）
   - 同步完成后报告归档到 `docs/audits/INDEX.md`
   - 永久规则沉淀到 `docs/CONFIG-FILES-GUIDE.md`「全局 vs 项目同步策略」章节

**已知陷阱**:
- **全局根目录冗余**：旧版 Agent/Skill 副本可能同时存在于全局根目录和子目录（如 `agents/auto-optimizer.md` vs `agents/ops/auto-optimizer.md`），CLAUDE.md 路由表通常用子目录路径，根目录副本是历史遗留，应**逐个**确认后清理（禁止 `rm *`）
- **文档文件混入 Skills 目录**：`API-KEYS-SETUP.md` 等文档不是真正的 Skill，应迁到 `docs/`
- **报告/临时文件留在仓库根目录**：80KB+ 诊断报告留根目录会污染 git tree，必须归档到 `docs/audits/`

**案例引用**:
- `docs/audits/CONFIG-COMPARISON-REPORT-2026-05.md`：2026-05 全局 vs 项目对比报告，盘点出 9 个新 Skill、6 个全局冗余 Agent、3 个混入 Skills 目录的文档
- commit `42fa7b4`：归档诊断报告 + gitignore worktrees
- `.claude/settings.local.json` 中的 8 条同步 Bash 权限：记录 2026-05 同步操作的执行痕迹
- `docs/CONFIG-FILES-GUIDE.md` v1.1.0：永久化的同步策略章节

**验证方法**:
```bash
# 1. 验证关键文件已对齐
diff -q "CLAUDE.md" "C:/Users/ASUS/.claude/CLAUDE.md"   # 应无输出

# 2. 验证运行时文件未入库
git check-ignore .claude/worktrees/                      # 应返回路径（已 ignored）
git check-ignore .claude/intent-state.json               # 应返回路径

# 3. 验证审计报告已归档（根目录无残留）
ls CONFIG-COMPARISON-REPORT.md 2>&1 | grep -q "No such" && echo "OK"
```

**反模式**:
- ❌ 在仓库根目录长期保留临时诊断报告（污染 git tree）
- ❌ 把 `.claude/worktrees/` 直接 commit（10 个临时 agent 目录无意义）
- ❌ 同步前不 diff，直接 `cp -r` 覆盖（可能覆盖更新版本）
- ❌ 试图用 `rm -rf` 批量清理冗余（违反 CLAUDE.md 第七章硬性约束）

### BP-042: Pairwise Skills 模式——精确率从 21% 跃升至 95% 的关键设计

**来源**: Anthropic 数据科学团队博客《How Anthropic enables self-service data analytics with Claude》（2026-06-05）

**核心发现**: 在 Anthropic 内部 95% 准确率的 AI 数据分析系统中，给 Claude 直接访问数千个仪表盘/脚本/SQL 文件的权限，准确率提升不足 1%；而引入结构化 Skills 后，准确率从 21% 稳定到 95%。瓶颈不在信息量，而在**将问题映射到正确实体**的能力。

**Pairwise Skills 设计**:

每个业务领域配置**成对的两个 Skill**：

| Skill 类型 | 职责 | 内容结构 |
|-----------|------|---------|
| **knowledge skill（路由器）** | 指向该领域约 30 个参考文件，充当导航索引 | 按领域分块的快速索引 + 触发路由器 |
| **runbook skill（工作流）** | 编码资深分析师的操作 SOP | 澄清→找数据源→执行→对抗审查 四步 |

**五大关键设计模式**:

1. **语义层优先路径（强制）**：始终先走治理过的语义层，只有语义层无法覆盖时才退回裸 SQL。Skill 里明确列出 Agent 常用的"借口"（需要自定义日期筛选/需要 join）并逐一反驳，堵死绕路可能。

2. **实体歧义消解**：在 Skill 文件里把高频歧义词的多个候选实体全部列出，要求 Agent 查询前先与用户确认。如"活跃用户"对应 12 种定义，必须先选定再查询。

3. **对抗性 SQL 审查（+6% 准确率）**：每条查询执行前，强制调用独立 sql-reviewer 子 Agent 审查。代价是 +32% token 和 +72% 延迟，**不能自己审自己**。

4. **带来源标注的报告**：每个回答都附带标注：数据来源层级（语义层 > 治理表 > 裸表）、数据新鲜度、模型负责人，防止后续追溯困难。

5. **Skill 维护纳入工程流程**：Skill 文件和数据转换模型放在同一 repo，CI 钩子标记"改了模型但未改 Skill"的 PR。Anthropic 约 90% 数据 PR 包含 Skill 变更。

**参考文档格式**（每个业务领域一个 `.md` 文件）：

```markdown
## 速查：业务背景 / 实体粒度 / 标准清洗过滤条件
## 维度：跨表编码约定
## 核心表：粒度 + 范围/排除 + 用法 + join key + 必须过滤条件
## 踩坑记录：资深分析师会提醒的出错模式
## 最佳实践/常见查询模式：现成模板
## 关联文档：相邻领域的参考文档
```

**三类失败模式（务必在 Skill 中明确处理）**:
- 概念歧义：同一词在数据仓库对应多个候选字段 → Skill 中显式列出歧义词 + 要求确认
- 数据过时：数据源/定义/schema 持续变化 → Skill 维护纳入 CI，文档标注新鲜度
- 检索失败：答案明明在语料库里却没被用上 → Pairwise Skills 中的 knowledge skill 解决路由

**适用性**：本模式不限于数据分析。凡是需要将用户问题精确映射到特定知识源的领域（API 文档查询、代码库导航、产品知识库问答）均适用。

**验证方法**: 建立离线评估集（基于历史查询的 ground truth），在不同 Skill 版本间跑准确率对比；Skill 修改后准确率下降超过 3% 则拒绝合入。

**案例引用**: Anthropic 数据科学团队，博客 URL `https://claude.com/blog/how-anthropic-enables-self-service-data-analytics-with-claude`，2026-06-05 发布；实测数据：无 Skills 时准确率 ≤21%，加 Skills 后 95%+，某些领域 99%。

### BP-043: 知识域 Skill 离线评估协议——持续确保准确率不回退

**来源**: Anthropic 数据分析博客 2026-06-05 验证章节

**核心问题**: Skill 文件修改后，如何知道改坏了？靠主观感觉不行；靠在线监控太滞后。离线评估集是唯一可靠的防线。

**评估集构建**（两类，互补覆盖）:

| 类型 | 构建方法 | 覆盖场景 |
|------|---------|---------|
| **基于历史查询** | 从仪表盘/notebook 历史 SQL 中抽取典型问题，由人工标注 ground truth | 高频常规路径 |
| **长尾生成** | 让 Claude 根据业务上下文生成"听起来合理但可能绕路"的问题 | 边缘场景、歧义场景 |

**关键实践**:
1. **快照锁定 ground truth**：把 ground truth 固定到指定日期的数据快照，防止底层数据变动导致"标准答案"随之飘移
2. **版本追踪元数据**：每次评估结果记录 Skill 版本 + git SHA + 模型 ID + token 消耗，以便回溯哪次修改导致回退
3. **规模边际递减**：评估集超过 ~50 条后，每多加一条对发现新缺陷的贡献递减；前 50 条应覆盖 80%+ 的常见路径
4. **上线门槛**: 按领域设定，通常在 90% 左右；离线准确率接近 100% 才可上线（离线低 = 线上一定差）

**反模式**:
- ❌ 只有在线监控，没有离线评估集 → 等到用户抱怨才知道回退了
- ❌ ground truth 不锁定快照 → 数据更新后答案自动变了，评估结论失效
- ❌ 评估集只覆盖 happy path → 歧义场景/边缘场景出问题无法提前发现

**适用范围**: 任何知识域 Skill（数据查询/API 文档/产品知识库）。代码类 Skill 用单元测试替代。

**案例引用**: Anthropic 数据分析系统，离线评估集分两类（dashboard-based + long-tail generated）；上线门槛 90%；50+ 条后边际收益递减。

### BP-044: 主动知识库自修复——定时 Agent 扫描纠错信号自动开 PR

**来源**: Anthropic 数据分析博客 2026-06-05 在线验证章节

**问题**: 参考文档写好之后会过时。谁来发现？靠人工巡检频率太低，靠用户反馈太被动。

**Anthropic 的做法**：定时运行 Agent，扫描 Slack 频道（或其他反馈渠道）中的纠错语言，自动生成参考文档的修复草稿并开 PR 给领域负责人审核。

**信号检测**（纠错语言模式）:
- "应该用 X 不是 Y"
- "这个字段已经废弃了"
- "你给的数字不对，实际上是..."
- "那个表现在叫..."
- 任何包含"但实际上"、"其实是"、"你搞错了"的反馈

**五步实施方案**:
1. **信号源注册**：明确哪些渠道是反馈来源（Slack 频道/GitHub Issues/内部 ticketing）
2. **定时 Agent**：按日/周频率扫描，提取纠错信号，关联到具体的业务领域
3. **草稿生成**：Agent 根据纠错信号生成参考文档的 diff（而非直接修改）
4. **人工审核门控**：PR 由领域负责人审核后合入，防止 Agent 自动写错
5. **闭环追踪**：追踪从"发现信号"到"文档更新合入"的时长，目标 ≤7 天

**额外在线监控指标**:
- **语义层解析命中率**：多少查询成功从语义层获取（命中率下降 → Skill 路由有问题）
- **回复中纠正语言频率**：用户多久说一次"不对"（频率上升 → 知识库过时）

**适用范围**: 任何对外提供回答的知识域 Skill，特别是数据/产品/API 相关领域。对话 Agent 也适用（用对话日志替代 Slack）。

**案例引用**: Anthropic 数据科学团队，定时 Agent 扫描 Slack 频道，自动起草参考文档修复并开 PR 给领域负责人。

### BP-045: 创作类长任务 Skill 工作流——模型选型与分阶段迭代原则

**来源**: ConardLi garden-skills 开源项目实战总结（web-video-presentation/web-design-engineer/gpt-image-2）

**核心发现**: 创作类长任务（网页视频/设计/图片生成）的成败，与"提示词写得多好"关系不大，与"工作流分阶段是否正确"关系极大。

**三项关键原则**:

1. **方向先于执行**（最高优先）：脚本/主题/章节大纲/视觉方向在第一轮确定清楚，方向错了后面做得再精致也没用。前 20% 时间用于对齐方向，后 80% 时间才做执行。
   - Before: 直接让 Agent 做完整视频 → 跑完发现风格不对 → 全返工
   - After: 第一轮只确认 outline + 主题 → 满意再推进执行

2. **模型选型影响输出质量**：创作类任务中，模型能力对审美判断、章节规划、代码实现和返工决策有直接影响。低能模型处理复杂创作任务时结果差异大。Anthropic 实测：Opus 4.7 效果明显优于较小模型。

3. **局部迭代优于全局重做**：整体跑通后，针对不满意的局部章节单独反馈（"第二章节奏太平"/"结尾缺记忆点"），而非重跑全局。Agent 局部迭代比全局重做效率高 3-5 倍。

**gpt-image-2 的额外发现**: 图片生成失败原因通常不是"模型听不懂风格词"，而是"任务缺少结构"——没有说清楚画面目标/构图/材质/光线/字体限制/输出尺寸/后续编辑空间。结构化任务描述是图片生成准确率的关键。

**Skill 设计启示**: 创作类 Skill 应内置"方向对齐检查点"（第一轮 review 是强制步骤），而非让用户直接跑到最终结果。

**案例引用**: ConardLi garden-skills，GitHub 7K Stars，web-video-presentation / web-design-engineer / gpt-image-2 三个 Skill 的用户反馈汇总，2026-06 发布。

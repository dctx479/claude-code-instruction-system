## 场景3: 实施优化方案

### 3.1 实施architect Prompt优化

**步骤1: 查看详细方案**

打开 `memory/optimization-proposals/2026-01-16.md`:

```markdown
## [2026-01-16] 优化方案 #001

### 背景
- 触发原因: architect Token消耗持续偏高
- 当前状态: 平均15,500 tokens/任务
- 目标: 降至12,000 tokens/任务

### 方案
1. 移除冗余示例: 从15个减少到5个核心示例
2. 提取共享内容: 将通用配置与使用规范移至 `docs/BEST-PRACTICES.md`
3. 精简指令: 合并相似的指令步骤

### A/B测试计划
- 测试组: 10个新任务使用优化prompt
- 对照组: 10个新任务使用原prompt
- 测试周期: 3天
- 成功标准: Token降低≥20%, 成功率≥95%
```

**步骤2: 准备优化内容**

阅读当前 architect prompt:
```bash
cat agents/architect.md
```

创建优化版本 (手动或让auto-optimizer生成):
```bash
/agents auto-optimizer "为architect生成优化后的prompt"
```

**步骤3: 启动A/B测试**

1. 保存原prompt为 `architect.md.backup`
2. 创建优化prompt为 `architect-optimized.md`
3. 配置A/B测试 (50%流量使用优化版)
4. 监控接下来的20个任务

**步骤4: 收集数据**

在测试期间，performance-monitor会自动记录:
- 每个任务的Token消耗
- 成功率
- 执行时间
- 用户评分

## 最佳实践

### 7.1 监控频率

**建议节奏**:
- **日报**: 任务量>10时自动生成，快速发现异常
- **周报**: 每周一固定查看，识别趋势
- **月报**: 月初审查，制定优化计划
- **即时查询**: 发现异常时立即深入分析

### 7.2 优化原则

1. **数据驱动**: 基于真实数据，不凭直觉
2. **小步快跑**: 优先实施低风险、高收益的优化
3. **A/B测试**: 所有优化必须经过验证
4. **质量优先**: 成本优化不能牺牲质量
5. **记录沉淀**: 成功经验形成可复用模式

### 7.3 风险控制

**低风险优化** (可直接实施):
- Prompt长度优化 (移除明显冗余)
- 并行化改造 (独立任务)
- 缓存策略 (幂等操作)

**中风险优化** (需A/B测试):
- Prompt结构调整
- 模型降级
- 工作流重组

**高风险优化** (需充分讨论):
- Agent角色重新定义
- 核心逻辑变更
- 大规模架构调整

### 7.4 成功案例学习

**案例1: architect Prompt优化**
- 问题: Token消耗过高
- 方案: 精简示例 + 提取共享内容
- 效果: Token降低27%，质量不变
- 可复用模式: 示例精简法

**案例2: 多文件并行审查**
- 问题: 串行审查效率低
- 方案: 使用PARALLEL策略
- 效果: 速度提升60%
- 可复用模式: 并行化改造

### 7.5 常见问题

**Q1: 数据不足怎么办?**
A: 至少需要1周数据才能生成有意义的报告。新系统请耐心等待数据积累。

**Q2: 优化后成功率下降怎么办?**
A: 立即回滚，重新分析原因，调整方案后再次测试。

**Q3: 多个优化冲突怎么办?**
A: 串行实施，每次只改变一个变量，确保能准确归因效果。

**Q4: 如何说服团队实施优化?**
A: 用数据说话，展示A/B测试结果，强调风险控制措施。

### C. 指标定义
- **成功率**: 成功任务/总任务
- **首次成功率**: 无需重试的成功任务/总任务
- **Token效率**: 平均Token消耗
- **成本效率**: 单位任务成本
- **用户满意度**: 用户评分平均值 (1-5分)
